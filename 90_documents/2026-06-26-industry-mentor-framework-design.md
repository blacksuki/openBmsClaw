> 日期：2026-06-26
> 来源讨论：`90_documents/IC_Tutor.md`（逻辑层思路）、`IC_Tutor_sample.md`（持久化/目录思路）
> 评审：经 superpowers:brainstorming + gstack /office-hours 核心结论双重审查后收敛
> 状态：待用户确认

---

## 0. 一句话

先做**一个储能行业、专家签收过的「知识地图 + 诊断脚本」+ 一个薄 mentor 提示 + 一份 2 周 roadmap**，跑通一次真实的"零基础到入门"，验证它打不打得过"直接问 AI"。不先建通用平台。诊断必须由专家引导，不让小白自己空想范围。

---

## 1. 目标与受众（已锁定）

- **目的**：快速的行业入门**引导**教程。
- **受众**：行业**初学者、入门级**（只分"完全零基础 / 略懂一点"两档）。
- **周期**：**1 天 – 2 周**。
- **首个真实用户**：openBattery 团队自己（日本储能行业），不是泛化的"任何行业任何人"。

## 2. 非目标（明确不做 / 推迟到 v2）

- ❌ 通用"换包即换行业"框架（半导体/医疗/金融…）—— **v2 愿景**。
- ❌ 四模块独立 subagent（Diagnoser / Planner / Mentor / Reviewer 各自成 agent）。
- ❌ 严格的 `learning_state.yaml` schema 与跨会话状态机。
- ❌ 认证级测验 / 掌握度晋级闸门。
- ❌ 服务专家、从业者，或 1 个月以上的深度路线。

> 推迟理由（office-hours 核心结论）：以上多为**通用品 + 架构先行**。在"2 周一次性入门"场景下属过度工程，验证前不投入。

## 3. 核心价值与护城河

**价值 = 专家签收的知识资产（GIGO 控制），含两块：**
1. **知识地图**（权威行业事实源）——避免模型自由幻觉行业内容。
2. **专家诊断脚本**——替"说不清自己要学什么"的小白把学习范围问出来。

Agent 编排（排课、提问、自测）是通用品，raw AI 也能做；别人给不了的是这两份**懂行专家签收过的资产**。诊断尤其关键：小白不知道自己不知道什么，只有专家预置的引导问题能摸清真实范围。

> 真正的竞争对手是"用户直接问 Claude/ChatGPT 教我储能"。v1 必须靠**知识地图的准确性与结构**赢过它，而不是靠 agent 花活。

## 4. v1 的四件交付物

### 4.1 `knowledge_map.md`（核心资产之一，唯一权威事实源）

**最小 schema（人工/外部系统产出，人工签收后才可用）：**

```markdown
---
industry: 储能 (energy storage, Japan)
status: signed-off            # draft | signed-off
signed_off_by: <人名>
signed_off_date: <YYYY-MM-DD>
sources: [<URL/文献/内部文档清单>]
---

# 行业骨架（节点）
1. <节点名>
   - 适合谁: [采购/销售/研发/投资/...]   # 专家判定: 哪类人需要这个节点
   - 识别信号: <小白说出什么话/有什么处境, 说明他需要它>
   - 核心概念: [...]            # 入门必须懂的几个词
   - 一句话事实: <可被引用的权威陈述>
   - 经典案例: <1 个>
   - 来源: <引用编号>
2. ...

# 入门必答的 N 个问题（题库, 供轻量自测）
- Q: ... / A: ...
```

> `适合谁 / 识别信号` 两个标签是**诊断的依据**（见 4.4）。专家在签收知识地图时一并填写——只有懂行的人能判断"哪句话的小白该学哪个节点"。

**硬规则（GIGO 闸门）**：
- `status != signed-off` ⇒ mentor **不得把内容当事实讲**，只能进"未校验研究模式"，每条标注存疑+来源。
- mentor **只消费、不生产** 知识地图。新增行业事实必须走"研究 → 人工签收"回流，不能由 agent 自我提拔。

### 4.2 `diagnostic.md`（专家诊断脚本，核心资产之二，与知识地图一起签收）

**为什么需要**：受众是行业小白，**说不清自己要学什么**（不知道自己不知道）。让小白自述范围 = 让病人自己开方。必须由懂行业的专家预置"把范围问出来"的引导问题。**诊断问题本身也受 GIGO 约束**：问错 → 范围错 → roadmap 全废。

**专家工量很轻**——只写一份精简脚本（约 5–7 个问题），形态如下：

```markdown
---
status: signed-off            # 与 knowledge_map 同步签收
signed_off_by: <人名>
---

# 诊断脚本（recognition 式, 一次一问, 按答案分叉)
Q1（定位用）: 你接触储能，更接近哪种处境？
  - 要采购/选型      → 侧重节点 [..]
  - 要卖给客户       → 侧重节点 [..]
  - 要做产品/研发    → 侧重节点 [..]
  - 只想看懂行业全貌 → 全节点浅讲
Q2（场景化探测）: 假设下周你要 <具体场景>，下面哪个你"完全没概念"？
  - <选项A> / <选项B> / <选项C>     → 命中的就是必学节点
Q3 ...（按需追问，封顶 ~7 问）
```

**设计原则（同时编进 `mentor.md` 的方法论）：**
1. **认得 > 想得起**：给具体选项/场景让他认，不要开放式"你想学什么"。
2. **具体场景代替抽象**：用他真实会遇到的处境，不用行业黑话。
3. **一次一问，按答案走分叉**，临场可追问。
4. **回放确认**：问完用大白话把范围还给他——"所以你要学这 5 块、跳过那 3 块，对吗？"——他确认/微调后才开课。

**诊断产出** = 在 `knowledge_map.md` 节点上的一份**圈选**（要学哪些 / 跳过哪些 / 什么深度）。这份圈选是 `roadmap.md` 的输入。

### 4.3 `roadmap.md`（2 周入门路径）

- 按**诊断圈选出的节点**（4.2 产出）排成 1–2 周的逐日/逐段计划——不是全量节点。
- 教学策略只用两种：**Executive Overview（快速全貌）** + **Case Study（案例落地）**，按诊断出的目的选侧重。
- 与 mentor 提示分离（roadmap 常调，提示稳定）。

### 4.4 `mentor.md`（薄提示，稳定不变）

职责（合并了原"诊断"和"轻量测验"，不拆 subagent）：
1. **专家式诊断（强化重点）**：执行 `diagnostic.md` 的脚本，用 recognition 式问题把小白对号入座到知识地图节点子集，问完**回放确认**范围。绝不让小白自己空想"想学什么"。
2. 按确认后的范围 + `roadmap.md` 推进，事实**只引用 `knowledge_map.md`**。
3. 每节后做**轻量自测**（"懂没懂"，用题库），不做严格闸门。
4. 遇到知识地图没覆盖的问题：明说"超出已校验范围"，不编。

## 5. Claude Code 落地

```
energy-storage-mentor/
├── mentor.md            # 薄提示（可作 skill 或贴进会话）
├── roadmap.md           # 2 周路径
├── knowledge/
│   ├── knowledge_map.md # 人工签收的权威源（核心资产之一）
│   └── diagnostic.md    # 专家诊断脚本（核心资产之二, 同步签收）
└── research/            # Researcher/外部抓取的原料（未校验, 不可当事实）
```

- 知识地图策展流：`research/` 抓取 → **人工校验签收** → `knowledge/knowledge_map.md` + `knowledge/diagnostic.md`（`status: signed-off`）。
- 会话即上下文，不靠复制长 prompt；v1 不引入状态文件。

## 6. 成功判据（验证假设，而非交付功能）

在一个真实零基础的人身上跑完一次 2 周入门后，回答：
1. **诊断是否摸清了正确范围**？小白没自己空想，专家脚本是否把他对号入座到了对的节点（事后回看：学的东西是不是他真正需要的）？
2. 它给出的行业事实**是否都可追溯到签收过的来源**（零幻觉）？
3. 学完后此人能否**看懂行业全貌 / 与从业者对话 / 做初步判断**（不承诺做研发/创业）？
4. 同一个人，这套**是否明显优于直接问 raw AI**？若不优于 → 价值不成立，回炉。

## 7. The Assignment（office-hours 强制：一个具体动作）

> **本周：人工产出储能行业的 `knowledge_map.md` v0（5–8 个核心节点，每节点带"适合谁/识别信号"标签 + 来源）和一份 5–7 问的 `diagnostic.md`，自己签收；然后在一个真实小白身上先跑诊断、回放确认范围，再跑一次 Day 1 入门，对照"直接问 Claude"看差异。**
> 不要先写 mentor 的任何编排逻辑——先证明这两份专家资产（知识地图 + 诊断脚本）有没有价值。

## 8. v2 回看清单（验证通过后再考虑）

四模块接口契约 · `learning_state.yaml` 跨会话状态 · Reviewer 晋级闸门 · 通用 Knowledge Pack 换包机制（半导体/金融/医疗…）。
