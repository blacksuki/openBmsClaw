# 本周执行计划：SoC 双核协同架构打通 (Week-0518)

> 最新状态核对：2026-07-01

本周计划的核心目标，是全面落地“以集成 SoC 为底层基础，以 MCU 为智能调度核心”的双核架构。本周抛弃过时的分散硬件（纯 BMS/协议板）联调路线，收敛到**打通 MCU 与高集成度 SoC 评估板的双向通信基线**这一最关键的战役上。

## 1. 基本术语与背景知识 (面向新手说明)

为了让团队中嵌入式开发新手和跨层级协作者准确理解本周任务，针对本执行计划中涉及的核心术语和底层概念说明如下：

### 1.1 UI 裁剪宏 (UI Pruning Macros)
- **角色与定义**：在 C/C++ 嵌入式代码中，利用条件编译指令（如 `#if CONFIG_ENABLE_OLED ... #endif`）所设立的配置开关。
- **工作原理与原因**：在资源极其受限的单片机（如当前用于早期验证的 STM32F103C8T6 只有 64KB Flash 与 20KB RAM）上，如果把华丽的 OLED 动画、图形界面驱动全部编译进程序，极易出现内存溢出或编译越界。
- **本周策略**：我们在 `config/` 模块中设立明确的裁剪宏开关。在当前 `V0` 通信打通阶段，通过将屏幕渲染等非核心模块设为关闭状态，极大地节约代码空间，使团队能聚焦于底层通信链路的单点验证。

### 1.2 逻辑分析仪 (Logic Analyzer)
- **角色与定义**：一种连接在单片机与外设通信引脚上、专门用来捕获和显示数字电路总线高低电平（'1' 和 '0'）时序变化的调试硬件。
- **工作原理与原因**：与示波器主要看电压模拟波形不同，逻辑分析仪可以多通道同步记录，并能直接对总线数据（如 I2C）进行协议自动解码。它能将枯燥的高低电平波形直接还原成直观的地址、指令和应答位（ACK/NACK）。
- **本周策略**：在移动电源大电流充放电或协议切换的恶劣电磁环境下，通信总线极易受到杂波干扰。当代码出现读取异常时，依靠肉眼或打印输出无法判断问题根源；而通过逻辑分析仪实时抓包，可以秒级定位是主控没发信号、从机没应答，还是总线被异常拉低。

### 1.3 I2C 总线 (Inter-Integrated Circuit)
- **角色与定义**：一种半双工、两线制（SCL 时钟线 + SDA 数据线）的同步串行通信总线标准，用于主控与外部集成芯片间的数据传输。
- **工作原理与原因**：在 `openBattery` 的“脑体协同”架构中，MCU 作为“大脑”（主机），通过这两根线向 SoC“身体”（从机）频繁下达读取实时电压、读取温度、设定过流保护阈值等命令。I2C 总线引脚占用极少且接口规范标准化，是专用 SoC 最常用的通信信道。
- **本周难点与防死锁**：标准外设库的传统 I2C 例程常带有死循环等待（如 `while(!event);`）。一旦从机由于大电流干扰发生时钟挂死（Clock Stretching）或死锁，整个 MCU 都会被卡死。因此本周任务强调编写带超时检测（Timeout）、非阻塞，并能在死锁时通过连续发送 9 个时钟脉冲来强制释放总线的高健壮性自愈驱动（Bus Recovery）。

### 1.4 大字库 (Large Font Library)
- **角色与定义**：存储用于在显示屏（如 OLED 或 LCD）上绘制多语言复杂文字（如汉字、日文假名）点阵位图的数据集合。
- **工作原理与原因**：基础的英文字母与数字（ASCII 字符集）总共只有 128 个，点阵字模通常只占用几 KB 空间。但要在高端移动电源屏幕上展示优雅的日文假名或中文提示语，完整点阵字库会占用数百 KB 甚至数 MB 的存储器空间。
- **本周策略**：当前 64KB Flash 的测试单片机根本无法装载大字库。在最小可行性验证阶段，必须在配置宏中严格关闭大字库功能，屏幕输出仅降级为基础数字显示或直接以串口打印日志替代。未来迁移到拥有更大存储器（如外挂 SPI Flash 或大容量 STM32G0/L4）的量产硬件平台时再行开启。

## 2. Scope (范围定义)
- **In (纳入本周范围)**:
  - 确立 `V0` 原型的边界，针对当前 64KB F103 学习板建立严格的 UI 裁剪宏。
  - 跟踪并确认“集成 SoC 官方评估板”（如英集芯 IP5328 等）及“逻辑分析仪”的采购到货状态。
  - 在当前 `openBmsClaw` 起步工程中，拉起 `app / services / drivers (SAL) / hal` 的全新软件分层骨架。
  - 实现非阻塞或带硬件 Timeout 的 `hal_i2c` 驱动，并着手编写 I2C `Bus Recovery`（总线自愈）代码。
  - 待 SoC 评估板到货后，立即实现标准 API (`soc_get_voltage`, `soc_get_temp`) 的通信打通。
  - 定义 SoC 硬件危急中断的“逆向高速通道”接口草案。
- **Out (移出本周范围)**:
  - 复杂的 OLED 预测算法渲染（防止 F103 爆内存，初期仅用串口/LED 验证数据提取）。
  - BLE 手机 APP 联调。
  - 多口动态功率智能分配的具体数学模型（本周仅验证通信机制是否可用，不追求分配策略多完美）。
  - 分离式 PD 协议芯片（如 DRP1M1）或 BMS 芯片（如 BQ76942）的适配（已废弃）。

## 3. 当前核心约束
- **防挂死纪律**：本周任何 I2C 代码严禁出现无 Timeout 的死循环 `while` 等待。
- **资源纪律**：即使是 V0 原型，凡是非主线必须的模块（大字库等），必须在 `config/` 提供宏将其关闭。
- **平台延续性**：当前执行平台依然是手头的 `野火 STM32F103C8T6 核心板`。但在代码编写时，必须将其视为“受限平台”，时刻准备向 128KB 的 G071 或 L4 迁移。
- **工具链要求**：必须准备逻辑分析仪，应对大功率跳变下的总线异常测试。

## 4. 本周核心风险与重点
- 本项目最大的技术难点，不再是业务逻辑，而是**如何保障恶劣环境下 MCU 与 SoC 之间的 I2C 通信鲁棒性 (Clock Stretching, 时钟挂死)**。
- 本周不追求适配多家厂商，而是集中精力打透一家厂商的 SoC，将其封装入 `drivers/` (SoC SAL)，固化接入模板。
