# 本周执行计划：SoC 双核协同架构打通 (Week-0518)

本周计划的核心目标，是全面落地“以集成 SoC 为底层基础，以 MCU 为智能调度核心”的双核架构。本周抛弃过时的分散硬件（纯 BMS/协议板）联调路线，收敛到**打通 MCU 与高集成度 SoC 评估板的双向通信基线**这一最关键的战役上。

## 1. Scope (范围定义)
- **In (纳入本周范围)**:
  - 确立 `V0` 原型的边界，针对当前 64KB F103 学习板建立严格的 UI 裁剪宏。
  - 跟踪并确认“集成 SoC 官方评估板”（如英集芯 IP5328 等）及“逻辑分析仪”的采购到货状态。
  - 在当前 `openBmsClaw` 起步工程中，拉起 `app / services / drivers (SAL) / hal` 的全新软件分层骨架。
  - 实现非阻塞或带硬件 Timeout 的 `hal_i2c` 驱动，并着手编写 I2C `Bus Recovery`（总线自愈）代码。
  - 待 SoC 评估板到货后，立即实现标准 API (`soc_get_voltage`, `soc_get_temp`) 的通信打通。
  - 定义 SoC 硬件危急中断的“逆向高速通道”接口草案。
- **Out (移出本周范围)**:
  - 复杂的 OLED 预测算法渲染（防止 F103 爆内存，初期仅用串口/LED 验证数据提取）。
  - BLE 手机 APP 联调。
  - 多口动态功率智能分配的具体数学模型（本周仅验证通信机制是否可用，不追求分配策略多完美）。
  - 分离式 PD 协议芯片（如 DRP1M1）或 BMS 芯片（如 BQ76942）的适配（已废弃）。

## 2. 当前核心约束
- **防挂死纪律**：本周任何 I2C 代码严禁出现无 Timeout 的死循环 `while` 等待。
- **资源纪律**：即使是 V0 原型，凡是非主线必须的模块（大字库等），必须在 `config/` 提供宏将其关闭。
- **平台延续性**：当前执行平台依然是手头的 `野火 STM32F103C8T6 核心板`。但在代码编写时，必须将其视为“受限平台”，时刻准备向 128KB 的 G071 或 L4 迁移。
- **工具链要求**：必须准备逻辑分析仪，应对大功率跳变下的总线异常测试。

## 3. 本周核心风险与重点
- 本项目最大的技术难点，不再是业务逻辑，而是**如何保障恶劣环境下 MCU 与 SoC 之间的 I2C 通信鲁棒性 (Clock Stretching, 时钟挂死)**。
- 本周不追求适配多家厂商，而是集中精力打透一家厂商的 SoC，将其封装入 `drivers/` (SoC SAL)，固化接入模板。
