# 后续阶段计划草案：Phase 1 到 Phase 4 收敛路线

> 草案日期：2026-07-02
> 适用范围：承接 `Plan-Week-0518.md` 完成后的后续规划
> 目标：把 `openBattery` 从“F103 上的软件结构验证”推进到“F030 + SoC 最小量产链路”

## 1. 文档目的

`Plan-Week-0518.md` 的核心价值，是把 `openBmsClaw` 从最小 STM32 生成工程，推进到具备 `board / hal / drivers/soc / services / app / config` 结构、并开始具备 SoC 抽象层雏形的状态。

但该计划完成后，项目仍然没有达到 `0_System/00.tech_architecure.md` 里的设计目标。当前缺的不是再写一轮“继续抽象 SAL”，而是需要把后续路径明确成几个可以逐步验收的阶段。

本文档用于回答三个问题：

1. `Week-0518` 结束后，后面还要做哪些阶段。
2. 每个阶段的目标、前置条件和验收标准是什么。
3. 哪些内容现在不该做，避免 startup 阶段继续积累技术债。

---

## 2. 当前基线

截至 `2026-07-02`，当前已经确认的事实如下：

### 已完成

1. `Phase 0 / F103 bring-up` 基本成立：
   - 目录边界已建立。
   - `cmake --preset Debug && cmake --build --preset Debug` 可通过。
   - `hal_i2c` 已有 timeout、bus recovery、probe 等基础能力。
2. `drivers/soc/` 的结构已初步形成：
   - `soc_api.h`
   - `soc_types.h`
   - `soc_driver.h`（adapter ops vtable）
   - `soc_manager.c`
   - `vendor/demo_soc.c`
3. 配置模型已从散乱宏收敛到：
   - `profile_config.h`
   - `feature_config.h`
   - `app_config.h`
4. 一部分 startup 技术债已经被隔离：
   - 公共 SoC API 不再泄露 HAL 细节
   - `app/` 不再直接触发测试 hook
   - `services/power` 已去掉 ISR 中的慢操作

### 未完成

1. `soc_sal_init()` 还没有进入真实运行路径。
2. `hal_exti_init()` 尚未在实际运行中完成 SoC INT 初始化。
3. 没有真实 SoC 芯片联调记录。
4. 没有逻辑分析仪抓包记录。
5. `srv_power_alloc`、`srv_thermal`、故障状态机尚未形成最小闭环。
6. `PROFILE_F030_MIN_PRODUCT` 只有配置入口，没有真实验证结果。

因此，当前项目的真实状态是：

> 结构层已进入 `Phase 1`，但真实硬件链路还没有闭环。

## 3. 后续阶段总览

后续路线建议固定为：

```text
Stage A  Phase 1 收口：真实 SoC 基线闭环
        ↓
Stage B  Phase 2 启动：服务层最小策略闭环
        ↓
Stage C  Phase 3 启动：F030 样片验证
        ↓
Stage D  Phase 4 启动：最小量产功能集闭环
        ↓
Stage E  量产前验证与资料沉淀
```

说明：

- `Stage A` 是当前最优先阶段，没有它，后续阶段都没有真实输入。
- `Stage B` 不是追求复杂算法，而是建立最小可运行策略闭环。
- `Stage C` 的目标是把 “F030 量产候补” 从文档判断变成工程事实。
- `Stage D` 是把平台从“能联调”推进到“能做最小产品”。
- `Stage E` 是量产前必要的验证和知识沉淀，不是附属动作。

## 4. Stage A：Phase 1 收口 - 真实 SoC 基线闭环

### 4.1 目标

把当前仅存在于代码结构中的 `SoC SAL`，推进到真实运行时闭环。

### 4.2 关键任务

1. 在 bring-up 或 service 初始化路径接入 `soc_sal_init()`。
2. 确认 `soc_poll_events()` 进入真实在线分支，而不是始终停留在离线/演示路径。
3. 让 `hal_exti_init()` 在运行时真正完成 SoC INT 初始化。
4. 接入至少一块真实 SoC 评估板。
5. 使用逻辑分析仪抓到至少一份真实 I2C 通信波形。
6. 打通以下三者至少之一：
   - 真实 chip id
   - 真实电压
   - 真实温度

### 4.3 输出物

- 运行路径中的 `soc_sal_init()` 调用点
- SoC INT 初始化日志
- 一份真实 I2C 抓包记录
- 一份联调记录，说明寄存器地址、读值、异常现象和恢复结果

### 4.4 验收标准

满足以下条件，才算 `Stage A` 完成：

1. 串口日志中能看到清晰的 SoC 初始化成功或失败信息。
2. `EXTI -> callback -> service` 链路至少完成一次真实或自测闭环验证。
3. 逻辑分析仪能抓到真实 SoC 应答波形。
4. 终端能打印至少一个真实 SoC 数据值，而不是 demo 常量。

### 4.5 风险与注意事项

- 不允许在 ISR 快路径里继续加入 I2C、printf、delay。
- 不允许把 demo adapter 的固定值当作“真实 SoC 已打通”。
- 如果 SoC 到货延迟，这个阶段可以先完成 `soc_sal_init()` 接线和 EXTI 自测闭环，但不能宣称阶段完成。

## 5. Stage B：Phase 2 启动 - 服务层最小策略闭环

### 5.1 目标

在真实 SoC 数据链路可用的前提下，建立最小服务层策略闭环。

### 5.2 范围

本阶段只做最小策略，不做复杂产品体验：

- 做：热管理、功率限制、故障锁存、恢复条件
- 不做：复杂 UI、BLE、预测算法、美化显示

### 5.3 关键任务

1. 建立 `srv_thermal` 最小逻辑：
   - 读取 NTC 或 SoC 温度
   - 达阈值后执行降额
2. 建立 `srv_power_alloc` 最小逻辑：
   - 基于端口状态或总功率池下发端口功率限制
3. 建立故障状态机最小版本：
   - 锁存
   - 提示
   - 恢复条件
4. 明确快路径/慢路径边界：
   - 快路径只锁存和通知
   - 慢路径负责 I2C 读取、显示、日志、恢复判断
5. 统一 `soc_set_port_power` 的单位口径，收敛到 `mW`（注：ISSUE-002 后适配器内部 `set_port_power_mw` 已是 `mW`、manager 做 W→mW 转换；本项仅剩公共 API 签名 `soc_set_port_power(..., uint16_t max_power_w)` 待收敛）

### 5.4 输出物

- `services/` 下最小策略闭环代码
- 故障状态定义表
- 温度阈值与降额策略表
- 端口功率限制下发策略说明

### 5.5 验收标准

1. 过温输入能触发降额或锁定状态。
2. 至少一条 SoC 参数下发动作可观察。
3. 服务层不直接 include `hal_i2c.h` / `hal_adc.h`。
4. UI 只展示状态，不直接下发硬件动作。

### 5.6 风险与注意事项

- 当前阶段不要把 `srv_power_alloc` 做成复杂数学模型。
- 不要为了赶进度把 vendor 寄存器名泄露回 `services/`。
- 如果温度来源暂时只有软件模拟，可先跑流程，但必须标明“未板测”。

## 6. Stage C：Phase 3 启动 - F030 样片验证

### 6.1 目标

把“F030 是量产候补”从理论判断推进为真实工程验证。

### 6.2 关键任务

1. 切换 `PROFILE_F030_MIN_PRODUCT` 完成首次构建。
2. 梳理 F103 与 F030 的板级适配差异：
   - 启动文件
   - 链接脚本
   - 时钟
   - I2C/UART/ADC/EXTI/WDG
3. 至少验证两类器件：
   - 一颗基准 F030
   - 一颗国产候补 F030
4. 完成最小板级验证：
   - 下载
   - 串口
   - I2C
   - ADC
   - EXTI
   - watchdog
5. 记录资源余量：
   - FLASH
   - RAM

### 6.3 输出物

- F030 构建结果
- F030 样片验证记录
- `F030 porting notes`
- F030 与 F103 差异清单

### 6.4 验收标准

1. `PROFILE_F030_MIN_PRODUCT` 可构建。
2. F030 样片至少完成一次下载和基础串口日志输出。
3. F030 上至少验证一条 I2C 通信链路。
4. 得到一版真实资源占用数据。

### 6.5 风险与注意事项

- F030 阶段不要默认“pin to pin 完全兼容”。
- 若 F030 资源或调试稳定性不满足要求，应立即记录回退条件，而不是硬顶。
- F030 验证失败，不代表架构失败；只代表 MCU 型号要上移。

## 7. Stage D：Phase 4 启动 - 最小量产功能集闭环

### 7.1 目标

在 F030 或回退 MCU 上，形成最小量产功能集。

### 7.2 最小量产功能集

只包含以下能力：

1. SoC 状态读取与参数下发
2. NTC / 温度采样
3. 热管理降额
4. 按键最小状态机
5. LED 或基础 OLED 显示
6. 故障锁存与恢复
7. 看门狗与错误码

明确不包含：

- BLE 联动
- 复杂预测算法
- 大字库
- 重 UI
- 双镜像 IAP / OTA

### 7.3 关键任务

1. 明确最小 SKU 的功能边界。
2. 固定最小 UI 方案：LED 或基础 OLED。
3. 固定最小按键逻辑。
4. 固定异常状态与用户提示方式。
5. 验证最小功能集在目标 MCU 上的资源余量。

### 7.4 输出物

- 最小量产功能列表
- 资源预算表
- 最小交互流程图
- 板级功能验证记录

### 7.5 验收标准

1. 最小功能集可以在目标 MCU + SoC 上闭环运行。
2. Flash / RAM 余量满足最小量产要求。
3. 基础交互、温度保护、故障状态均可观测。

## 8. Stage E：量产前验证与资料沉淀

### 8.1 目标

把当前工程从“能跑”推进到“可复制、可量产验证、可交接”。

### 8.2 关键任务

1. I2C 异常场景验证：
   - 短接 SDA
   - SoC 无应答
   - 总线挂死恢复
2. 温升与降额验证：
   - 正常负载
   - 高负载
   - 异常保护触发
3. SoC 异常状态覆盖表：
   - 过温
   - 过流
   - 短路
   - 通信失败
4. 技术资料沉淀：
   - SoC 型号
   - 寄存器/API 文档
   - 保护策略说明
   - 测试记录
   - 量产口径

### 8.3 输出物

- 异常测试清单
- 温升/降额测试记录
- SoC 接口资料归档
- 量产前验证 checklist

### 8.4 验收标准

1. 异常场景不是靠口头说明，而有记录。
2. SoC 和 MCU 的关键接口、配置和保护参数可追溯。
3. 新成员能够根据文档复现最小验证路径。

---

## 9. 当前不建议展开的内容

在 `Stage A ~ Stage C` 之前，不建议展开以下内容：

1. BLE 手机 APP 联调
2. 复杂 OLED 图形界面
3. 多语言大字库
4. 复杂剩余时长预测算法
5. 多 SoC 并行适配
6. 双镜像 IAP / OTA

原因很简单：

- 这些内容会显著增加资源占用和验证面。
- 当前最核心瓶颈仍是 SoC 基线闭环和 F030 可行性验证。

## 11. 当前建议结论

如果只按最短路径推进，建议执行顺序固定为：

1. **先做 `Stage A`**：真实 SoC 闭环  
2. **再做 `Stage B`**：服务层最小闭环  
3. **再做 `Stage C`**：F030 样片验证  
4. **最后做 `Stage D + E`**：最小量产闭环与验证资料沉淀  

对当前项目而言，真正的关键路径不是“再加多少功能”，而是：

> 先把真实 SoC、真实中断、真实样片、真实资源边界逐个打实。
