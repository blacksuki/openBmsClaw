# Plan-Week-0325 item-04

## 1. 目标
整理并稳定当前 `openBmsClaw` 的 STM32 生成工程，确保该工程可以被下载、调试、重复打开和持续扩展。

## 2. 与主干目标的关系
没有可稳定复用的最小工程，后续所有模块化设计都只能停留在文档层。这个动作是把当前仓库从“只有文档”推进到“有可运行起点工程”的关键一步。

## 3. 前置条件
- 必备软件已安装完成
- 已具备可用外部 `SWD` 调试器，或至少已明确当前暂以串口 Bootloader 作为下载应急路径
- 已确认主路径使用 `STM32CubeIDE for Visual Studio Code`
- 已确认当前工程目录为 `openBmsClaw/`

## 4. 实施步骤
1. 先确认当前工程事实，而不是重新假设一个新工程：
   - 当前工程名：`openBmsClaw`
   - 当前工程路径：`openBmsClaw/`
   - 当前起步工程对应 `STM32F103C8T6`
   - 当前学习实物对应 `野火小智 STM32F103C8T6 核心板（双 USB 款）`
2. 优先整理和验证现有工程，只有在现有工程无法满足需要时，才重新生成替代工程。首版只启用或确认本周需要的外设：
   - `GPIO`
   - `USART`
   - `ADC`
   - `I2C`
   - `TIM`
3. 完成基础工程命名与目录落位，避免后续出现“演示工程名”和“当前项目名”混用。
4. 先验证三件基础事实：
   - 工程能成功生成
   - 工程能成功编译
   - 工程能通过外部 `SWD` 调试器下载并连接调试
5. 为工程建立最小约定：
   - 串口波特率固定一版
   - 日志输出函数保留统一入口
   - 中断和外设初始化不直接写在业务层
6. 记录首个工程整理过程中的坑点，例如：
   - 调试配置
   - 工具链路径
   - 下载失败现象
   - 串口占用问题

## 5. 完成判定
- 工程能反复打开、编译、下载
- `SWD` 调试链路连接稳定
- 最小外设初始化已进入当前工程

## 6. 输出物
- 当前 `openBmsClaw` 最小工程
- 开发环境验证记录
- 工程整理过程中的问题清单

## 7. 注意事项
- 不要在最小工程阶段引入 `FreeRTOS`、复杂协议栈或大量第三方库。
- 工程先求“稳定可复现”，再求“功能丰富”。
- 不要把“推荐平台路线”和“当前代码事实”混成一个结论。
