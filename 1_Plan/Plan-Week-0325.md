# Plan

本周计划的目标，是在不偏离“可定制、低成本、合规的开源 BMS 系统方案”主线的前提下，把第一轮实施工作拆成可执行、可验证、可回写文档的动作。结合当前实际条件，本周主交付先收敛为“最小可运行验证”，以现有 `野火小智 STM32F103C8T6 核心板（双 USB 款）`、外部 `SWD` 调试器和当前 `openBmsClaw` F103 起步工程为执行平台；`USB-C/PD` 与 `BMS` 评估板的真实联调作为条件满足后的追加动作。长期平台参考路线仍保持 `STM32G0 + Nucleo`。

## Scope
- In:
  - 明确首周 `V0` 原型范围和模块边界
  - 以现有 `野火小智 STM32F103C8T6 核心板（双 USB 款）` + 外部 `SWD` 调试器完成最小 bring-up 和基础可观测闭环
  - 跟踪首批 `USB-C/PD` 与 `BMS` 板的到货状态，并按实际节奏安排后续联调
  - 整理并稳定当前 `openBmsClaw` STM32 生成工程，补齐分层骨架和基础可观测能力
  - 先完成输入链路方向的首轮验证准备；若扩展板到货，再启动真实板级联调
  - 建立首批测试、排错、复盘方法，便于新手持续推进
- Out:
  - 完整多口功率分配算法
  - 全量 `BLE / OTA`
  - 五联托盘整机联调
  - 量产 PCB 定版与器件定型

## 当前约束
- 默认主线顺序：先输入链路，后输出链路。
- 当前现成主开发硬件：`野火小智 STM32F103C8T6 核心板（双 USB 款）`。
- 当前调试链路：当前学习板带标准 `5-pin SWD`，可接外部 `DAP / JLINK / ST-LINK`；项目标准推荐链路仍是 `ST-LINK`。
- 当前代码事实：活动工程为 `openBmsClaw`，器件事实为 `STM32F103C8T6`。
- `NUCLEO-G071RB`、`NUCLEO-G0B1RE` 属于长期平台参考开发板，不是本周唯一执行前提。
- `X-NUCLEO-DRP1M1`、`BQ76942EVM` 等扩展板属于后续接入目标，不视为本周已具备前提。
- 个人开发主路径：`STM32CubeIDE for Visual Studio Code`。
- 本周若无扩展板到货，仍需完成最小可运行验证，不等待完整联调条件。
- 本周不把“跑通完整输入链路闭环”设为唯一交付标准。

## 新增风险判断
- 当前项目最大的技术难点，不是通用框架代码量，而是首批主流硬件的 `BSP / HAL / 驱动适配`。
- 对 `BMS` 和 `USB-C/PD` 芯片的接入，本质上是“先啃文档、再做初始化和联调、最后沉淀接入模板”的工作。
- 起步阶段不追求同时支持多款芯片，而是先把 `1` 款 `MCU`、`1` 款 `USB-C/PD` 板、`1` 款 `BMS` 板的接入路径做实。
- 本周计划需要为后续开源协作提前准备“芯片支持矩阵、接入记录模板、已知问题清单”，否则后续新增芯片仍会回到重复踩坑。

## 建议周节奏
| 时间 | 重点 |
| --- | --- |
| Day 1 | 锁定 `V0` 范围、确认采购与到货、梳理工程骨架 |
| Day 2 | 创建最小工程，完成 `GPIO/UART/ADC/I2C` 基础配置 |
| Day 3 | 完成 bring-up、测试表、排错顺序，准备联调脚本和记录模板 |
| Day 4 | 在当前 F103 学习板 + 外部 `SWD` 调试器上固化可下载、可观察、可排错的最小工程；若协议板到货，再追加联调 |
| Day 5 | 持续完善输入链路抽象、芯片支持矩阵、寄存器要点和已知问题记录 |
| Day 6 | 若 `BMS` 板到货，则进行上电与通信探测；未到货则整理 BMS 接口清单 |
| Day 7 | 输出复盘、更新系统文档、明确下周进入 `BMS`、`power path` 或协议细化的入口 |

## 到货分支规则
- 若 `USB-C/PD` 扩展板本周到货：在当前 F103 学习板 + 外部 `SWD` 调试器稳定运行后，追加输入链路实板联调。
- 若 `BMS` 评估板也在本周到货：本周追加“上电 + 通信探测”，但不改变“最小可运行验证优先”的主线。
- 若 `BMS` 评估板未到货：本周只完成 BMS 接口准备和资料梳理，不强行承诺实板验证结果。
- 若两类扩展板都延迟：本周交付目标保持为“最小可运行验证完成，联调前置条件就绪”。
