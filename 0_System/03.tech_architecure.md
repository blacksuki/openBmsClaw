# openBattery 顶层技术架构与实施路线

> 最新更新：2026-07-18
> 架构状态：Accepted / 继续沿用
> 决策范围：openBattery 平台战略、MCU + SoC 双芯片协同架构、NFC(ST25DV)数据读出、软件分层边界、startup 阶段实施约束

## 1. 文档目的

本文用于明确 `openBattery` 的战略定位、核心架构决策、软件分层规则和当前实施路线。

它不是完整产品规格书，也不是某一块 PCB 的定板说明。它的作用是让后续计划、编码、采购、样片验证和文档更新都沿着同一条主线执行，避免 startup 阶段因为临时验证代码不断堆积成长期技术债。

本文的核心结论：

1. `openBattery` 继续采用 **MCU + 专用电源 SoC** 的双芯片协同架构，**辅以 NFC(ST25DV)作为数据读出窗口**(见 ADR-003)。
2. SoC 负责高频电源转换、快充协议握手和基础硬件保护。
3. MCU 负责状态读取、策略调度、**数据可读(经 I²C 写入 NFC 标签供 tap 读出,而非设备端屏幕)**、日志记录和系统级安全管理。
4. 数据用途为**安全合规 + 监控(记录)**;用户 tap 读当前状态即可,**实时/隔空/远程监控/OTA 不在需求内(2026-07-18 锁定)**,BLE 降为 premium 备选。
5. 当前开发入口是 `STM32F103C8T6` 学习板和 `openBmsClaw/` 工程。
6. 当前量产候补优先评估 64KB Flash 档位 `F030`，但必须通过样片、资源和可靠性验证后才能进入量产参考设计;NFC 只用 I²C、无射频栈,放宽了对量产 MCU 的资源压力。

## 2. 架构决策记录

### ADR-001：采用 MCU + 专用电源 SoC 的双芯片协同架构

**状态**：Accepted

#### 背景

移动电源平台需要同时处理电源转换、快充协议、电芯保护、温升控制、多口功率分配、用户显示和后续可扩展能力。若让 MCU 直接承担所有底层电源闭环、快充协议栈和 BMS 保护，会快速提高开发复杂度、认证风险和量产不确定性。

专用电源 SoC 已经在升降压、PD/QC/UFCS 协议、功率路径和基础保护方面具备成熟能力。`openBattery` 的价值不应重复制造这些底层能力，而应建立可观察、可配置、可迁移的开放控制平台。

#### 决策

采用“**SoC 负责体力活，MCU 负责脑力活**”的架构。

- SoC：负责升降压、电源路径、快充协议握手、基础 OVP/OCP/OTP 保护。
- MCU：负责读取状态、下发策略、动态降额、显示交互、日志、异常状态管理和平台扩展。

#### 取舍

| 方案 | 优点 | 缺点 | 当前结论 |
| --- | --- | --- | --- |
| MCU 大包大揽 | 可控性最高，理论上最开放 | 开发量大，认证风险高，对低成本 MCU 不现实 | 不采用 |
| 纯 SoC 黑盒方案 | 成本低，开发快 | 数据不可见，策略不可控，平台差异化弱 | 不采用 |
| MCU + SoC 协同 | 复用成熟电源能力，同时保留开放策略层 | 需要定义稳定接口和抽象层 | 采用 |

#### 后果

- 架构重点从“自己实现所有底层驱动”转为“定义稳定 SoC 接口和策略层”。
- 必须坚持 SoC 抽象层，否则上层业务会被某一家芯片绑定。
- 必须明确安全快路径和慢路径，不能让 MCU 成为唯一硬件保护来源。

### ADR-002：当前 MCU 路线采用 F103 bring-up，F030 量产候补

**状态**：Accepted for startup / F030 Proposed pending validation

#### 背景

当前代码事实是 `openBmsClaw/` 下的 STM32 工程，当前学习板为 `STM32F103C8T6`。为了降低初期学习、调试和工具链风险，startup 阶段应先把基础外设、日志、I2C、ADC、EXTI 和分层结构跑稳。

量产阶段需要更关注 BOM 成本，因此优先评估 `F030` 级别低成本 MCU。但 F030 资源有限，不适合作为“全功能平台 MCU”。

#### 决策

当前路线固定为：

```text
F103 bring-up（当前学习和开发）
        ↓
F030 量产候补（生产环境预计使用，需样片验证）
        ↓
F030 + 专用 SoC 最小量产链路
        ↓
若 F030 验证失败，再上移到 STM32G0 / GD32E230 / 更宽裕 MCU
```

#### 约束

F030 最小量产版本只能默认包含：

- SoC 状态读取与配置下发
- NTC / ADC 采样
- **NFC(ST25DV)状态读出:MCU 经 I²C 写快照+日志摘要,手机 tap 读回**
- **关键计数/事件日志的掉电持久化(充放电次数、过温/故障事件+时间戳,小体量),用于安全合规与售后诊断**
- LED 最小指示(轻量 OLED 数字页为可选,非必需)
- 按键状态机
- 轻量功率分配与温度降额
- I2C/UART 日志与错误码
- watchdog 和异常恢复

以下能力不能默认进入 F030 最小版本：

- LVGL 或复杂 UI、设备端图形显示屏
- 大字库、多语言资源
- 复杂 BLE 协议栈(BLE 仅作 premium 备选)
- 实时/隔空监测、远程监控、隔空 OTA
- 双镜像 IAP / OTA
- **大量历史日志/长时间序列持久化**(区别于上方“关键计数/事件日志”:后者小体量、为合规必须包含;前者的大容量历史记录不默认进入最小版)
- 多颗 SoC、多屏、多通信外设同时在线

### ADR-003：采用 NFC(ST25DV)作为数据读出窗口

**状态**：Accepted（2026-07-18）

#### 背景

需求澄清后,充电宝数据的用途是**安全合规 + 监控(内部记录)**;用户实时查询非实际需求,tap 读到当前状态(电量、充放电次数)即可,离开后再次 tap 取更新可接受。因此“数据可读”的载体不需要隔空实时链路,也不需要设备端显示屏。

#### 决策

MCU 经 I²C 把电池状态**快照 + 关键日志摘要**写入 **ST25DV**(被动 NFC 动态标签,NFC Forum Type 5 / ISO15693,I²C + RF 双接口,支持能量收割);手机或 Web NFC **tap 读出**,再次 tap 取更新。BLE 降为**未来出现实时/隔空/远程监控/OTA 需求时**才启用的 premium 备选。

#### 取舍

| 方案 | 优点 | 缺点 | 当前结论 |
| --- | --- | --- | --- |
| NFC(ST25DV) | 仅 I²C、无射频栈、成本最低;被动标签基本免日本技適;掉电可读;贴合软件团队 | 仅 tap、无实时监测;金属机身需屏蔽验证 | **采用(主线)** |
| BLE | 可隔空实时 | 需技適(或预认证模组)、射频/协议栈资源、App 复杂 | premium 备选,非首发 |
| Type-C 有线 | 免技適、带宽足 | 必须插线、iOS 受限、需 USB 器件 | 诊断候补 |

#### 后果

- NFC 只用 I²C、无射频栈 → **放宽 F030 量产 MCU 的资源压力**(不同于 BLE)。
- 被动标签、充电宝非主动发射机 → **基本免日本主动发射技適**(13.56MHz 仍受电波法约束,量产前须与认证机构确认口径)。
- ST25DV 作为 `drivers/` 下新增 I²C 外设,走现有 `hal/i2c`,**不改动现有分层**;电池状态数据模型与日志置于 `services/`。
- 遗留风险:13.56MHz 天线在金属外壳/电芯附近会失谐,读距须打样实测(铁氧体屏蔽、天线开窗/避让)。

## 3. 当前项目边界

### 3.1 仓库边界

```text
openBattery/
├── 0_System/          系统文档、架构规范、开发准备
├── 1_Plan/            周计划和任务拆解
├── 2_Action/          实施记录和验证记录
├── 90_documents/      外部文档、立项资料与参考教程
├── 91.reference/      外部资料归档
└── openBmsClaw/       当前 STM32 工程事实来源
```

边界规则：

- `openBmsClaw/` 是当前代码事实来源。
- `90_documents/` 是外部文档与立项/参考资料，不能直接当作当前实现或代码事实。
- `91.reference/` 是外部资料归档，不默认等于项目决策。
- `build/` 是生成物，不作为长期维护源码。

### 3.2 产品边界

`openBattery` 的目标不是先做满一台移动电源的所有功能，而是先形成可复用的开放平台骨架。

当前优先级：

1. 脑体协同架构正确。
2. SoC 接口抽象稳定。
3. MCU bring-up 可观测。
4. F030 量产候补约束明确。
5. 服务层策略能逐步验证。

## 4. 软件分层架构

### 4.1 分层总览

```text
app/             产品应用层：显示、按键、用户交互、状态页面
services/        系统服务层：功率分配、热管理、故障状态机
drivers/soc/     SoC 抽象层：统一 API、事件、vendor adapter
hal/             MCU 外设层：I2C、UART、ADC、GPIO、EXTI、watchdog
board/           板级支持层：引脚映射、板型能力、外设实例绑定
config/          配置层：功能裁剪、资源 profile、产品策略开关
```

架构原则：

> 上层只表达产品行为和系统策略，中间层屏蔽不同 SoC 差异，下层处理 MCU 外设和板级映射。

**NFC(ST25DV)落位:** 作为 `drivers/` 下新增的 I²C 外设驱动,走现有 `hal/` transport;电池状态数据模型(v0)与掉电日志置于 `services/`(如 `services/telemetry` 或数据/日志模块)。二者均不改动上述分层与依赖矩阵,ST25DV 与 SoC 同为 I²C 从设备。

### 4.2 允许依赖关系

| 当前层 | 可以依赖 | 禁止依赖 |
| --- | --- | --- |
| `app/` | `services/`、`config/`、应用级事件类型 | `hal/`、MCU 寄存器、SoC 寄存器、具体 vendor adapter |
| `services/` | `drivers/soc` 公共 API、`config/`、必要的 board capability 查询 | `hal_i2c.h`、`hal_adc.h`、具体 I2C 寄存器表 |
| `drivers/soc/` | `hal/` transport、SoC 类型定义、vendor adapter | `app/`、UI、业务页面 |
| `hal/` | MCU SDK / 寄存器 / 工具链生成代码 | `app/`、`services/`、SoC 业务语义 |
| `board/` | `hal/`、芯片引脚定义、板型能力描述 | 产品策略、功率分配算法 |
| `config/` | 无运行期依赖 | 不应包含可执行业务逻辑 |

### 4.3 严禁事项

1. `app/` 直接读写 SoC I2C 寄存器。
2. `services/` 直接 include `hal_i2c.h`、`hal_adc.h` 等 MCU 外设头文件。
3. SoC 公共 API 头文件暴露具体 HAL 句柄或 MCU 寄存器。
4. 某一颗 SoC 的寄存器名、地址、私有事件码进入 `app/` 或 `services/`。
5. 为了调试方便，把一次性测试 hook 留在正式业务路径中。
6. 使用多个配置宏表达同一个功能开关，造成配置双真相。

## 5. SoC 抽象层设计

### 5.1 目标

SoC 抽象层的目标不是把所有 SoC 能力一次性封装完整，而是先建立稳定的最小公共能力集。

最小公共能力集包括：

- 读取电压、电流、温度、端口状态
- 下发 OCP / 功率限制 / 端口使能配置
- 读取故障码
- 注册 SoC 告警事件回调
- I2C/UART 超时、错误计数和总线恢复

### 5.2 推荐结构

```text
drivers/soc/
├── soc_api.h              对 app/services 暴露的公共 API
├── soc_types.h            平台标准状态、错误码、事件类型
├── soc_manager.c          当前 SoC 实例、初始化、调度入口
├── soc_transport.h        I2C/UART 传输抽象
├── soc_transport_i2c.c    基于 hal_i2c 的传输实现
└── vendor/
    ├── injoinic_ip53xx.c  英集芯适配器
    └── zhirong_sw35xx.c   智融适配器
```

当前 startup 阶段可以先保留较薄的 `soc_sal.c`，但必须遵守两条退出条件：

1. 第一个真实 SoC 接入后，寄存器表和转换逻辑必须移入 vendor adapter。
2. 面向上层的公共头文件只能暴露平台类型和领域 API，不能暴露 `hal_i2c_t`、`I2C_TypeDef` 等底层细节。

### 5.3 公共 API 语义

公共 API 应描述平台领域能力，而不是描述某颗芯片的寄存器动作。

推荐表达：

```c
soc_status_t soc_get_voltage_mv(uint16_t port, uint16_t *voltage_mv);
soc_status_t soc_get_temperature_c(int16_t *temperature_c);
soc_status_t soc_set_port_power_limit(uint16_t port, uint32_t power_mw);
soc_status_t soc_set_ocp_limit(uint16_t port, uint16_t current_ma);
soc_status_t soc_poll_events(soc_event_t *events, uint8_t max_count);
```

> 单位口径：平台统一采用毫伏 (mV)、毫安 (mA)、毫瓦 (mW),温度用摄氏度整数。功率用 `uint32_t power_mw` 而非 `uint16_t`,因为 uint16 的 mW 上限约 65.5 W,无法表达 100 W 以上端口。当前代码事实是 `soc_sal` 基线仍用整数瓦 `soc_set_port_power(soc_port_t port, uint16_t max_power_w)`,与该口径不一致,需在 Phase 1 收敛为统一 mW。

不推荐表达：

```c
uint8_t ip53xx_read_reg(uint8_t reg);
void sw35xx_write_page(uint8_t page, uint8_t reg, uint8_t value);
```

## 6. 安全与异常处理架构

### 6.1 安全职责划分

移动电源产品不能把安全完全交给 MCU 软件判断。基础硬件保护必须由 SoC、保护芯片、保险丝、NTC 和电源路径硬件共同承担。

职责划分：

- SoC / 保护硬件：过压、过流、短路、过温等快速保护。
- MCU：读取状态、执行降额策略、锁存故障、提示用户、记录日志、恢复流程控制。
- `services/`：根据温度、端口、SoC 告警和产品策略做系统级仲裁。
- `app/`：只负责显示和用户交互，不直接决定硬件安全动作。

### 6.2 快路径与慢路径

紧急事件必须分成快路径和慢路径。

快路径：

- 来源：SoC INT、保护脚、EXTI。
- 目标：快速锁存状态，通知服务层进入保护状态。
- 禁止：I2C 读写、printf、delay、复杂算法、动态内存。

慢路径：

- 来源：主循环或服务层调度。
- 目标：读取 SoC 故障码、记录日志、更新 UI、决定恢复或保持锁定。
- 允许：I2C 读取、串口日志、错误码上报、状态页刷新。

## 7. 配置与资源 Profile

### 7.1 单一配置真相

每一个功能开关只能有一个权威来源。禁止同时存在多个含义相近但状态不一致的宏，例如一个文件启用 power service，另一个文件关闭 power feature。

推荐按 profile 管理：

```text
PROFILE_F103_BRINGUP       当前学习和基础外设验证
PROFILE_F030_MIN_PRODUCT   F030 最小量产候补
PROFILE_G0_EXTENDED        更大资源 MCU 的扩展功能
```

### 7.2 F030 最小量产 Profile

`PROFILE_F030_MIN_PRODUCT` 默认策略：

- 默认关闭 BLE、实时/远程监控、复杂 UI、大字库、双镜像 IAP。
- **默认启用 NFC(ST25DV)状态读出(I²C)** 作为数据可读主线。
- **默认启用关键计数/事件日志的掉电持久化(充放电次数、过温/故障),小体量,用于合规;不含大量历史序列。**
- 日志默认支持错误码和少量关键文本。
- 数据读出以 NFC tap 为主;设备端显示按需,优先 LED 最小指示,轻量 OLED 数字页为可选。
- 所有 I2C 操作必须有 timeout。
- 必须启用 watchdog。
- 编译后 Flash 至少保留 15% 余量，SRAM 必须完成最坏路径检查。

## 8. 实施路线与阶段门

### Phase 0：F103 bring-up

目标：

1. 稳定 `openBmsClaw/` 工程构建。
2. 跑通 LED、UART、I2C、ADC、EXTI、watchdog。
3. 建立日志、错误码、I2C timeout 和总线恢复。
4. 固定 `board / hal / drivers / services / app / config` 初始边界。

阶段门：

- 能稳定编译。
- UART 能输出启动日志和错误码。
- I2C 失败不会卡死主循环。
- 上层没有直接访问 SoC 寄存器。

### Phase 1：SoC SAL 基线

目标：

1. 定义 `soc_api.h` 和 `soc_types.h`。
2. 接入第一颗英集芯或智融 SoC。
3. 验证读取电压、温度、端口状态。
4. 验证 OCP 或功率限制参数下发。
5. 验证 SoC 告警 INT 和事件回调。

阶段门：

- `app/` 不 include `hal/`。
- `services/` 不 include `hal_i2c.h`。
- vendor 寄存器表不进入 `app/` 或 `services/`。
- I2C timeout、错误码和恢复路径可观察。

### Phase 2：服务层策略闭环

目标：

1. `srv_thermal` 根据 NTC 和 SoC 温度执行降额。
2. `srv_power_alloc` 根据端口状态执行功率分配。
3. 故障状态机支持锁定、提示、恢复条件。
4. 看门狗喂狗位置与故障恢复策略明确。

阶段门：

- 过温、过流、SoC 无响应都有明确状态。
- 快路径不执行慢操作。
- UI 只展示状态，不直接控制底层硬件。

### Phase 3：F030 样片验证

目标：

1. 至少验证一颗 STM32F030C8T6 级别基准器件。
2. 至少验证一颗国产 F030 候补器件。
3. 验证下载、SWD、UART、I2C、ADC、EXTI、watchdog。
4. 记录启动文件、链接脚本、HAL 或寄存器兼容差异。
5. 用真实编译结果评估 Flash / SRAM 余量。

阶段门：

- `PROFILE_F030_MIN_PRODUCT` 编译通过。
- Flash 至少保留 15%。
- SRAM 不发生栈或缓冲溢出。
- SoC 通信在异常场景下可恢复或可降级。

### Phase 4：最小量产链路

目标：

1. F030 + 专用 SoC + NFC(ST25DV) 完成最小闭环。
2. 完成热管理、功率降额、按键、LED 最小指示与 NFC tap 数据读出。
3. 整理 PSE、温升、异常保护、100Wh 策略,以及 NFC 天线读距/技適口径所需技术记录。

阶段门：

- 量产 MCU 型号不再写成“F030 类”，而是具体料号。
- SoC 型号、通信接口、关键寄存器/API 文档归档。
- 保护策略、异常处理和验证记录可追溯。

## 9. 技术债预防规则

startup 阶段允许出现临时验证代码，但必须被标记、隔离并有退出条件。

### 9.1 可接受的临时代码

- 用于验证 I2C 读写的 demo register。
- 用于验证 EXTI 的模拟触发函数。
- 用于 bring-up 的串口调试打印。
- 用于验证板级 LED 的直接调用。

### 9.2 不允许长期保留的债务

- `app/` 直接调用低层测试 hook。
- SoC 公共 API 头文件暴露 HAL 类型。
- vendor 寄存器表混在平台公共 SAL 中。
- 同一功能存在多个配置开关。
- 中断回调里执行 I2C、printf 或 delay。
- 为了当前板子方便，把板级细节写进服务策略。

### 9.3 每次进入新阶段前必须清理

1. 删除或隔离上一阶段的临时测试 hook。
2. 检查 include 方向是否符合依赖矩阵。
3. 检查配置开关是否只有一个权威来源。
4. 检查 Flash / SRAM 占用是否仍符合目标 profile。
5. 检查新增功能是否可裁剪。

## 10. 当前建议动作

短期按以下顺序推进：

1. 保持当前 `MCU + SoC` 架构不变，不推翻重来。

3. 收敛配置开关，建立 `F103_BRINGUP` 与 `F030_MIN_PRODUCT` 两类 profile。
4. 把 app 层直接触发底层测试 hook 的逻辑移入 bring-up/self-test 路径。
5. 每次新增服务或驱动时，先检查是否违反依赖矩阵。
6. 在接入真实 SoC 前，先明确最小 API 清单和错误码模型。

最终判断：

> 当前架构设计方向正确，startup 阶段的主要风险不是路线错误，而是分层规则没有被工程化执行。后续工作的重点是把“脑体分离”和“SoC 抽象”落实到头文件、目录、配置和阶段门中。
