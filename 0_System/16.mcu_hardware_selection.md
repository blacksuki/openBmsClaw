# openBattery MCU 硬件选型与 F030 量产候补评估

> 状态：Proposed / 待样片验证
> 最新核对日期：2026-06-25

本文用于明确 `openBattery` 当前 MCU 路线、F030 作为未来量产候补的边界，以及进入量产参考设计前必须完成的验证项。

本文结论先行：

- **当前学习和开发主线**：继续使用 `野火小智 STM32F103C8T6 核心板（双 USB 款）` 完成 bring-up、GPIO、UART、I2C、ADC、EXTI 和基础分层。
- **未来量产候补主线**：以 `F030` 级别 MCU 作为生产环境预计使用的低成本候补方向，优先评估 `STM32F030C8T6`、`HK32F030C8T6/R8T6`、`CKS32F030C8T6/R8T6` 等 64KB Flash 档位器件。
- **当前判断**：F030 **有条件符合**未来量产最低版本要求，但不适合作为“功能全开”的平台 MCU。它适合承担 SoC 协同控制、采样、状态机、保护策略和轻量交互；不适合承载 LVGL、大字库、复杂 BLE、双镜像 IAP 或重 UI。
- **关键前提**：电源转换、快充协议握手和大部分电芯基础保护必须继续交给专用 SoC；MCU 只做“脑力活”，不把底层电源闭环硬塞进 F030。

---

## 1. 当前主线与角色划分

### 1.1 当前学习和开发主线

当前阶段不立即切换物理芯片，仍围绕现有 `openBmsClaw/` 工程和 `STM32F103C8T6` 学习板推进。

这一阶段的目标是：

1. 跑通基础外设：`GPIO / LED / UART / I2C / ADC / EXTI`。
2. 固定软件边界：`board / hal / drivers(SAL) / services / app / config`。
3. 建立可观测性：串口日志、错误码、I2C 超时恢复、看门狗复位原因记录。
4. 为后续迁移到 F030 候补器件做资源裁剪准备。

这里的 `F103` 是学习和开发入口，不是当前量产定板结论。

### 1.2 未来量产候补主线

未来生产环境预计使用 `F030` 级别低成本 MCU 作为量产候补，原因是：

- 成本低，适合移动电源方案的量产 BOM 压力。
- Cortex-M0 生态成熟，工具链、调试器和裸机开发门槛较低。
- 外设能力覆盖 MCU 协同 SoC 的最低需求：I2C/UART、ADC、GPIO/EXTI、定时器、看门狗。
- 与当前 F103 学习阶段的基础外设训练路径相近，便于新手从 F103 迁移。

但 `F030` 只能定义为**量产候补**，不能在未完成样片验证前写成“已定型量产主控”。

---

## 2. F030 是否符合未来量产要求

### 2.1 结论

F030 符合 `openBattery` 未来量产最低版本要求，但必须限定产品边界。

它适合的量产版本是：

- 单 MCU + 专用电源 SoC 的双芯片协同架构。
- MCU 通过 `I2C` 或 `UART` 读取 SoC 状态并下发配置。
- MCU 负责按键、LED 或轻量 OLED 数字显示。
- MCU 负责 NTC 温度采样、阈值判断、降额策略和异常上报。
- MCU 负责轻量级功率分配策略，不直接承担高频电源控制闭环。

它不适合的量产版本是：

- MCU 直接实现复杂 BMS、电源转换或快充协议栈。
- 使用 LVGL、大字库、多语言复杂 UI。
- 集成 BLE 协议栈并同时保留复杂显示与 IAP。
- 双镜像 OTA/IAP、加密升级和大量日志持久化。
- 多颗 SoC、多屏、多通信外设同时在线的中高端平台版本。

因此，本项目对 F030 的定位是：

> `F030 = 成本敏感量产候补 MCU`，不是 `openBattery` 全功能平台上限。

### 2.2 最低资源要求

F030 候补器件至少应满足：

| 资源项 | 最低建议 | 原因 |
| --- | --- | --- |
| CPU | Cortex-M0 / 48MHz 级别 | 足够处理状态机、采样、I2C 通信和轻量策略 |
| Flash | 64KB 起步 | 低于 64KB 会压缩日志、IAP、显示和抽象层空间 |
| SRAM | 4KB 仅可做极简版，8KB 以上更稳 | 需要容纳栈、I2C 缓冲、事件队列和显示缓存 |
| I2C | 至少 1 路 | 连接英集芯/智融等电源 SoC 或传感器 |
| UART | 至少 1 路 | 调试日志或备用 SoC 通信 |
| ADC | 至少 4 路可用通道 | NTC、电压分压、电流采样或板级诊断 |
| EXTI/GPIO | 足够覆盖按键、SoC 告警、LED/OLED 控制 | 支撑交互和紧急告警 |
| Watchdog | 必须具备 | 防止 I2C 卡死或状态机异常后长期失控 |
| SWD | 必须具备 | 支撑下载、断点调试和量产调试 |

不建议把 16KB Flash / 2KB SRAM 级别的 `F030M` 小封装器件作为 `openBattery` 默认量产候补。它可以用于极简外设控制，但不适合承载本项目的 SoC SAL、服务层和可观测性要求。

## 3. 候选器件评估

### 3.1 候选档位

| 档位 | 候选器件 | 当前判断 |
| --- | --- | --- |
| 基准评估 | `STM32F030C8T6` | 作为 F030 生态和资源基准，用于判断国产替代器件是否真正可迁移 |
| 成本候补 A | `HK32F030C8T6/R8T6` | 可作为重点样片验证对象，需确认封装、引脚、Flash/SRAM、I2C、ADC、SWD 和工具链 |
| 成本候补 B | `CKS32F030C8T6/R8T6` | 可作为第二候选，需重点验证 SRAM 余量、启动文件、HAL/寄存器兼容性和调试稳定性 |
| 暂不推荐 | `HK32F030M`、16KB Flash 小资源 F030 | 资源过紧，不适合当前平台最低可观测和分层目标 |

### 3.2 不再使用的绝对化表述

本项目不再使用以下表述：

- “Pin-to-Pin 完全兼容”
- “软件零学习成本”
- “秒级平替”
- “采用 ARM 架构即可降低 PSE 认证风险”
- “仅需重写 hal/，上层完全不用修改”

更准确的表述是：

- F030 候补器件需要逐项验证封装、引脚、电源脚、BOOT、NRST、SWD、时钟、I2C、ADC、EXTI 和下载工具。
- Cortex-M0 生态能降低学习和工具链门槛，但不能自动保证 STM32 HAL、启动文件、链接脚本和寄存器行为完全兼容。
- PSE 关注的是产品安全、温升、保护、标识、测试和技术文件，不应把 MCU 指令集写成认证通过条件。
- 分层架构的目标是把迁移影响限制在 `board/hal/config/porting` 层，但实际迁移仍可能影响编译系统、启动文件、链接脚本和资源裁剪。

## 4. 软件架构约束

F030 能否进入量产候补，取决于软件是否从一开始按受限 MCU 设计。

```text
+-------------------------------------------------------------------+
| app/         轻量交互：按键、LED、基础 OLED 数字显示、状态页          |
+-------------------------------------------------------------------+
| services/    热管理、功率分配、故障状态机、看门狗喂狗策略             |
+-------------------------------------------------------------------+
| drivers/     SoC SAL：soc_get_voltage(), soc_get_temp(), soc_set_ocp() |
+-------------------------------------------------------------------+
| hal/         i2c, uart, adc, gpio, exti, watchdog                   |
+-------------------------------------------------------------------+
| board/       F103 学习板 / F030 量产候补板的引脚映射                 |
+-------------------------------------------------------------------+
| config/      资源裁剪、功能开关、合规策略、板型选择                  |
+-------------------------------------------------------------------+
```

### 4.1 上层约束

`app/` 和 `services/` 必须遵守：

1. 不直接访问 MCU 寄存器。
2. 不直接拼具体 SoC 的 I2C 寄存器读写细节。
3. 不假设一定有大 RAM、DMA、RTOS 或复杂显示缓存。
4. 所有功能必须可通过 `config/` 裁剪。

### 4.2 底层约束

`hal/`、`board/` 和 `config/` 必须承担：

1. 当前 F103 学习板和 F030 候补板的差异。
2. I2C 超时、总线恢复、错误计数和重初始化。
3. ADC 通道映射和采样滤波。
4. EXTI 告警输入和按键输入去抖。
5. Watchdog 初始化、喂狗位置和复位原因记录。

## 5. F030 量产候补验证清单

F030 进入量产参考设计前，必须完成以下验证。

### 5.1 芯片与板级验证

| 验证项 | 通过标准 |
| --- | --- |
| 封装和引脚 | 与目标板 PCB 约束匹配，电源脚、BOOT、NRST、SWD、晶振脚无冲突 |
| 下载调试 | `ST-LINK`、`J-Link` 或 DAPLink 至少一种可稳定下载和断点 |
| 时钟 | 内部 RC 和外部晶振策略明确，主频满足 48MHz 级别需求 |
| I2C | 能连续读写 SoC 寄存器，具备超时恢复和总线释放 |
| UART | 能稳定输出日志，波特率误差可接受 |
| ADC | NTC 和电压采样稳定，具备滤波和异常值抑制 |
| EXTI | SoC 告警脚和按键输入可靠触发，不误触发 |
| Watchdog | I2C 卡死和主循环卡死时能复位，复位原因可记录 |

### 5.2 固件资源验证

| 验证项 | 通过标准 |
| --- | --- |
| Flash | 开启 `hal + drivers + services + app_min + log_min` 后仍保留至少 15% 空间 |
| SRAM | 最坏路径下栈、缓冲、显示缓存和事件队列不溢出 |
| 日志 | 可裁剪为错误码模式，不依赖大字符串 |
| 显示 | OLED 仅保留基础数字/图标显示，不引入大字库 |
| IAP | 若启用 IAP，只支持单镜像或极简 Bootloader；双镜像升级需上移到更大 MCU |
| RTOS | 默认不依赖 RTOS；如启用 FreeRTOS，必须重新评估 SRAM |

### 5.3 产品级验证

| 验证项 | 通过标准 |
| --- | --- |
| SoC 协同 | 能读取电压、电流、温度、端口状态，并能下发限流/保护参数 |
| 热管理 | NTC 温度达到阈值后可下调功率或触发保护 |
| 异常处理 | I2C 失败、SoC 无响应、过温、过流、短路告警均有明确状态 |
| PSE 准备 | 软件保护策略、温升记录、异常处理和技术文件可追溯 |
| 100Wh 策略 | 容量与输出策略可配置，避免产品规格越过航空便携边界 |

## 6. 演进路线

### 6.1 Phase 0：当前 F103 学习和开发

目标：

1. 稳定 `openBmsClaw/` 当前工程。
2. 完成 `GPIO / UART / I2C / ADC / EXTI / watchdog` 最小验证。
3. 建立 `board / hal / drivers / services / app / config` 的目录和接口边界。
4. 所有新增功能默认考虑 64KB Flash 级别裁剪。

### 6.2 Phase 1：F030 样片验证

目标：

1. 采购或打样 `STM32F030C8T6`、`HK32F030C8T6/R8T6`、`CKS32F030C8T6/R8T6` 至少两类样片。
2. 跑通下载、串口日志、I2C、ADC、EXTI 和 watchdog。
3. 对比启动文件、链接脚本、HAL/寄存器兼容性和调试稳定性。
4. 输出 `F030_porting_notes.md`，记录不能兼容和需要绕开的点。

### 6.3 Phase 2：F030 + 专用 SoC 最小量产链路

目标：

1. 接入首个英集芯或智融 SoC。
2. 跑通 `soc_get_voltage()`、`soc_get_temp()`、`soc_set_ocp()`、事件告警回调。
3. 验证热管理、功率降额、按键/LED/OLED 最小交互。
4. 用真实编译结果判断 64KB Flash 和 SRAM 是否可接受。

### 6.4 Phase 3：量产参考板决策

只有当 F030 样片通过资源、外设、SoC 协同和产品级保护验证后，才能把某个具体 F030 型号写入量产参考设计。

若验证失败，回退路线是：

- 保留 F103 作为学习开发入口。
- 将量产 MCU 上移到 `STM32G0`、`GD32E230` 或其它资源更宽裕的 Cortex-M0/M0+ 器件。
- 继续保持 `board/hal/config` 与上层业务分离，避免推倒重来。

## 7. 当前结论

当前项目主线更新为：

```text
F103 bring-up（当前学习和开发）
        ↓
F030 量产候补（生产环境预计使用，需样片验证）
        ↓
F030 + 专用 SoC 最小量产链路
        ↓
通过验证后再写入量产参考设计
```

F030 的能力结论：

- **符合**：轻量 SoC 协同控制、NTC 采样、I2C/UART 通信、LED/基础 OLED、按键、保护状态机、看门狗。
- **勉强**：极简 IAP、极简日志、有限事件队列、简单功率分配。
- **不符合**：复杂 UI、LVGL、大字库、BLE 协议栈、双镜像 OTA、完整平台全功能版本。

因此，F030 可以作为 `openBattery` 面向成本敏感量产版本的候补主控，但必须通过样片验证和资源预算验证后，才能从“候补”升级为“默认量产参考设计”。
