# 必备软件安装指南

> 适用范围：基于 `0_System/11.prepare_dev.md` 第 5 节“必备软件准备”  
> 最新核对日期：2026-06-25

## 1. 文档目的
本文档覆盖当前项目启动阶段的软件安装方案，并明确三类工具：

- **默认主路径**：`STM32CubeIDE` 桌面版
- **可选主路径**：`STM32CubeIDE for Visual Studio Code`
- **当前工程终端构建链**：`CMake` + `Ninja` + `arm-none-eabi-*`

前两条路径用于 IDE 级开发、调试和团队 onboarding。第三条路径用于当前仓库内 `openBmsClaw/` 工程的命令行构建，是当前代码事实要求的工具链。

除 IDE 路径外，以下工具属于当前项目的通用必备项：

- `GNU Arm Embedded Toolchain` / `GNU Tools for STM32`
- `CMake`
- `Ninja`
- `STM32CubeMX`
- `STM32CubeProgrammer`
- `Git`

当前工程验证过的失败点是：如果 `arm-none-eabi-gcc` 与 `arm-none-eabi-g++` 不在 `PATH` 中，`cmake --preset Debug` 会在配置阶段失败。因此，新环境必须先完成终端工具链验证，再认为开发环境可用。

---

## 2. 当前建议基线
截至 `2026-03-25`，建议按以下基线准备环境：

| 软件 | 当前建议基线 | 角色 |
| --- | --- | --- |
| `STM32CubeIDE` | `2.1.1` | 默认主开发环境 |
| `Visual Studio Code` | 当前稳定版 | VS Code 主环境 |
| `STM32CubeIDE for Visual Studio Code` | ST 官方扩展当前版 | 可选主开发环境 |
| `GNU Arm Embedded Toolchain` / `GNU Tools for STM32` | `13.3.1` 或更高 | 当前 `openBmsClaw` 交叉编译器 |
| `CMake` | `3.20` 或更高 | 当前 `openBmsClaw` 工程配置工具 |
| `Ninja` | 当前稳定版 | 当前 `openBmsClaw` 构建器 |
| `STM32CubeMX` | `6.17.0` | 外设配置与工程生成 |
| `STM32CubeProgrammer` | `2.22` | 烧录、擦写、验证 |
| `Git` | `2.53.0` | 版本管理 |

说明：

- `STM32CubeIDE 2.x` 已与 `STM32CubeMX` 解耦，因此两者必须分别安装。
- `STM32CubeIDE for Visual Studio Code` 是 ST 官方提供的 VS Code 扩展方案，可以创建/导入、构建和调试 STM32 工程。
- 当前 `openBmsClaw/cmake/gnu-tools-for-stm32.cmake` 明确要求 `arm-none-eabi-` 工具前缀必须在 `PATH` 中。
- 当前 `openBmsClaw/CMakePresets.json` 使用 `Ninja` 作为 CMake generator，因此必须安装 `Ninja`。
- 对当前项目，若使用 VS Code 路线，仍建议保留 `STM32CubeMX`，因为当前 F103 学习工程、后续 F030 候补验证，以及必要时上移到 STM32G0/GD32E230 等资源更宽裕平台，都需要图形化外设配置或工程生成能力。

## 3. 路径选择建议

### 3.1 默认主路径：STM32CubeIDE 桌面版
适合：

- 团队统一 onboarding
- 早期 bring-up
- 按 ST 官方最成熟路径推进
- 降低工具链分歧

### 3.2 可选主路径：STM32CubeIDE for Visual Studio Code
适合：

- 已经以 VS Code 为主要编辑环境的开发者
- 希望使用 VS Code 工作流，但仍保留 ST 官方 STM32 工具链支持
- 需要官方支持的 VS Code 路线，而不是第三方拼装方案

### 3.3 当前项目建议
当前项目建议这样落地：

- 团队默认文档、培训和交付，优先按 **桌面版 `STM32CubeIDE`** 编写
- 若个人开发者偏好 VS Code，可采用 **`STM32CubeIDE for Visual Studio Code`** 路线
- 不建议在文档中把 VS Code 路线直接替换桌面版，而应并列为两条受支持路径

## 4. 推荐安装顺序

### 4.1 路径 A：桌面版默认路径
建议顺序：

1. `STM32CubeIDE`
2. `GNU Arm Embedded Toolchain` / `GNU Tools for STM32`
3. `CMake`
4. `Ninja`
5. `STM32CubeMX`
6. `STM32CubeProgrammer`
7. `Git`

### 4.2 路径 B：VS Code 可选路径
建议顺序：

1. `Visual Studio Code`
2. `STM32CubeIDE for Visual Studio Code`
3. `GNU Arm Embedded Toolchain` / `GNU Tools for STM32`
4. `CMake`
5. `Ninja`
6. `STM32CubeMX`
7. `STM32CubeProgrammer`
8. `Git`

### 4.3 当前仓库终端构建路径
如果目标是先把当前 `openBmsClaw/` 工程在终端里构建起来，最小顺序如下：

1. `GNU Arm Embedded Toolchain` / `GNU Tools for STM32`
2. `CMake`
3. `Ninja`
4. `Git`
5. `STM32CubeProgrammer`

说明：

- 两条路径都建议单独安装 `STM32CubeMX` 和 `STM32CubeProgrammer`
- VS Code 路线下，`STM32CubeMX` 不是“IDE 本体的一部分”，仍应单独安装
- `STM32CubeIDE` 可能自带或管理一部分工具，但当前仓库的 CMake 工程要求终端能直接找到 `arm-none-eabi-gcc`、`cmake` 和 `ninja`
- 如果只使用桌面版 IDE 构建，可以暂时不走命令行；但本仓库的当前验收标准仍以命令行 `cmake --preset Debug` 可通过为准

## 5. 当前工程命令行工具链安装步骤

### 5.1 当前工程为什么需要这些工具
当前 `openBmsClaw/` 工程不是纯 IDE 工程，已经包含：

- `openBmsClaw/CMakePresets.json`：指定使用 `Ninja` 作为构建器。
- `openBmsClaw/cmake/gnu-tools-for-stm32.cmake`：指定使用 `arm-none-eabi-gcc`、`arm-none-eabi-g++`、`arm-none-eabi-objcopy` 和 `arm-none-eabi-size`。
- `openBmsClaw/CMakeLists.txt`：集中列出当前源码并生成 `.elf`、`.hex`、`.bin`。

因此，只安装 `STM32CubeIDE`、`STM32CubeMX` 和 `STM32CubeProgrammer` 不一定足够。终端中必须能执行以下命令：

```bash
arm-none-eabi-gcc --version
arm-none-eabi-g++ --version
arm-none-eabi-objcopy --version
arm-none-eabi-size --version
cmake --version
ninja --version
```

### 5.2 macOS 推荐安装方式
当前开发机以 macOS 为主，推荐先用 Homebrew 安装通用工具：

```bash
brew install cmake ninja git
```

然后安装 Arm 交叉编译器，优先选择以下任一方式：

1. 安装 ST 官方 `STM32CubeCLT`，并把其中的 GNU Tools for STM32 `bin` 目录加入 `PATH`。
2. 安装 Arm 官方 GNU Arm Toolchain，并把其 `bin` 目录加入 `PATH`。
3. 如 Homebrew 可用，也可以尝试安装提供 `arm-none-eabi-gcc` 的工具链包。

如果安装后终端仍找不到 `arm-none-eabi-gcc`，需要在 `~/.zshrc` 中加入工具链 `bin` 目录，例如：

```bash
export PATH="/path/to/arm-gnu-toolchain/bin:$PATH"
```

修改后执行：

```bash
source ~/.zshrc
```

### 5.3 Windows 推荐安装方式
Windows 环境建议：

1. 安装 `STM32CubeCLT` 或 Arm 官方 GNU Arm Toolchain。
2. 安装 `CMake`。
3. 安装 `Ninja`。
4. 确认 `arm-none-eabi-*`、`cmake`、`ninja` 所在目录已经加入系统 `PATH`。
5. 重新打开 `PowerShell` 或命令提示符后执行版本验证。

可使用 `where` 检查命令来源：

```powershell
where arm-none-eabi-gcc
where cmake
where ninja
```

### 5.4 安装后验证
安装完成后，先验证工具本身：

```bash
arm-none-eabi-gcc --version
arm-none-eabi-g++ --version
arm-none-eabi-objcopy --version
arm-none-eabi-size --version
cmake --version
ninja --version
git --version
```

再验证当前工程：

```bash
cd /Users/huoward/Project/11.openBattery/openBmsClaw/openBmsClaw
cmake --preset Debug
cmake --build --preset Debug
```

预期结果：

1. `cmake --preset Debug` 能完成配置，不再报 `arm-none-eabi-gcc was not found in the PATH`。
2. `cmake --build --preset Debug` 能开始编译，并在成功时生成 `.elf`、`.hex` 和 `.bin`。
3. 若出现源码编译错误，应按源码问题处理；若仍提示找不到工具，则继续检查 `PATH`。

## 6. STM32CubeIDE 桌面版安装步骤

### 6.1 Windows
1. 打开 ST 官方 `STM32CubeIDE` 下载或发布页面。
2. 确认当前版本号。
3. 下载对应的 Windows 安装包。
4. 关闭已经运行的 ST 工具。
5. 双击安装包，按向导完成安装。
6. 首次启动 `STM32CubeIDE`。
7. 选择本地 workspace 目录。

### 6.2 macOS
1. 打开 ST 官方 `STM32CubeIDE` 下载或发布页面。
2. 确认当前版本号。
3. 下载对应的 macOS 安装包。
4. 按安装向导完成安装。
5. 首次启动 `STM32CubeIDE`。
6. 选择本地 workspace 目录。

### 6.3 安装后验证
至少完成以下验证：

1. `STM32CubeIDE` 能正常启动。
2. 能创建或打开一个 workspace。
3. “Help > Check for updates” 菜单可用。

### 6.4 注意事项
- `STM32CubeIDE 2.x` 不再内置 `STM32CubeMX`。
- ST 官方说明中，`Help > Check for updates` 可更新当前安装，但 **不会更新驱动和 STLINK server tool**；若需要这些组件同步更新，应执行完整安装。
- 当前项目中，桌面版仍是默认主路径。

## 7. STM32CubeIDE for Visual Studio Code 安装步骤

### 7.1 适用性结论
ST 官方已经提供 `STM32CubeIDE for Visual Studio Code` 扩展、官方在线文档和安装说明；它可以作为 STM32 开发工具使用。  
但对当前项目，建议它作为 **可选主路径**，而不是直接替换桌面版 `STM32CubeIDE`。

### 7.2 前置条件
安装前应准备：

1. `Visual Studio Code`
2. 网络可访问 VS Code Marketplace
3. 后续调试所需的调试探针驱动安装能力

### 7.3 安装步骤
1. 安装 `Visual Studio Code`。
2. 打开 VS Code。
3. 进入 Extensions 视图。
4. 搜索并安装 ST 官方扩展：`STM32CubeIDE for Visual Studio Code`。
5. 安装完成后，按 ST 官方文档要求完成首次初始化。

### 7.4 安装后验证
至少完成以下验证：

1. 在 VS Code 中能看到 ST 官方 STM32 扩展已启用。
2. 能打开 ST 官方提供的 STM32 工作流入口。
3. 能创建空工程，或导入已有 STM32 工程。

### 7.5 注意事项
- ST 官方 Marketplace 页面明确说明，该扩展是 **ST 官方出品** 的 VS Code IDE 方案。
- ST 官方文档说明它支持创建、导入、构建和调试 STM32 工程。
- ST 官方发布说明中提到，**调试探针驱动仍需要手动安装**。
- ST 官方同时说明，在线文档会持续补充，因此团队内部仍应保留一条更稳妥的桌面版默认路径。

### 7.6 当前项目建议的使用方式
若采用 VS Code 路线，建议：

1. 把它作为个人开发工具选择，而不是强制团队统一唯一工具。
2. 仍单独安装 `STM32CubeMX`，用于当前 F103、后续 F030 候补和必要的回退平台图形化配置与工程生成。
3. 保留 `STM32CubeProgrammer` 作为烧录和连接验证工具。

## 8. STM32CubeMX 安装步骤

### 8.1 Windows
1. 打开 ST 官方 `STM32CubeMX` 下载或发布页面。
2. 确认当前版本号。
3. 下载对应的 Windows 安装包。
4. 双击安装包，按安装向导完成安装。
5. 首次启动 `STM32CubeMX`。
6. 在需要生成代码前，打开软件包管理器并安装目标 MCU 系列软件包。

### 8.2 macOS
1. 打开 ST 官方 `STM32CubeMX` 下载或发布页面。
2. 确认当前版本号。
3. 下载对应的 macOS 安装包。
4. 按安装向导完成安装。
5. 首次启动 `STM32CubeMX`。
6. 在需要生成代码前，安装目标 MCU 系列软件包。

### 8.3 安装后验证
至少完成以下验证：

1. `STM32CubeMX` 能正常启动。
2. 能创建基于 MCU 或 Board 的新工程。
3. 能打开嵌入式软件包管理器。
4. 能安装目标系列软件包。

### 8.4 当前项目建议的首次配置
1. 安装当前 F103 工程对应的软件包，并准备后续 F030 候补器件对应的软件包。
2. 建一个基于当前 F103 或目标 F030 候补器件的最小工程；若 F030 验证失败，再建立 STM32G0/GD32E230 等回退平台工程。
3. 若使用桌面版路径，在 Project Manager 中将目标工具链设置为 `STM32CubeIDE`。
4. 若使用 VS Code 路径，可生成并导入 ST 支持的工程工作流。

### 8.5 注意事项
- `STM32CubeMX` 本身只是图形化配置和代码生成工具。
- 安装完成并不等于已经具备目标 MCU 的软件包。
- ST 官方资料明确要求通过软件包管理入口安装目标系列包或扩展包。

## 9. STM32CubeProgrammer 安装步骤

### 9.1 Windows
1. 打开 ST 官方 `STM32CubeProgrammer` 下载或发布页面。
2. 确认当前版本号。
3. 下载对应的 Windows 安装包。
4. 双击安装包，按向导完成安装。
5. 首次启动 `STM32CubeProgrammer`。

### 9.2 macOS
1. 打开 ST 官方 `STM32CubeProgrammer` 下载或发布页面。
2. 确认当前版本号。
3. 下载对应的 macOS 安装包。
4. 按安装向导完成安装。
5. 首次启动 `STM32CubeProgrammer`。

### 9.3 安装后验证
至少完成以下验证：

1. `STM32CubeProgrammer` 可以正常启动。
2. 接入 `ST-LINK` 后，工具能识别连接入口。
3. 在 GUI 中可以看到 `Connect` 操作入口。

### 9.4 当前项目建议的首次验证
1. 将当前 F103 学习板、F030 候补板或其他 STM32 板通过 `ST-LINK` 连接到电脑。
2. 打开 `STM32CubeProgrammer`。
3. 选择调试连接方式。
4. 点击 `Connect`，确认工具能够连上目标板。

### 9.5 注意事项
- `STM32CubeProgrammer` 是独立工具，适用于 Windows、Linux、macOS。
- ST 官方资料明确区分了 GUI 与命令行两种使用方式。
- 当前项目启动阶段，先用 GUI 跑通，再考虑脚本化。

## 10. Git 安装步骤

### 10.1 Windows
推荐两种官方支持路径，二选一即可。

#### 路径 A：使用 winget
1. 打开 `PowerShell` 或命令提示符。
2. 执行：

```bash
winget install --id Git.Git -e --source winget
```

3. 安装完成后，重新打开终端。
4. 执行 `git --version` 验证安装结果。

#### 路径 B：使用官方安装器
1. 打开 Git 官方下载页面。
2. 下载最新的 Windows 安装器。
3. 按安装向导完成安装。
4. 安装完成后，重新打开终端。
5. 执行 `git --version` 验证安装结果。

### 10.2 macOS
推荐两种官方支持路径，二选一即可。

#### 路径 A：安装 Xcode Command Line Tools
1. 打开终端。
2. 执行：

```bash
git --version
```

3. 如果系统提示安装 Command Line Tools，则按提示完成安装。
4. 安装完成后再次执行 `git --version`。

也可以直接执行：

```bash
xcode-select --install
```

#### 路径 B：使用 Homebrew
1. 确认系统已安装 Homebrew。
2. 执行：

```bash
brew install git
```

3. 安装完成后执行 `git --version`。

### 10.3 首次配置
安装完成后，按 Git 官方建议至少完成以下全局配置：

```bash
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
```

如需查看当前配置，可执行：

```bash
git config --list --show-origin
```

### 10.4 注意事项
- Git 官方文档指出，macOS 上很多非源码发行版由第三方维护，版本节奏可能与源码版不同。
- 若团队要求统一版本，应在团队文档中明确 Git 安装路径和最低版本。
- 安装完成后必须至少验证 `git --version` 和 `user.name` / `user.email` 配置。

## 11. 统一安装后的最小验收清单
无论使用哪条 IDE 路线，至少做一次完整验收：

1. 默认路径下：`STM32CubeIDE` 能启动并创建 workspace。
2. 可选路径下：VS Code 中的 `STM32CubeIDE for Visual Studio Code` 能正常启用。
3. `arm-none-eabi-gcc --version` 正常返回版本号。
4. `arm-none-eabi-g++ --version` 正常返回版本号。
5. `arm-none-eabi-objcopy --version` 正常返回版本号。
6. `arm-none-eabi-size --version` 正常返回版本号。
7. `cmake --version` 正常返回版本号。
8. `ninja --version` 正常返回版本号。
9. `STM32CubeMX` 能启动并安装当前 F103、后续 F030 候补或回退平台所需的软件包。
10. `STM32CubeMX` 能生成一个最小工程，且可被所选 IDE 路线使用。
11. `STM32CubeProgrammer` 能通过 `ST-LINK` 连接目标板。
12. `git --version` 正常返回版本号。
13. `git config --list --show-origin` 能看到用户身份配置。
14. 当前工程能完成 CMake 配置：

```bash
cd /Users/huoward/Project/11.openBattery/openBmsClaw/openBmsClaw
cmake --preset Debug
```

15. 当前工程能启动构建：

```bash
cmake --build --preset Debug
```

如果第 14 步仍出现 `arm-none-eabi-gcc was not found in the PATH`，说明 Arm 交叉编译器没有正确安装或没有加入 `PATH`，不应继续排查源码。
