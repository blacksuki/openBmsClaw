## 1. 项目摘要与当前结论

本文档用于将当前分散在 `0_System/00.tech_architecure.md`、`0_System/01.update_arch.md`、`0_System/10.hardware_softare_frame.md` 中的主线内容收敛为一份统一文档，作为 `openBattery` 当前阶段的总文档。

本稿坚持以下约束：

- 不把未验证的市场判断写成既成事实。
- 不把抽象合作思路写成可直接执行的工程方案。
- 不把过期文档状态写成当前代码事实。
- 所有关键结论尽量区分为“已确认输入”“当前假设”“待验证项”“工程要求”。

当前可收敛的核心结论如下：

1. `openBattery` 当前服务的首要客户定位为**日本中端客户**。
2. 当前首发主场景定位为**磁吸便携**。
3. 当前产品方向不是做传统黑盒充电宝，而是围绕“**安全 + 性能数字化管理**”形成差异化。
4. 当前技术路线继续采用 **MCU + 专用电源 SoC** 的双芯协同架构。
5. 当前 SoC 合作主线已统一到 **英集芯**，后续合作前提不是泛泛理念，而是能够提出可工程化落地的接口、边界和验证要求。
6. 当前团队出身以软件开发为主，对嵌入式、硬件保护、电源系统、热设计和量产工程的理解仍不足，因此文档必须主动标出能力边界与风险点。

### 1.1 已确认输入

- 当前项目主名称统一为 `openBattery`。
- 当前代码事实入口为 `openBmsClaw/`。
- 当前学习与 bring-up 基线为 `STM32F103C8T6`。
- 当前长期量产候补优先评估 `F030` 档位 MCU，但尚未完成样片验证。
- 当前 SoC 供应链合作对象按用户给定信息，已统一为 **英集芯**。
- 当前首要客户与主场景按用户给定信息，分别为**日本中端客户**与**磁吸便携**。

### 1.2 当前假设

- 日本中端客户会为“安全状态可视、性能可解释、体验更精细”的移动电源方案付费。
- 磁吸便携场景足以作为首发产品与平台设计的共同牵引场景。
- 英集芯现有能力可以支撑当前设想中的接口开放深度、事件模型与参数配置需求。

### 1.3 待验证项

- 上述市场判断是否有足够用户、渠道、竞品与价格带证据支撑。
- 磁吸便携方案对结构、热设计、认证与续航的要求是否与当前架构设想一致。
- 英集芯可提供的 API、文档、事件、调试支持是否达到平台化合作要求。
- `F030` 是否能承接“最小量产版本”所需的资源与可靠性要求。

## 2. 市场理解与用户洞察

本章回答的问题是：为什么当前项目主线不是“做一个参数更高的移动电源”，而是“围绕日本中端市场做安全与性能数字化管理的磁吸便携方案”。

### 2.1 已确认输入

- 当前项目文档主线明确面向**日本市场**。
- 当前差异化方向集中在“安全 + 性能数字化管理”，而不是单纯追求规格堆叠。
- 当前首发主场景由项目输入明确为**磁吸便携**。

### 2.2 当前理解

基于现有文档和当前项目输入，可以先收敛出如下理解：

1. 日本市场的产品说服力不能只来自参数表，还需要来自**安全感、合规感、做工感和体验解释力**。
2. 对于磁吸便携场景，用户感知最直接的，不是底层协议有多复杂，而是：
   - 是否稳定吸附和便于携带；
   - 是否安全不过热；
   - 是否能清楚理解剩余可用时间、当前功率状态与异常原因；
   - 是否在多设备、多状态切换时表现稳定。
3. 因此，`openBattery` 的差异化重点应落在：
   - 安全状态可视；
   - 性能状态可视；
   - 动态功率调度；
   - 面向日本市场约束的裁剪能力；
   - 可复用的平台骨架。

### 2.3 当前假设

以下内容当前仍属于项目假设，不应直接写成已证实事实：

- 日本中端客户是否明确偏好“数字化显示”而非仅接受“隐形稳定”。
- 磁吸便携是否优先于更大功率或更多接口数量，成为更高优先级购买驱动因素。
- “安全与性能数字化管理”在日本市场是否足以形成可感知溢价。

### 2.4 待验证项

为防止市场理解停留在团队推演层，本章结论至少需要以下证据之一支撑：

- 日本市场竞品梳理：价格带、卖点、显示方式、结构形态、磁吸方案占比。
- 用户访谈或渠道反馈：购买决策因素、退货/投诉因素、对发热与安全提示的敏感度。
- 合规与渠道信息：PSE、航空携带、包装/宣传限制是否直接影响产品定义。
- 样机测试：磁吸便携场景下的热、厚度、重量、吸附强度、续航、显示可读性。

## 3. 产品定义与首发范围

本章回答的问题是：当前项目到底在定义一个什么产品，第一阶段做什么，不做什么。

### 3.1 产品定义

`openBattery` 当前不是先做一台功能堆满的移动电源整机，也不是一上来就做大而全的平台，而是先用一个产品样板证明“安全 + 性能数字化管理”以及 `MCU + SoC` 双芯分工里最关键的价值，再把其中可复用的部分沉淀成平台能力。

当前更准确的定义应为：

> 围绕日本中端客户的磁吸便携场景，先做一个以“安全 + 性能数字化管理”为差异化特征、以 `MCU + SoC` 双芯协同为底座的最小可验证产品样板，验证关键平台能力；在工程上从第一天预留 `SoC` 抽象边界，但在业务推进上先不做多 SoC 并行适配，再逐步沉淀为开放电源平台。

### 3.2 首发范围内的内容

基于现有文档与当前输入，首发范围应优先包括：

- 适配磁吸便携场景的基础硬件架构。
- 基于 SoC 的电源转换、协议和基础保护能力。
- 由 MCU 承担的状态读取、策略仲裁与显示交互。
- 最小可用的状态可视化能力，例如电压、温度、功率、剩余状态提示。
- 基础安全与异常策略：过温、异常状态提示、降额或锁定逻辑。

### 3.3 暂不应默认进入首发范围的内容

以下内容即使长期可能有价值，当前也不应默认进入首发版本：

- 复杂 BLE 协议栈。
- 大型 UI 框架与重显示资源。
- 双镜像 IAP / OTA。
- 多颗 SoC、多屏、多通信外设同时在线。
- App、底座、无线充电、多档量产路线等第二阶段扩展项。
- 未明确需求来源的复杂扩展功能。

### 3.4 当前假设

- 首发产品与后续平台能力可以共用同一套核心架构，但工程推进必须坚持“先产品、后平台”。
- 首期软件从第一天就保留 `SAL` 抽象边界，但只实现 1 个 SoC vendor adapter，不做多 SoC 并行适配。
- 磁吸便携首发场景足以覆盖后续平台抽象中最关键的安全、热、交互与功率调度问题。

### 3.5 待验证项

- 首发 SKU 的目标容量、功率档位、厚度与重量边界。
- 磁吸结构方案是否需要明确对齐某一具体兼容标准。
- 首发用户可见功能中，哪些是真正必须有，哪些只是“看起来先进”。

## 4. 商业角色与合作定位

本章回答的问题是：`openBattery` 在产业链中是谁，以什么身份与 SoC 原厂及其他合作方协作。

### 4.1 当前定位

当前更合理的项目定位不是“芯片原厂替代者”，也不是“单一成品品牌固件团队”，而是：

> 面向日本中端市场的产品定义与系统整合方。

这个定位意味着我们负责的是：

- 对目标市场与场景做收敛；
- 把产品需求转换为可实现的系统架构；
- 在 SoC、MCU、显示、结构、热设计与软件之间完成系统整合；
- 把数字化可视、安全逻辑和产品体验做成可复用平台能力。

### 4.2 与英集芯合作的目标

既然当前 SoC 合作已统一到英集芯，那么合作目标不应停留在“理念一致”，而应明确为：

- 让原厂理解我们并非索要源码，而是需要可工程化集成的接口能力；
- 让原厂看到我们具备足够的系统理解，知道哪些能力由 SoC 承担，哪些能力由 MCU 承担；
- 让合作建立在可验证的交付物上，而不是建立在口头承诺上。

### 4.3 当前风险

如果这一章不写清楚，后续最容易出现三类偏差：

1. 我方把原厂能力想象得过于开放，导致平台设计依赖了拿不到的接口。
2. 原厂把我方理解为“只会提需求的软件团队”，不愿投入足够技术支持。
3. 双方都没有把工程边界说清楚，最终把系统集成问题变成互相推责。

## 5. 硬件总体架构

本章回答的问题是：当前产品和平台从硬件结构上如何成立，以及哪些职责必须由硬件承担。

### 5.1 当前架构结论

当前硬件主线继续采用 **MCU + 专用电源 SoC** 双芯协同架构：

- SoC：负责升降压、电源路径、快充协议握手和基础硬件保护。
- MCU：负责状态读取、策略仲裁、显示交互、日志与系统级异常处理。

该结论来自现有主线文档，且与当前代码和计划保持一致。

### 5.2 硬件角色划分

在当前文档口径下，硬件角色可先明确为：

- 电芯与相关保护件：提供基础储能与物理安全约束。
- 专用电源 SoC：承担电源转换、协议、端口控制与基础保护。
- MCU：承担上层逻辑、状态读取、策略调度与可视化。
- NTC / 温度采样：为热管理与安全策略提供输入。
- 显示 / LED / 按键：承接用户可见交互。
- 磁吸结构相关件：服务首发磁吸便携场景，但具体结构方案仍待验证。

### 5.3 安全职责边界

当前必须明确的原则是：

- 安全不能只依赖 MCU 软件判断。
- 快速保护必须由 SoC、保护硬件、保险丝、NTC 和电源路径硬件共同承担。
- MCU 只能在较慢的系统层做读取、降额、锁定、提示和恢复控制。

### 5.4 当前假设

- 英集芯现有 SoC 能覆盖当前磁吸便携方案所需的电源转换、协议与基础保护能力。
- 首发产品所需的差异化并不来自自研电源闭环，而来自更好的系统控制与展示。

### 5.5 待验证项

- 磁吸便携结构对热路径、厚度、电芯布局和 SoC 选型的约束。
- 高功率与小体积共存时的温升与保护策略是否可接受。
- 量产设计中还需要哪些额外硬件保护件，才能满足合规与实际使用风险。

## 6. 软件总体架构

本章回答的问题是：软件如何支撑产品定义，并把 SoC 差异、板级差异和产品逻辑分层隔离。

### 6.1 当前架构结论

当前软件分层继续采用如下主线：

```text
app/             产品应用层：显示、按键、用户交互、状态页面
services/        系统服务层：功率分配、热管理、故障状态机
drivers/soc/     SoC 抽象层：统一 API、事件、vendor adapter
hal/             MCU 外设层：I2C、UART、ADC、GPIO、EXTI、watchdog
board/           板级支持层：引脚映射、板型能力、外设实例绑定
config/          配置层：功能裁剪、资源 profile、产品策略开关
```

这一分层来自现有主线文档，并且当前代码目录已经建立了相应骨架。

### 6.2 分层原则

当前应继续坚持以下原则：

- `app/` 只表达产品行为与用户交互。
- `services/` 只表达系统策略与状态仲裁。
- `drivers/soc/` 统一屏蔽不同 SoC 的寄存器或 SDK 差异。
- `hal/` 只处理 MCU 外设与传输细节。
- `board/` 只处理板型和引脚映射。
- `config/` 只承接配置裁剪，不承载可执行策略逻辑。

### 6.3 SoC 抽象层（SAL）的意义

当前路线中，SoC 抽象层不是“写得更好看”的封装，而是整个合作模式和平台化能力的基础：

- 它决定上层是否会被某一家 SoC 厂商绑定。
- 它决定后续能否用统一方式处理状态读取、参数下发与异常事件。
- 它决定与供应商谈判时，需求是否能被表达为明确的接口能力。

### 6.4 安全与异常处理的软件边界

当前软件必须明确区分快路径与慢路径：

- 快路径：SoC INT、保护脚、EXTI 触发后，只做快速锁存和状态切换。
- 慢路径：主循环或服务层调度中读取状态、记录日志、更新 UI、执行恢复逻辑。

禁止把 I2C 读写、`printf`、延时、复杂算法放进中断快路径。

### 6.5 当前代码事实

基于当前仓库代码与 `Plan-StageA-Implementation-0714.md`，可确认以下事实：

- `openBmsClaw/` 中已存在 `board/`、`hal/`、`drivers/soc/`、`services/` 等目录骨架。
- `soc_sal_init()` 已进入 `bringup_service` 的真实运行路径。
- `soc_sal_is_initialized()` 已作为运行期观测接口存在。
- `bringup_service` 已具备 SoC 初始化、在线轮询与 EXTI 自测相关日志。
- 当前仍缺少真实 SoC 的实板日志、逻辑分析仪证据与 `2_Action` 记录闭环。

### 6.6 当前假设

- 当前软件分层足以承接首发产品与后续平台演进，不需要另起一套架构。
- 以 `F103 bring-up -> F030 量产候补` 的软件裁剪路线仍成立。

### 6.7 待验证项

- SoC 公共 API 是否足以覆盖英集芯真实合作接口。
- `F030` 资源能否承接最小量产版本的显示、日志、温度策略和异常恢复需求。
- 当前 SAL 与服务层边界在接入真实 SoC 后是否仍足够稳定。

## 7. 与英集芯合作的工程化接口规范

本章回答的问题是：既然合作对象已统一到英集芯，我们到底应该拿什么去谈，谈成什么算有效合作。

### 7.1 基本原则

当前合作原则仍然是：

> 要接口，不要源码。

但这句话只有在被展开为可执行交付物时才有谈判价值。

### 7.2 我方需要的最小交付物

在不要求原厂开放底层源码的前提下，至少应明确争取以下工程交付物：

1. 芯片基础资料：
   - 型号清单；
   - 功能边界说明；
   - 推荐使用场景；
   - 关键资源限制。
2. 接口资料：
   - I2C / UART 通信方式；
   - 地址、命令、寄存器或等价 API 文档；
   - 单位口径；
   - 读写时序与异常返回规则。
3. 状态读取能力：
   - 电压、电流、温度、端口状态、故障状态的读取方式。
4. 配置下发能力：
   - OCP、功率限制、端口使能、模式切换等可配置项。
5. 事件与中断能力：
   - 握手成功、异常状态、中断触发等事件模型；
   - 事件清除与恢复方式。
6. 调试与联调支持：
   - 典型初始化序列；
   - 异常案例说明；
   - 首轮联调窗口与 FAE 响应方式。
7. 版本与变更信息：
   - 芯片版本差异；
   - 文档版本；
   - 参数或接口变更通知机制。

### 7.3 不应接受的合作状态

以下状态不应被视为足够的合作基础：

- 只有口头说明，没有正式接口资料。
- 只能给 demo，不说明异常与边界条件。
- 只能跑通默认配置，不能读取关键状态或下发关键参数。
- 关键行为只能“看现象调”，没有事件和错误模型。
- 接口会变，但没有版本控制和变更通知规则。

### 7.4 我方需要展示的技术理解能力

既然当前合作对象已统一，且需要我方展示匹配的技术理解能力，那么我方至少应能清楚表达：

- 哪些能力应由 SoC 承担，哪些能力必须由 MCU 承担。
- 我方需要的是“可读取、可配置、可感知异常”的开放集成能力，而不是窥探底层实现。
- 我方的软件架构已经为多厂商 SoC 抽象预留了 SAL 边界，因此合作接口可以被稳定落进现有框架。
- 我方知道哪些风险必须通过硬件保护兜底，不能要求 MCU 软件替代。

### 7.5 待验证项

- 英集芯当前能提供的开放深度是否满足上述最小交付物。
- 当前合作中哪些内容需要 NDA、哪些能进入团队公共开发资料。
- 原厂是否能配合后续量产前的参数确认、异常支持与版本冻结。

## 8. 关键假设、验证方法与证据闭环

本章回答的问题是：哪些关键判断还只是推演，如何把它们变成证据支持的结论。

### 8.1 假设 A：日本中端客户重视“安全 + 性能数字化管理”

- 当前状态：假设
- 为什么关键：这是产品差异化方向的核心来源。
- 需要的证据：
  - 竞品分析；
  - 用户或渠道反馈；
  - 样机展示后的反应；
  - 价格带接受度。
- 若验证失败：
  - 需要重新判断用户真正关心的是显示、发热、体积、品牌感还是价格。

### 8.2 假设 B：磁吸便携应作为首发主场景

- 当前状态：项目输入已确定，但仍需市场与工程双重验证。
- 为什么关键：它决定结构、热设计、容量、厚度、显示、接口数量与功率档位。
- 需要的证据：
  - 场景优先级分析；
  - 样机结构与热验证；
  - 竞品场景分布。
- 若验证失败：
  - 需要重新收敛首发 SKU，而不是直接推翻整个平台。

### 8.3 假设 C：英集芯可提供足够的开放集成能力

- 当前状态：假设
- 为什么关键：这决定平台是否能建立在真实接口能力之上。
- 需要的证据：
  - 正式接口文档；
  - 可运行 demo；
  - 参数配置与状态读取清单；
  - 异常事件模型；
  - 联调结果。
- 若验证失败：
  - 需要降低平台抽象承诺，或重新评估替代 SoC 路线。

### 8.4 假设 D：F030 能承接最小量产版本

- 当前状态：假设
- 为什么关键：这决定成本路线是否成立。
- 需要的证据：
  - 编译结果；
  - Flash / SRAM 余量；
  - 外设完整性；
  - 样片运行稳定性；
  - 异常恢复表现。
- 若验证失败：
  - 按既有主线，上移到 `STM32G0 / GD32E230 / 更宽裕 MCU`。

### 8.5 证据闭环要求

所有关键假设的验证不应只停留在口头结论，至少应沉淀为以下一种或多种证据：

- 文档记录；
- 构建记录；
- UART 日志；
- 逻辑分析仪抓包；
- 温度/功率测试记录；
- 供应商接口版本资料；
- 访谈或渠道反馈摘要。

## 9. 合规、量产与交付约束

本章回答的问题是：从样机到量产，哪些约束不能被“软件先做出来”所掩盖。

### 9.1 当前已明确的约束方向

基于现有文档，当前已明确需要关注的约束包括：

- 日本市场相关合规要求；
- 电池容量与便携约束，例如 `<=100Wh` 的边界意识；
- 热设计风险；
- 过压、过流、过温、短路等保护链路；
- 结构尺寸与便携性约束。

### 9.2 当前仍需补实的内容

以下内容当前不能写成“已闭环”，只能写成后续必须补实的交付项：

- PSE 相关测试项与技术资料清单；
- 样机到量产的产测项目；
- 关键参数校准流程；
- 量产版本追溯方式；
- 售后故障诊断与日志保留策略。

### 9.3 当前原则

- 不能把“代码能跑”当成“产品能量产”。
- 不能把“实验室可用”当成“合规和长期可靠”。
- 不能把“有供应商 demo”当成“交付风险已经消失”。

## 10. 团队能力边界与协作风险

本章回答的问题是：在团队以软件背景为主的前提下，当前最容易误判什么，哪些事情必须保持边界感。

### 10.1 当前边界

按用户给定信息，当前团队出身以软件开发为主，因此必须正视以下现实：

- 对 MCU、寄存器、中断、时序、总线恢复等嵌入式基础的理解仍在补课阶段。
- 对电源路径、热设计、模拟保护、EMC、量产可制造性的理解不足。
- 对 SoC 原厂资料的判断能力、对“能跑 demo”和“可量产集成”的区分能力仍需加强。

### 10.2 当前高风险误区

1. 认为 MCU 可以替代底层电源保护。
2. 认为只要 I2C 能通信，就等于 SoC 已可稳定集成。
3. 认为原厂愿意合作，就等于接口和调试支持一定足够。
4. 认为软件分层画出来了，工程边界就自然成立。
5. 认为样机演示通过，就等于热、可靠性、认证路径不会出问题。

### 10.3 当前协作建议

- 与原厂谈合作时，必须以接口、异常、边界、版本和验证项为中心，而不是只谈方向。
- 涉及硬件保护、热设计、结构限制、认证边界的问题，不应仅凭软件团队内部推理下结论。
- 在真实仪器、样机、日志与抓包出现之前，不应把任何底层链路写成“已打通”。
- 文档中凡是没有证据支撑的结论，都应显式标记为假设或待验证。

## 11. 当前状态、阶段门与路线图

本章回答的问题是：当前项目已经做到哪里，后续按什么节奏推进。

## 12. 附录与术语表

### 12.1 关键术语

- `BMS`：电池管理系统，负责电池监测、保护与相关管理能力。
- `SoC`：这里指专用电源管理芯片，承担电源转换、协议与基础保护。
- `MCU`：微控制器，负责策略、状态读取、显示与系统级控制。
- `SAL`：SoC Abstraction Layer，SoC 抽象层，用于屏蔽不同 SoC 的接口差异。
- `OCP`：过流保护阈值。
- `PSE`：日本电气用品安全法相关认证要求。
