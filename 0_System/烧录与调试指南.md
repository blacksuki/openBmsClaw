# 烧录与调试指南

> 最新核对日期：2026-07-16
> 适用范围：`openBattery` 当前 `openBmsClaw/` STM32 起步工程

## 1. 文档目的

本文档用于把当前仓库中分散在课程文档、安装文档、sample README 和阶段计划里的“烧录与调试”说明统一收口，形成一份当前项目可执行的操作口径。

本文档重点回答 4 个问题：

1. 当前项目到底推荐用什么方式把固件烧到板子里。
2. 烧录前需要准备什么工具和产物。
3. 烧录后应该怎么看程序是否真的跑起来。
4. 哪些路径只是补充方案，不能误写成当前已验证主路径。

---

## 2. 当前项目推荐路径

当前推荐路径如下：

1. 在 `openBmsClaw/` 工程目录完成命令行构建。
2. 使用外部 `ST-LINK` / `J-LINK` / `DAP` 通过 `SWD` 接口烧录。
3. 使用 `STM32CubeProgrammer` 作为主烧录工具。
4. 使用板载 `USB TO UART` 观察启动日志。

一句话理解：

> **SWD 负责下载和调试，USB-UART 负责看日志。**

当前口径必须固定为：

- 当前学习与最小验证板：`野火小智 STM32F103C8T6 核心板（双 USB 款）`
- 当前推荐烧录链路：`外部 SWD 调试器 + STM32CubeProgrammer`
- 当前推荐日志链路：`USB TO UART + 串口助手`

## 3. 当前代码事实

当前仓库可确认的事实如下：

- 当前工程目录是 `openBmsClaw/`
- 当前构建产物位于 `openBmsClaw/build/<Preset>/`
- 当前可生成的常见产物包括：
  - `openBmsClaw.elf`
  - `openBmsClaw.hex`
  - `openBmsClaw.bin`
- 当前仓库**没有**内置 SWD 烧录自动化
- 当前仓库**没有**把“已编译成功”等同于“已实板烧录验证”

因此必须明确区分：

- **已实现**：命令行构建、产物生成、文档说明
- **未默认完成**：真实烧录、断点调试、串口联调、SoC 实板验证

## 4. 烧录前准备

### 4.1 硬件准备

至少准备以下硬件：

1. `野火小智 STM32F103C8T6 核心板（双 USB 款）`
2. 外部 `ST-LINK` / `J-LINK` / `DAP`
3. `SWD` 接线
4. `USB TO UART` 数据线

如果后续进入 SoC 联调，还应补齐：

1. SoC 评估板
2. 逻辑分析仪
3. 稳定供电

### 4.2 软件准备

至少保证以下工具已安装：

1. `arm-none-eabi-gcc`
2. `cmake`
3. `ninja`
4. `STM32CubeProgrammer`
5. 串口助手或 VS Code Serial Monitor

### 4.3 构建准备

先进入工程目录：

```bash
cd openBmsClaw
```

然后执行：

```bash
cmake --preset Debug
cmake --build --preset Debug
```

预期结果：

1. 构建成功
2. `build/Debug/` 下生成 `.elf` / `.hex` / `.bin`

## 5. 推荐方式 A：使用 SWD + STM32CubeProgrammer GUI 烧录

这是当前项目推荐主路径，也是后续板级调试主路径。

### 5.1 连接方式

把外部调试器接到核心板 `SWD` 接口，至少保证以下信号正确：

1. `SWDIO`
2. `SWCLK`
3. `GND`
4. `3V3` 或按调试器要求提供参考电压

### 5.2 GUI 烧录步骤

1. 打开 `STM32CubeProgrammer`
2. 选择 `ST-LINK` 连接方式
3. 点击 `Connect`
4. 选择构建产物 `openBmsClaw.bin`
5. 写入地址使用 `0x08000000`
6. 点击 `Start Programming`
7. 烧录完成后复位板子

### 5.3 产物选择建议

当前建议优先这样理解：

- `elf`：更适合调试器和 IDE 调试入口
- `hex`：常见烧录文件
- `bin`：裸二进制，适合直接指定起始地址烧录

如果使用 `STM32CubeProgrammer` 手动写入，`bin + 0x08000000` 最直接。

---

## 6. 推荐方式 B：macOS 下使用 STM32CubeProgrammer CLI 烧录

如果你已经在 macOS 上装好 `STM32CubeProgrammer`，可以直接走命令行。

示例命令如下：

```bash
/Applications/STMicroelectronics/STM32Cube/STM32CubeProgrammer/STM32CubeProgrammer.app/Contents/MacOs/bin/STM32_Programmer_CLI -c port=SWD -w build/Debug/openBmsClaw.bin 0x08000000 -v -rst
```

命令含义：

- `-c port=SWD`：通过 `SWD` 连接
- `-w ... 0x08000000`：把 `bin` 写到 Flash 起始地址
- `-v`：写后校验
- `-rst`：烧录完成后复位

建议先在 `openBmsClaw/` 目录执行该命令，这样路径和构建产物对应最直接。

## 7. 烧录后的调试方式

### 7.1 先看串口日志

当前最直接的运行确认方式，不是先看功能现象，而是先看串口启动日志。

串口建议配置：

- 波特率：`115200`
- 格式：`8N1`

烧录完成后：

1. 打开串口助手
2. 连接板子的 `USB TO UART`
3. 按下板上的 `RST` 复位键
4. 观察启动日志

### 7.2 当前建议关注的关键字

至少优先关注以下日志是否出现：

```text
PowerService: Init OK, emergency callback registered.
boot ok
[Test] Simulating hardware INT low pulse via SWIER...
```

理解方式如下：

- `boot ok`：主循环已进入运行
- `PowerService: Init OK...`：服务层初始化已进入工作路径
- `[Test] Simulating...`：当前软件自测链路已被触发

### 7.3 进入 SoC 联调阶段后应追加观察的日志

如果后续已经接入 Stage A 的 SoC 初始化链路，还应重点看：

```text
SAL: init start
SAL: EXTI init ok
bringup: soc sal init ok
bringup: soc poll online mask=...
bringup: soc int selftest PASS
```

如果这些日志不存在，不应宣称 SoC 真实初始化已经跑通。

## 8. 非主路径：只有 USB-UART 时怎么理解

如果手头只有 `USB TO UART`，当前应这样理解：

1. 可以看日志
2. 可以辅助判断程序是否启动
3. 不能默认等同于 `SWD` 下载与在线调试

只有在以下条件都满足时，串口下载才可能成立：

1. `BOOT0` 配置正确
2. 复位时序正确
3. 串口烧录工具正确
4. 芯片系统 bootloader 路径可用

因此当前项目不把“UART 下载”写成推荐主路径。

## 9. 推荐的最小执行顺序

建议严格按下面顺序操作：

1. 安装工具链与 `STM32CubeProgrammer`
2. 在 `openBmsClaw/` 跑通 `cmake --preset Debug`
3. 在 `openBmsClaw/` 跑通 `cmake --build --preset Debug`
4. 确认 `build/Debug/openBmsClaw.bin` 已生成
5. 接入 `SWD`
6. 用 `STM32CubeProgrammer` 烧录
7. 接入 `USB TO UART`
8. 观察 `115200 8N1` 启动日志
9. 再进入 I2C、EXTI、SoC 和逻辑分析仪联调

这样做的原因是：

- 先排除工具链问题
- 再排除烧录链路问题
- 再排除程序运行问题
- 最后才排查 SoC 外设问题

## 10. 常见误判

以下说法在当前项目里都不严谨：

### 10.1 “编译通过了，所以板子已经验证”

不成立。

编译通过只说明：

- 工具链可用
- 当前代码能生成产物

不说明：

- 固件已烧入目标板
- MCU 已经真正运行
- SoC 已经联调成功

### 10.2 “USB TO UART 能看到串口，所以等于已经完成下载”

不成立。

`USB TO UART` 主要用于日志观测，不等于 `SWD` 下载。

### 10.3 “已经有 SoC 相关日志代码，所以真实 SoC 已打通”

不成立。

只有出现真实 UART 运行日志、I2C 读值记录、逻辑分析仪抓包记录后，才能写“已完成实板联调”。
