## 1. 架构决策记录

### ADR-001：采用 MCU + 专用电源 SoC 的双芯片协同架构

**状态**：Accepted

#### 背景

移动电源平台需要同时处理电源转换、快充协议、电芯保护、温升控制、多口功率分配、用户显示和后续可扩展能力。若让 MCU 直接承担所有底层电源闭环、快充协议栈和 BMS 保护，会快速提高开发复杂度、认证风险和量产不确定性。

专用电源 SoC 已经在升降压、PD/QC/UFCS 协议、功率路径和基础保护方面具备成熟能力。`openBattery` 的价值不应重复制造这些底层能力，而应建立可观察、可配置、可迁移的开放控制平台。

#### 决策

采用"**SoC 负责体力活，MCU 负责脑力活**"的架构。

- SoC：负责升降压、电源路径、快充协议握手、基础 OVP/OCP/OTP 保护。
- MCU：负责读取状态、下发策略、动态降额、显示交互、日志、异常状态管理和平台扩展。

#### 取舍

| 方案 | 优点 | 缺点 | 当前结论 |
| --- | --- | --- | --- |
| MCU 大包大揽 | 可控性最高，理论上最开放 | 开发量大，认证风险高，对低成本 MCU 不现实 | 不采用 |
| 纯 SoC 黑盒方案 | 成本低，开发快 | 数据不可见，策略不可控，平台差异化弱 | 不采用 |
| MCU + SoC 协同 | 复用成熟电源能力，同时保留开放策略层 | 需要定义稳定接口和抽象层 | 采用 |

#### 后果

- 架构重点从"自己实现所有底层驱动"转为"定义稳定 SoC 接口和策略层"。
- 必须坚持 SoC 抽象层，否则上层业务会被某一家芯片绑定。
- 必须明确安全快路径和慢路径，不能让 MCU 成为唯一硬件保护来源。

### ADR-002：当前 MCU 路线采用 F103 bring-up，F030 量产候补

**状态**：Accepted for startup / F030 Proposed pending validation

#### 背景

当前代码事实是 `openBmsClaw/` 下的 STM32 工程，当前学习板为 `STM32F103C8T6`。为了降低初期学习、调试和工具链风险，startup 阶段应先把基础外设、日志、I2C、ADC、EXTI 和分层结构跑稳。

量产阶段需要更关注 BOM 成本，因此优先评估 `F030` 级别低成本 MCU。但 F030 资源有限，不适合作为"全功能平台 MCU"。

#### 决策

当前路线固定为：

```text
F103 bring-up（当前学习和开发）
        ↓
F030 量产候补（生产环境预计使用，需样片验证）
        ↓
F030 + 专用 SoC 最小量产链路
        ↓
若 F030 验证失败，再上移到 STM32G0 / GD32E230 / 更宽裕 MCU
```

#### 约束

F030 最小量产版本只能默认包含：

- SoC 状态读取与配置下发
- NTC / ADC 采样
- **NFC(ST25DV)状态读出：MCU 经 I²C 写快照+日志摘要，手机 tap 读回**
- **关键计数/事件日志的掉电持久化（充放电次数、过温/故障事件+时间戳，小体量），用于安全合规与售后诊断**
- LED 最小指示（轻量 OLED 数字页为可选，非必需）
- 按键状态机
- 轻量功率分配与温度降额
- I2C/UART 日志与错误码
- watchdog 和异常恢复

以下能力不能默认进入 F030 最小版本：

- LVGL 或复杂 UI、设备端图形显示屏
- 大字库、多语言资源
- 复杂 BLE 协议栈（BLE 仅作 premium 备选）
- 实时/隔空监测、远程监控、隔空 OTA
- 双镜像 IAP / OTA
- **大量历史日志/长时间序列持久化**（区别于上方"关键计数/事件日志"：后者小体量、为合规必须包含；前者的大容量历史记录不默认进入最小版）
- 多颗 SoC、多屏、多通信外设同时在线

### ADR-003：采用 NFC(ST25DV)作为数据读出窗口

**状态**：Accepted（2026-07-18）

#### 背景

需求澄清后，充电宝数据的用途是**安全合规 + 监控（内部记录）**；用户实时查询非实际需求，tap 读到当前状态（电量、充放电次数）即可，离开后再次 tap 取更新可接受。因此"数据可读"的载体不需要隔空实时链路，也不需要设备端显示屏。

#### 决策

MCU 经 I²C 把电池状态**快照 + 关键日志摘要**写入 **ST25DV**（被动 NFC 动态标签，NFC Forum Type 5 / ISO15693，I²C + RF 双接口，支持能量收割）；手机或 Web NFC **tap 读出**，再次 tap 取更新。BLE 降为**未来出现实时/隔空/远程监控/OTA 需求时**才启用的 premium 备选。

#### 取舍

| 方案 | 优点 | 缺点 | 当前结论 |
| --- | --- | --- | --- |
| NFC(ST25DV) | 仅 I²C、无射频栈、成本最低；被动标签基本免日本技適；掉电可读；贴合软件团队 | 仅 tap、无实时监测；金属机身需屏蔽验证 | **采用（主线）** |
| BLE | 可隔空实时 | 需技適（或预认证模组）、射频/协议栈资源、App 复杂 | premium 备选，非首发 |
| Type-C 有线 | 免技適、带宽足 | 必须插线、iOS 受限、需 USB 器件 | 诊断候补 |

#### 后果

- NFC 只用 I²C、无射频栈 → **放宽 F030 量产 MCU 的资源压力**（不同于 BLE）。
- 被动标签、充电宝非主动发射机 → **基本免日本主动发射技適**（13.56MHz 仍受电波法约束，量产前须与认证机构确认口径）。
- ST25DV 作为 `drivers/` 下新增 I²C 外设，走现有 `hal/i2c`，**不改动现有分层**；电池状态数据模型与日志置于 `services/`。
- 遗留风险：13.56MHz 天线在金属外壳/电芯附近会失谐，读距须打样实测（铁氧体屏蔽、天线开窗/避让）。

## 2. 实施路线与阶段门

### Phase 0：F103 bring-up

目标：

1. 稳定 `openBmsClaw/` 工程构建。
2. 跑通 LED、UART、I2C、ADC、EXTI、watchdog。
3. 建立日志、错误码、I2C timeout 和总线恢复。
4. 固定 `board / hal / drivers / services / app / config` 初始边界。

阶段门：

- 能稳定编译。
- UART 能输出启动日志和错误码。
- I2C 失败不会卡死主循环。
- 上层没有直接访问 SoC 寄存器。

### Phase 1：SoC SAL 基线

目标：

1. 定义 `soc_api.h` 和 `soc_types.h`。
2. 接入第一颗英集芯或智融 SoC。
3. 验证读取电压、温度、端口状态。
4. 验证 OCP 或功率限制参数下发。
5. 验证 SoC 告警 INT 和事件回调。

阶段门：

- `app/` 不 include `hal/`。
- `services/` 不 include `hal_i2c.h`。
- vendor 寄存器表不进入 `app/` 或 `services/`。
- I2C timeout、错误码和恢复路径可观察。

### Phase 2：服务层策略闭环

目标：

1. `srv_thermal` 根据 NTC 和 SoC 温度执行降额。
2. `srv_power_alloc` 根据端口状态执行功率分配。
3. 故障状态机支持锁定、提示、恢复条件。
4. 看门狗喂狗位置与故障恢复策略明确。

阶段门：

- 过温、过流、SoC 无响应都有明确状态。
- 快路径不执行慢操作。
- UI 只展示状态，不直接控制底层硬件。

### Phase 3：F030 样片验证

目标：

1. 至少验证一颗 STM32F030C8T6 级别基准器件。
2. 至少验证一颗国产 F030 候补器件。
3. 验证下载、SWD、UART、I2C、ADC、EXTI、watchdog。
4. 记录启动文件、链接脚本、HAL 或寄存器兼容差异。
5. 用真实编译结果评估 Flash / SRAM 余量。

阶段门：

- `PROFILE_F030_MIN_PRODUCT` 编译通过。
- Flash 至少保留 15%。
- SRAM 不发生栈或缓冲溢出。
- SoC 通信在异常场景下可恢复或可降级。

### Phase 4：最小量产链路

目标：

1. F030 + 专用 SoC + NFC(ST25DV) 完成最小闭环。
2. 完成热管理、功率降额、按键、LED 最小指示与 NFC tap 数据读出。
3. 整理 PSE、温升、异常保护、100Wh 策略，以及 NFC 天线读距/技適口径所需技术记录。

阶段门：

- 量产 MCU 型号不再写成"F030 类"，而是具体料号。
- SoC 型号、通信接口、关键寄存器/API 文档归档。
- 保护策略、异常处理和验证记录可追溯。

## 3. 当前建议动作

短期按以下顺序推进：

1. 保持当前 `MCU + SoC` 架构不变，不推翻重来。
2. 将当前 `soc_sal` 从 demo 形态逐步拆成公共 API、类型定义、transport 和 vendor adapter。
3. 收敛配置开关，建立 `F103_BRINGUP` 与 `F030_MIN_PRODUCT` 两类 profile。
4. 把 app 层直接触发底层测试 hook 的逻辑移入 bring-up/self-test 路径。
5. 每次新增服务或驱动时，先检查是否违反依赖矩阵。
6. 在接入真实 SoC 前，先明确最小 API 清单和错误码模型（参见 [08.api_spec.md](08.api_spec.md)）。

最终判断：

> 当前架构设计方向正确，startup 阶段的主要风险不是路线错误，而是分层规则没有被工程化执行。后续工作的重点是把"脑体分离"和"SoC 抽象"落实到头文件、目录、配置和阶段门中。
