## 1. 软件分层架构

### 1.1 分层总览

```text
app/             产品应用层：显示、按键、用户交互、状态页面
services/        系统服务层：功率分配、热管理、故障状态机
drivers/soc/     SoC 抽象层：统一 API、事件、vendor adapter
hal/             MCU 外设层：I2C、UART、ADC、GPIO、EXTI、watchdog
board/           板级支持层：引脚映射、板型能力、外设实例绑定
config/          配置层：功能裁剪、资源 profile、产品策略开关
```

架构原则：

> 上层只表达产品行为和系统策略，中间层屏蔽不同 SoC 差异，下层处理 MCU 外设和板级映射。

**NFC(ST25DV)落位：** 作为 `drivers/` 下新增的 I²C 外设驱动，走现有 `hal/` transport；电池状态数据模型（v0）与掉电日志置于 `services/`（如 `services/telemetry` 或数据/日志模块）。二者均不改动上述分层与依赖矩阵，ST25DV 与 SoC 同为 I²C 从设备。

### 1.2 允许依赖关系

| 当前层 | 可以依赖 | 禁止依赖 |
| --- | --- | --- |
| `app/` | `services/`、`config/`、应用级事件类型 | `hal/`、MCU 寄存器、SoC 寄存器、具体 vendor adapter |
| `services/` | `drivers/soc` 公共 API、`config/`、必要的 board capability 查询 | `hal_i2c.h`、`hal_adc.h`、具体 I2C 寄存器表 |
| `drivers/soc/` | `hal/` transport、SoC 类型定义、vendor adapter | `app/`、UI、业务页面 |
| `hal/` | MCU SDK / 寄存器 / 工具链生成代码 | `app/`、`services/`、SoC 业务语义 |
| `board/` | `hal/`、芯片引脚定义、板型能力描述 | 产品策略、功率分配算法 |
| `config/` | 无运行期依赖 | 不应包含可执行业务逻辑 |

### 1.3 严禁事项

1. `app/` 直接读写 SoC I2C 寄存器。
2. `services/` 直接 include `hal_i2c.h`、`hal_adc.h` 等 MCU 外设头文件。
3. SoC 公共 API 头文件暴露具体 HAL 句柄或 MCU 寄存器。
4. 某一颗 SoC 的寄存器名、地址、私有事件码进入 `app/` 或 `services/`。
5. 为了调试方便，把一次性测试 hook 留在正式业务路径中。
6. 使用多个配置宏表达同一个功能开关，造成配置双真相。

## 2. SoC 抽象层设计

### 2.1 目标

SoC 抽象层的目标不是把所有 SoC 能力一次性封装完整，而是先建立稳定的最小公共能力集。

最小公共能力集包括：

- 读取电压、电流、温度、端口状态
- 下发 OCP / 功率限制 / 端口使能配置
- 读取故障码
- 注册 SoC 告警事件回调
- I2C/UART 超时、错误计数和总线恢复

### 2.2 推荐结构

```text
drivers/soc/
├── soc_api.h              对 app/services 暴露的公共 API
├── soc_types.h            平台标准状态、错误码、事件类型
├── soc_manager.c          当前 SoC 实例、初始化、调度入口
├── soc_transport.h        I2C/UART 传输抽象
├── soc_transport_i2c.c    基于 hal_i2c 的传输实现
└── vendor/
    ├── injoinic_ip53xx.c  英集芯适配器
    └── zhirong_sw35xx.c   智融适配器
```

当前 startup 阶段可以先保留较薄的 `soc_sal.c`，但必须遵守两条退出条件：

1. 第一个真实 SoC 接入后，寄存器表和转换逻辑必须移入 vendor adapter。
2. 面向上层的公共头文件只能暴露平台类型和领域 API，不能暴露 `hal_i2c_t`、`I2C_TypeDef` 等底层细节。

> 公共 API 的具体函数签名与契约注解见 [08.api_spec.md](08.api_spec.md)。

## 3. 安全与异常处理架构

### 3.1 安全职责划分

移动电源产品不能把安全完全交给 MCU 软件判断。基础硬件保护必须由 SoC、保护芯片、保险丝、NTC 和电源路径硬件共同承担。

职责划分：

- SoC / 保护硬件：过压、过流、短路、过温等快速保护。
- MCU：读取状态、执行降额策略、锁存故障、提示用户、记录日志、恢复流程控制。
- `services/`：根据温度、端口、SoC 告警和产品策略做系统级仲裁。
- `app/`：只负责显示和用户交互，不直接决定硬件安全动作。

### 3.2 快路径与慢路径

紧急事件必须分成快路径和慢路径。

快路径：

- 来源：SoC INT、保护脚、EXTI。
- 目标：快速锁存状态，通知服务层进入保护状态。
- 禁止：I2C 读写、printf、delay、复杂算法、动态内存。

慢路径：

- 来源：主循环或服务层调度。
- 目标：读取 SoC 故障码、记录日志、更新 UI、决定恢复或保持锁定。
- 允许：I2C 读取、串口日志、错误码上报、状态页刷新。

## 4. 配置与资源 Profile

### 4.1 单一配置真相

每一个功能开关只能有一个权威来源。禁止同时存在多个含义相近但状态不一致的宏，例如一个文件启用 power service，另一个文件关闭 power feature。

推荐按 profile 管理：

```text
PROFILE_F103_BRINGUP       当前学习和基础外设验证
PROFILE_F030_MIN_PRODUCT   F030 最小量产候补
PROFILE_G0_EXTENDED        更大资源 MCU 的扩展功能
```

### 4.2 F030 最小量产 Profile

`PROFILE_F030_MIN_PRODUCT` 默认策略：

- 默认关闭 BLE、实时/远程监控、复杂 UI、大字库、双镜像 IAP。
- **默认启用 NFC(ST25DV)状态读出（I²C）** 作为数据可读主线。
- **默认启用关键计数/事件日志的掉电持久化（充放电次数、过温/故障），小体量，用于合规；不含大量历史序列。**
- 日志默认支持错误码和少量关键文本。
- 数据读出以 NFC tap 为主；设备端显示按需，优先 LED 最小指示，轻量 OLED 数字页为可选。
- 所有 I2C 操作必须有 timeout。
- 必须启用 watchdog。
- 编译后 Flash 至少保留 15% 余量，SRAM 必须完成最坏路径检查。

## 5. 技术债预防规则

startup 阶段允许出现临时验证代码，但必须被标记、隔离并有退出条件。

### 5.1 可接受的临时代码

- 用于验证 I2C 读写的 demo register。
- 用于验证 EXTI 的模拟触发函数。
- 用于 bring-up 的串口调试打印。
- 用于验证板级 LED 的直接调用。

### 5.2 不允许长期保留的债务

- `app/` 直接调用低层测试 hook。
- SoC 公共 API 头文件暴露 HAL 类型。
- vendor 寄存器表混在平台公共 SAL 中。
- 同一功能存在多个配置开关。
- 中断回调里执行 I2C、printf 或 delay。
- 为了当前板子方便，把板级细节写进服务策略。

### 5.3 每次进入新阶段前必须清理

1. 删除或隔离上一阶段的临时测试 hook。
2. 检查 include 方向是否符合依赖矩阵。
3. 检查配置开关是否只有一个权威来源。
4. 检查 Flash / SRAM 占用是否仍符合目标 profile。
5. 检查新增功能是否可裁剪。
