# 开发准备清单 (SoC 双核协同版)

> 最新核对日期：2026-07-01

## 1. 文档目的
本文档用于整理 `openBattery` 项目启动开发前必须准备的软硬件资料。目标不是一次性堆满所有设备，而是围绕**“基于集成 SoC 的双核协同架构”**，建立一套 **能快速启动、防范资源陷阱、符合系统分层目标** 的准备方案。

同时需要明确：

- 当前主项目是 `openBattery`
- 当前代码起点是仓库内已有的 STM32 生成工程

> **当前可先用手头的 `野火小智 STM32F103C8T6 核心板（双 USB 款）` + 外部 SWD 调试器完成基础学习和环境搭建。未来生产环境预计优先评估 `F030` 级别 MCU 作为量产候补，但必须按 64KB Flash 资源边界裁剪功能，并通过样片验证后才能写入量产参考设计。**

这份清单服务于两类目标：

- **快速启动验证**：用最少投入跑通 MCU 最小系统，并打通 MCU 到外部集成 SoC 的标准通信协议 (I2C/UART)。
- **后续量产演进**：准备的软硬件应直接映射后续量产时的 SoC 驱动抽象 (SAL) 与数字化协同能力，避免重复造轮子。

---

## 2. 准备原则

### 2.1 与系统架构保持一致
根据 `03.tech_architecure.md` 和 `10.hardware_softare_frame.md`，开发准备必须符合以下原则：

1. **废弃散件拼凑**：摒弃原本分散采购纯 BMS 评估板、独立 PD 协议板的思路。直接以**“集成 SoC 评估板”**（如英集芯 IP53xx，智融 SW35xx）为核心搭配 MCU。
2. **防范资源陷阱**：主控评估以 F030 量产候补为目标时，必须默认按 64KB Flash 受限平台设计，关闭大字库、复杂 UI、BLE 协议栈和双镜像 IAP；若样片验证失败，再上移到 STM32G0、GD32E230 或更宽裕 MCU。
3. **工具链统一**：当前仓库的实际验收路径是 `STM32CubeCLT + CMake + Ninja`；`STM32CubeMX` 负责外设配置和代码生成，VS Code / STM32CubeIDE 只作为前端编辑与调试入口。
4. **重视边界验证**：优先验证 MCU 对 SoC 的“异常情况介入”能力（例如 I2C 卡死时的总线自愈、SoC 紧急中断响应），而不仅是看快乐路径 (Happy Path) 下的握手成功。

### 2.2 准备范围
本清单分为四部分：
- 必备硬件
- 建议增配硬件
- 必备软件
- 建议准备的资料与验证环境

## 3. 必备硬件准备

### 3.1 主控 MCU 开发板
针对 MCU，分为“起步学习”“量产候补验证”和“资源回退”三个阶段：

| 优先级 | 板卡 | 用途 | 说明 |
| --- | --- | --- | --- |
| 起步 | `野火 STM32F103C8T6 核心板` | 最小化跑通代码框架 | 用于跑通 GPIO/UART/I2C 基础功能，**但在后期须引入严格的裁剪宏**，关闭大字库和复杂 UI 以防爆 Flash。 |
| 推荐 | `STM32F030C8T6` / `HK32F030C8T6/R8T6` / `CKS32F030C8T6/R8T6` 样片或最小板 | 量产候补验证 | 重点验证 64KB Flash 档位是否足够承载 `hal + drivers + services + app_min + log_min`。 |
| 备选 | `STM32G0` / `GD32E230` / `STM32L4` 系列开发板 | 资源回退与扩展 | 当 F030 无法满足资源、调试或可靠性要求时，再作为更宽裕平台评估。 |

### 3.2 核心外设：集成 SoC 评估板 (关键变更)
我们不再推荐购买纯粹的 ST 官方独立协议板 (如 X-NUCLEO-SNK1M1)，而是强烈建议直接采购：

| 硬件 | 用途 | 说明 |
| --- | --- | --- |
| **集成 SoC 官方评估板 (如英集芯 IP5328 / 智融 SW3516 DEMO 板)** | 用于与 MCU 的双核协同联调 | 评估板上自带电芯接口、多口 USB 母座、完整的升降压电感电容。MCU 仅需飞出 GND, SDA, SCL, INT 线连接，即可开始编写 `drivers/soc/` 层的标准接口 (SAL)。 |

### 3.3 调试与烧录工具
至少准备以下其中一种：

| 工具 | 是否必备 | 用途 | 说明 |
| --- | --- | --- | --- |
| 板载 ST-LINK | 是 | 下载、调试 | Nucleo 板通常已自带，优先使用 |
| 外部 `DAP / ST-LINK V2/V3` | 建议 | F103 学习板或自研板调试 | 必须带完整的 SWD 接口 |

### 3.4 仪器仪表 (警惕 I2C 阻塞的利器)
移动电源的高功率跳变极易引发总线死锁，因此测试设备优先级必须调整：

| 工具 | 是否必备 | 用途 |
| --- | --- | --- |
| **逻辑分析仪** | **强烈推荐 (必备)** | **这是开发阶段最重要的防具**。用于观测 I2C Clock Stretching、总线挂死现象，以及校验 MCU 总线自愈机制（模拟发出 9 个时钟脉冲）是否成功释放了 SDA。 |
| 可调直流电源 | 是 | 模拟电池侧或充电头输入。 |
| USB 功率表 | 是 | 验证各端口的电压/电流/快充档位。 |
| 数字万用表 | 是 | 测量连续性、电阻网络等。 |
| 电子负载 | 建议 | 测试高负载温升与功率限制调度 (srv_power_alloc)。 |
| 示波器 | 建议 | 观察纹波、电源噪声和信号边沿。 |

### 3.5 外围扩展硬件
| 模块 | 用途 | 说明 |
| --- | --- | --- |
| OLED 模块 (0.96/1.14寸) | 数字化界面验证 | 用于展示剩余时长、电池健康度(SOH)。注意字库大小。 |
| 串口 BLE 模块 | 外部通信验证 | 用于与手机 APP 的对接与固件升级通道模拟。 |

## 4. 建议增配硬件

当准备脱离开发板进入量产过渡时：
- 面向目标结构的**自研双核主板样机**（贴装目标 MCU 和目标 SoC）。
- 底座连接器和五联托盘模拟治具。
- 温箱或冷热环境测试设备（验证 `srv_thermal` 的降额策略）。

## 5. 必备软件准备

### 5.1 主开发工具链
当前需要按“代码生成工具”和“命令行构建工具”拆开理解：

| 工具 | 是否必备 | 作用 | 建议定位 |
| --- | --- | --- | --- |
| **STM32CubeMX** | 是 | 外设配置、代码生成 | 当前 STM32 工程入口之一 |
| **STM32CubeCLT** | 是 | 提供 `arm-none-eabi-gcc` 等命令行工具链 | 当前仓库构建验收必备 |
| **CMake** | 是 | 读取 preset，生成构建系统 | 当前仓库构建入口 |
| **Ninja** | 是 | 执行实际编译 | 当前默认生成器 |
| Visual Studio Code | 可选 | 编辑器 / 扩展前端 | 可搭配 CMake Tools 和 ST 扩展使用 |
| STM32CubeIDE | 可选 | IDE 前端 / 图形化调试 | 适合作为辅助前端，不是当前仓库唯一构建入口 |
| STM32CubeProgrammer | 是 | 下载、擦写、量产烧录验证 | 主烧录工具，用于清空写保护、验证 IAP |
| Git | 是 | 版本管理 | 基础能力 |

### 5.2 软件基线建议
版本建议以 `13.software_installation.md` 为准，但当前仓库至少要满足：

1. 终端可执行 `arm-none-eabi-gcc --version`。
2. 终端可执行 `cmake --version` 与 `ninja --version`。
3. 在 `openBmsClaw/` 工程目录可执行：

```sh
cmake --preset Debug
cmake --build --preset Debug
```

只有这两条命令能通过，才算当前开发环境真正就绪。

### 5.3 不建议的软件路线
- 不要只装 VS Code 扩展就认为“工具链已完成”，扩展本身不等于编译器。
- 不要只依赖 IDE 按钮构建而跳过命令行验证；当前仓库验收标准是命令行构建通过。
- 不建议再把“只依赖 CubeIDE 图形构建”写成唯一主路线，这会和当前工程实际构建方式冲突。

## 6. 必要资料准备

在开始写第一行代码前，以下资料必须放置在项目文档区 (`91.reference/`)：
- **MCU 数据手册与参考手册** (RM)。
- **目标 SoC 的详细寄存器手册 (极度关键)**：由于我们不索要源码，必须拿到 SoC 厂方的 I2C 通信地址、寄存器映射表以及中断定义说明。
- SoC 原厂的 DEMO 板原理图。

此外，开发前必须明确以下**核心协同接口**：
1. **MCU 与 SoC 通信总线定义**（I2C 的速率、地址）。
2. **SoC 紧急中断管脚定义**（用于触发高速告警通道）。
3. **外部 NTC 管脚分配**。
4. UI 交互矩阵与按键定义。

## 7. 最小验证环境与演进顺序

### 7.1 最小硬件连接 (不再是散件堆砌)
```text
MCU 核心板
├── I2C (SDA, SCL) + INT ──> 目标集成 SoC 评估板 (含电源、协议、BMS)
├── I2C / SPI ─────────────> OLED 显示屏
├── UART ──────────────────> BLE 模块
└── ADC ───────────────────> 外部 NTC
```

### 7.2 演进与验证顺序 (对齐新架构)
根据 `10.hardware_softare_frame.md` 中指出的脑体分工与防挂死要求，采取以下验证步骤：

#### 阶段 1：稳定 F103 起步环境
- 跑通时钟、串口、GPIO、ADC。
- 确认命令行 `cmake --preset Debug && cmake --build --preset Debug` 可通过。
- 强制要求 I2C/UART 驱动具备 timeout 或非阻塞能力。

#### 阶段 2：SoC 基线接入与防呆验证
- 在 `drivers/soc/` 层固定 SoC SAL 标准接口。
- 在运行路径真正接通 `soc_sal_init()`。
- **验证双向读写**：能成功读取 SoC 的电压、电流，能下发 OCP 阈值。
- **破坏性演习**：故意短接 SDA 或强干扰总线，验证 MCU 的总线自愈 (Bus Recovery) 是否能成功解死锁。

#### 阶段 3：智能服务策略落地
- **紧急告警通道验证**：触发 SoC 过温短路，看系统能否不经 UI 业务层，而在几毫秒内直接降额。
- 多口插拔时验证 `srv_power_alloc` 的动态分配指令下发。

#### 阶段 4：F030 样片验证
- 切换 `PROFILE_F030_MIN_PRODUCT` 进行构建。
- 验证 F030 样片的下载、I2C、ADC、EXTI、watchdog。
- 记录启动文件和链接脚本差异。

#### 阶段 5：透传升级、交互与合规
- 验证 MCU 透传功能，将新的 SoC 固件包“喂”给 SoC 的内部 Flash。
- 根据资源余量接入最小 OLED/LED 交互。
- 开展 PSE 测试预检。

## 8. 启动建议
如果您今天就打算开干：

1. 掏出您的 `F103` 核心板，配置好 `CubeMX` 和 `CubeIDE`，点亮 LED，跑通串口日志。
2. 安装 `STM32CubeCLT`、`CMake`、`Ninja`，在工程目录先跑通命令行构建。
3. 立即上淘宝/代理商渠道**申请一块对应 SoC（如 IP5328）的官方 DEMO 板**，和一台**逻辑分析仪**。
4. 飞线对接，从读取第一个 SoC 寄存器开始联调。
