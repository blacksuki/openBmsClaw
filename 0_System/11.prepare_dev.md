# 开发准备清单

> 最新核对日期：2026-04-05

## 1. 文档目的
本文档用于整理本项目启动开发前必须准备的软硬件资料，目标不是一次性堆满所有设备，而是围绕以下核心思路，建立一套 **能快速启动、可持续扩展、符合系统架构目标** 的准备方案：

同时需要明确：

- 当前主项目是 `bmsClaw`
- 当前代码起点是仓库内已有的 STM32 生成工程
- `90.mini-Lite` 是参考项目，不是当前工程目录

> **以 STM32G0 作为长期平台 MCU 路线，以 STM32G0 Nucleo 作为参考开发板路线；当前可先用手头的 `野火小智 STM32F103C8T6 核心板（双 USB 款）` + 外部 SWD 调试器完成学习和最小可运行验证。这里的 F103 仅作为当前学习硬件，不作为长期平台选型。**

这份清单服务于两类目标：

- **快速启动验证**：用最少投入验证主控、协议、采样、显示与底座逻辑是否跑通。
- **后续量产演进**：准备的软件与硬件尽量与后续模块化量产平台一致，避免原型阶段和量产阶段完全脱节。

---

## 2. 准备原则

### 2.1 与系统架构保持一致
根据 `00.tech_archture.md` 和 `10.hardware_softare_frame.md`，开发准备必须符合以下原则：

1. 主控平台优先围绕 **STM32G0** 搭建，而不是继续投入到 8 位 MCU 或无迁移价值的平台。
2. 所有准备项都应优先服务于 **模块化验证**，而不是一次性定制整机。
3. 工具链应优先选择 **官方免费工具 + 可持续维护的开源工具**。
4. 验证顺序应从 **通信链路打通** 开始，再逐步进入协议、电源、热管理、底座和整机逻辑。

这里需要明确区分两个概念：

- **Nucleo 开发板型号**：例如 `NUCLEO-G071RB`、`NUCLEO-G0B1RE`，它们用于开发、下载、调试和参考实现。
- **量产 MCU 型号**：例如 `STM32G030C8T6`、`STM32G071RBT6`、`STM32G0B1RET6`，它们才是企业最终在产品 BOM 中选择的芯片。

因此，开发阶段推荐采购 Nucleo 板；量产阶段则根据成本、资源和封装重新确定 MCU。

### 2.2 准备范围
本清单分为四部分：

- 必备硬件
- 建议增配硬件
- 必备软件
- 建议准备的资料与验证环境

## 3. 必备硬件准备

### 3.1 主控开发板
建议将 **STM32G0 Nucleo 开发板** 作为最小启动平台，而不是 STM32F4 大板。

注意：

- `NUCLEO-G0B1RE` 是 **开发板型号**，不是量产时直接放进产品里的器件型号。
- 它板上搭载的是 `STM32G0B1RE` MCU，并自带调试器和扩展接口，适合作为参考开发板。
- 真正量产时，企业应选择具体 MCU，而不是选择 `NUCLEO-*` 开发板本身。

推荐顺序：

| 优先级 | 板卡 | 用途 | 说明 |
| --- | --- | --- | --- |
| 推荐 | `NUCLEO-G071RB` | 最小启动与主流验证 | 成本低、生态成熟、适合作为 STM32G0 入门板 |
| 推荐 | `NUCLEO-G0B1RE` | 中端主控验证 | 资源更宽裕，适合协议、显示、BLE 并行验证 |
| 备选 | STM32F4 系列 Nucleo | 高性能扩展验证 | 仅用于高配路线，不建议作为默认起步板 |

当前项目可以采用三层路线来理解：

| 角色 | 当前建议 | 说明 |
| --- | --- | --- |
| 学习/起步板 | `野火小智 STM32F103C8T6 核心板（双 USB 款）` + 外部 SWD 调试器 | 用于先学下载、调试、串口、GPIO、I2C、ADC |
| 平台参考开发板 | `NUCLEO-G071RB` / `NUCLEO-G0B1RE` | 用于沉淀官方参考工程和后续模块联调 |
| 量产 MCU 候选 | `STM32G0` 系列 | 用于最终产品 BOM 选型，而不是直接使用 Nucleo 板 |

### 3.1.1 当前学习板的准确硬件事实
当前手头学习板统一按下面的规格理解：

- 板卡名称：`野火小智 STM32F103C8T6 核心板（双 USB 款）`
- MCU：`STM32F103C8T6`，`ARM Cortex-M3`，`72 MHz`，`64 KB Flash`，`20 KB SRAM`
- 板上资源：`4` 个 LED（`1` 个电源灯 + `3` 个可控 LED），`3` 个按键（`1` 个复位键 + `2` 个可控按键）
- 接口：`2` 个 USB，其中一个是 `USB Device`，一个通过 `CH340C` 提供 `USB-UART`
- 调试接口：板上带 `5-pin` 标准 `SWD`，可接外部 `DAP / JLINK / ST-LINK`
- 串口下载：`USB-UART` 可用于供电、日志与串口下载；相关 `BOOT0 / BOOT1` 条件需要按板卡资料设置

这块板的定位要固定为：

- 当前学习 / bring-up 硬件
- 不是长期平台 MCU 选型结论
- 不是量产硬件推荐口径

关于 `STM32G030C8T6`：

- 它可以作为 **低成本量产候选 MCU**；
- 但更适合在平台结构稳定后，再评估是否作为成本版 SKU；
- 不建议在当前新手起步阶段，用它替代带板载调试器和扩展接口的 Nucleo 参考板。

为什么不再把 `STM32F407ZGT6` 作为默认起点：

- 它与当前系统文档中的 **STM32G0 主平台路线** 不一致。
- 对“低成本、可量产迁移”的目标帮助有限。
- 早期启动阶段更需要的是 **接口验证和模块组合能力**，不是过高的 MCU 性能冗余。

为什么当前仍允许继续使用 `STM32F103C8T6` 学习：

- 你手头已经有板子，学习成本最低；
- 这块双 USB 学习板自带 `CH340C USB-UART`、标准 `SWD`、多颗 LED 和按键，足够支撑下载、调试、串口、基础外设和工程结构学习；
- 但它更适合 **学习板/起步板角色**，不应被写成当前长期平台 MCU 定型。

### 3.2 调试与烧录工具
至少准备以下其中一种：

| 工具 | 是否必备 | 用途 | 说明 |
| --- | --- | --- | --- |
| 板载 ST-LINK | 是 | 下载、调试 | Nucleo 板通常已自带，优先使用 |
| `STLINK-V3MINIE` / 同级调试器 | 建议 | 脱离板载调试时使用 | 后续自研板 bring-up 更方便 |
| 外部 `DAP / JLINK` | 可用 | 当前 F103 学习板临时 SWD 下载 / 调试 | 板上 `5-pin SWD` 可直连，但工具链适配需按实际调试器确认 |

建议规则：

- 原型阶段优先使用 **Nucleo 板载调试器**。
- 当前 F103 学习板如已具备外部 `DAP / JLINK`，可先跑通 `SWD` 基础链路；项目标准推荐调试器仍是 `ST-LINK`。
- 一旦开始自制底板或小批打样，尽快补齐 **独立 ST-LINK 调试器**。

### 3.3 电源与测量工具
以下工具不要省略，否则很难定位协议、电流、温升和异常保护问题：

| 工具 | 是否必备 | 用途 |
| --- | --- | --- |
| 可调直流电源 | 是 | 模拟电池侧/适配器侧输入 |
| 数字万用表 | 是 | 电压、电流、连续性基础测试 |
| USB 功率表 | 建议 | 验证 Type-C / USB-A 输出档位 |
| 逻辑分析仪 | 建议 | 观察 I2C / UART / GPIO 时序 |
| 电子负载 | 建议 | 验证输出功率、保护点、降额策略 |
| 示波器 | 建议 | 观察纹波、插拔瞬态、协议边沿行为 |

对于最小化启动，前 3 项优先级最高；对于协议、电源和保护类问题，后 3 项会很快变成刚需。

### 3.4 最小功能模块
为了符合“模块化搭积木”的核心思路，建议按模块准备最小验证集：

| 模块 | 最小准备项 | 用途 |
| --- | --- | --- |
| BMS | 分立模拟电路或 BMS 评估板 | 验证电压、电流、温度采样与保护逻辑 |
| 协议 | PD/USB-C 扩展板或协议评估板 | 验证 C1 / C2 / A 口角色与功率协商 |
| 显示 | LED + 0.96 寸 OLED | 先跑状态机，再考虑更复杂显示 |
| BLE | 串口 BLE 模块 | 验证状态同步与后续 OTA 通道 |
| 底座 | 电阻网络 + 连接器治具 | 验证单底座/托盘识别逻辑 |

### 3.5 建议优先采购的扩展板/治具
如果项目从一开始就要验证 USB-C/PD 和角色切换，建议优先考虑：

- USB-C Sink 扩展板：用于验证输入充电链路。
- USB-C DRP / 双角色扩展板：用于验证 `C2` 的输入/输出切换逻辑。
- 简单底座识别治具：用于模拟单底座与托盘模式。
- NTC / 电压分压治具：用于温度与底座检测算法联调。

原则是：**先用可替换治具把关键链路跑通，再决定是否定制专用板。**

## 5. 必备软件准备

### 5.1 主开发工具链
建议的软件基线如下：

| 工具 | 是否必备 | 作用 | 建议定位 |
| --- | --- | --- | --- |
| STM32CubeIDE | 是 | 代码工程、编译、调试 | 默认主开发环境 |
| Visual Studio Code | 是（若选择 VS Code 路线） | VS Code 主环境 | 可选主开发环境 |
| STM32CubeIDE for Visual Studio Code | 是（若选择 VS Code 路线） | 创建、导入、构建、调试 STM32 工程 | 可选主开发环境 |
| STM32CubeMX | 是 | 外设配置、代码生成 | 主配置工具 |
| STM32CubeProgrammer | 是 | 下载、擦写、量产烧录验证 | 主烧录工具 |
| Git | 是 | 版本管理 | 基础能力 |
| PlatformIO | 可选 | 轻量化开发/自动化探索 | 辅助路线 |
| STM32CubeCLT | 可选 | CLI 构建与自动化 | 后续 CI/批量构建路线 |

### 5.2 当前推荐的软件安装顺序
建议按以下顺序安装，避免后续工具链混乱：

1. 先确定使用哪条 IDE 路线：
   - 默认主路径：`STM32CubeIDE`
   - 可选主路径：`Visual Studio Code + STM32CubeIDE for Visual Studio Code`
2. 安装所选 IDE 路线的主工具
3. 安装 `STM32CubeMX`
4. 安装 `STM32CubeProgrammer`
5. 安装 `Git`
6. 根据需要再安装 `PlatformIO`、`Cortex-Debug`、`STM32CubeCLT`

### 5.3 关键说明

### 5.3.1 不要把 CubeIDE 当成“全家桶”
当前开发准备中，`STM32CubeIDE` 和 `STM32CubeMX` 应视为 **两个独立安装项**。

这意味着：

- 新环境搭建时，不能只装 CubeIDE 就默认具备完整图形化配置能力。
- 文档、培训和新成员 onboarding 里，要单独写明 CubeMX 的安装步骤。

### 5.3.2 VS Code 路线可以使用，但不替代默认主路径
`STM32CubeIDE for Visual Studio Code` 可以作为当前项目的开发工具，但建议定位为：

- **可选主路径**
- **个人偏好或高级用户路径**
- **不直接替代桌面版 `STM32CubeIDE` 的默认团队路径**

原因是：

- 它已经具备官方支持的 STM32 开发能力；
- 但当前团队文档、培训和统一交付仍更适合围绕桌面版主路径展开；
- VS Code 路线应作为并列支持路径，而不是把桌面版路径删除。

### 5.3.3 CubeProgrammer 应作为标准工具
不要只依赖 IDE 内下载按钮。项目从原型走向样机和量产时，需要：

- 独立烧录
- 擦写验证
- 批量下载
- 产线脚本化

因此 `STM32CubeProgrammer` 应被视为 **标准必备工具**，而不是附属工具。

### 5.3.4 PlatformIO 是辅助路线，不是主路线
`VS Code + PlatformIO` 适合：

- 轻量编辑
- 跨平台开发体验
- 辅助自动化与脚本化

但在本项目早期，**主路径仍应以 STM32 官方工具链为主**，原因是：

- 外设初始化、芯片包、示例和生态与 ST 官方工具链结合更紧密。
- 未来涉及协议、电源和复杂外设时，官方资料更容易对齐。

### 5.4 截至 2026-03-25 的工具基线建议
如果当前要搭建一套新环境，建议按下面的基线准备：

| 工具 | 建议基线 | 用途 |
| --- | --- | --- |
| `STM32CubeIDE` | `2.1.x` | 默认主开发环境 |
| `Visual Studio Code` | 当前稳定版 | VS Code 主环境 |
| `STM32CubeIDE for Visual Studio Code` | ST 官方扩展当前版 | 可选主开发环境 |
| `STM32CubeMX` | 独立安装版 | 外设配置与工程生成 |
| `STM32CubeProgrammer` | `2.22` | 烧录、擦写、验证 |
| `STM32CubeCLT` | `1.21.0` | CLI 构建与自动化 |
| `PlatformIO for VS Code` | 当前官方文档版 | 辅助开发路线 |

说明：

- `CubeIDE 2.x` 的核心变化是 **与 CubeMX 解耦**，因此安装文档和培训文档必须同步更新。
- `STM32CubeIDE for Visual Studio Code` 可作为当前项目的可选主开发路径，但不建议直接替代桌面版默认路径。
- `CubeProgrammer 2.22` 和 `CubeCLT 1.21.0` 可以作为当前自动化和量产过渡阶段的参考基线。
- `PlatformIO` 适合 Windows、macOS、Linux 的跨平台辅助开发，但在本项目中仍建议作为辅助环境而非主路径。

## 8. 最小验证环境

为了尽快进入可执行状态，建议先搭一个“最小可运行验证环境”。

### 8.1 最小硬件连接
```text
STM32G0 Nucleo
├── I2C / ADC -> BMS 模拟输入或评估板
├── I2C / UART -> 协议模块或 USB-C 扩展板
├── GPIO / I2C -> LED / OLED 显示
├── UART -> BLE 模块
├── ADC / GPIO -> NTC / 底座识别治具
└── ST-LINK -> 下载与调试
```

### 8.2 最小软件流程
```text
需求确认
-> 用 CubeMX 建立最小外设工程
-> 用 CubeIDE 编译与调试
-> 用 CubeProgrammer 下载与验证
-> 跑通 LED / 按键 / 采样 / 协议最小链路
-> 再引入 BLE、底座和多模块协同
```

### 8.3 第一阶段必须跑通的功能
第一阶段不追求整机完整功能，优先完成：

1. GPIO 与 LED 状态指示
2. ADC 采样与 NTC 读取
3. I2C / UART 基础通信
4. Type-C 输入或输出单链路验证
5. 底座识别电压/电阻检测
6. 基础日志输出

## 9. 推荐验证顺序

为了降低风险，建议按以下顺序推进：

### 阶段 1：主控与基础外设
- 跑通时钟、串口、GPIO、ADC、I2C
- 建立调试和日志机制

### 阶段 2：显示与状态机
- 先用 LED 验证状态机
- 再接入 OLED 或其他显示模块

### 阶段 3：协议与功率链路
- 先验证单口
- 再验证 `C1 + C2`
- 最后验证 `C1 + A`

### 阶段 4：BMS 与保护逻辑
- 采样
- 保护阈值
- 故障状态反馈

### 阶段 5：底座与托盘模式
- 单底座识别
- 托盘模式识别
- `C2` 输入屏蔽与切换逻辑

### 阶段 6：BLE 与 OTA
- 状态同步
- 参数下发
- 后续升级路径预留

## 10. 预算建议

不建议在文档中把预算写死为单一数字，而应按阶段规划：

| 阶段 | 预算级别 | 内容 |
| --- | --- | --- |
| 最小启动 | 低 | Nucleo 主控板 + 基础测量工具 + LED/OLED + 简单协议验证模块 |
| 功能联调 | 中 | 增加 BLE、协议扩展板、逻辑分析仪、电子负载 |
| 量产预研 | 中高 | 增加自研样机、夹具、独立调试器、可靠性测试条件 |

控制成本的原则不是“买最少”，而是：

- 先买能打通主链路的设备
- 再补充能提升定位效率的工具
- 避免过早投入整机定制硬件

## 11. 启动建议

如果现在就要开始，建议按下面的最小路径执行：

1. 先采购一块 `STM32G0` Nucleo 板
2. 同时准备 `CubeIDE + CubeMX + CubeProgrammer`
3. 先做 LED / ADC / UART / I2C 基础 bring-up
4. 再接入一个最小协议模块或 USB-C 扩展板
5. 然后验证底座识别和 `C2` 输入输出切换
6. 最后再加 BLE、OLED 和更复杂的协同逻辑

这条路径最符合当前系统的设计目标：**用最小投入快速验证模块化架构，而不是在一开始就做复杂整机。**
