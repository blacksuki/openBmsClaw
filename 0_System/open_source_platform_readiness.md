# openBattery 开源基础平台就绪度定义

> 最新更新：2026-07-02
> 状态：Proposed
> 适用范围：用于判断 `openBattery` 是否已具备作为“开源基础平台”对外发布和持续演进的条件

## 1. 文档目的

本文档不讨论 `openBattery` 的架构方向是否正确，而是回答一个更具体的问题：

> 当前项目距离“可作为开源基础平台”还差哪些硬条件。

这里的“开源基础平台”不是指：

- 把当前仓库代码直接公开
- 把一个样机工程加上 README 就对外发布
- 把一块板子的固件当成平台

这里的“开源基础平台”是指：

- 有稳定的架构边界
- 有可复用的接口契约
- 有最小可验证的硬件闭环
- 有明确的支持范围和不支持范围
- 有足够文档让外部开发者能够理解、移植、复现和扩展

---

## 2. 当前判断

当前 `openBattery` 已经具备以下基础：

1. 已形成 `MCU + 专用 SoC` 的双芯片主架构。
2. 已形成 `app / services / drivers/soc / hal / board / config` 的分层边界。
3. 已形成 `F103 bring-up -> F030 量产候补` 的现实路线。
4. 已形成 `profile`、资源裁剪、快路径/慢路径等平台化思维。

因此，当前状态可定义为：

> **具备平台雏形，但尚未达到开源基础平台的发布就绪状态。**

缺口主要不在“架构思想”，而在：

- 真实硬件闭环证据不足
- 平台公共契约文档不足
- 支持矩阵和验证矩阵不足
- 对外发布边界和维护规则不足

---

## 3. 就绪度分级

建议把平台成熟度拆成 5 级：

| 级别 | 名称 | 含义 |
| --- | --- | --- |
| L0 | Idea | 只有方向和概念，没有稳定边界 |
| L1 | Skeleton | 有分层骨架和初始代码，但主要是内部验证 |
| L2 | Internal Platform | 有真实硬件闭环，能在内部复用 |
| L3 | Open Beta | 对外可试用，但支持范围有限 |
| L4 | Open Foundation | 可作为长期维护的开源基础平台 |

当前建议判断：

- `openBattery` 当前大致处于 `L1 -> L2` 过渡状态。
- 达到 `L2` 的关键，是完成真实 SoC 闭环和最小策略闭环。
- 达到 `L3` 的关键，是补齐对外文档、支持矩阵和最小示例。
- 达到 `L4` 的关键，是验证矩阵、发布规范和长期维护约束成型。

---

## 4. 开源就绪的硬条件

以下 6 类条件，缺任何一类，都不应宣称“已具备开源基础平台就绪度”。

### 4.1 架构硬条件

必须满足：

1. 平台分层已经固定：
   - `app`
   - `services`
   - `drivers/soc`
   - `hal`
   - `board`
   - `config`
2. 依赖边界明确且被遵守：
   - `app` 不直接访问 SoC 寄存器
   - `services` 不直接 include `hal_i2c.h` / `hal_adc.h`
   - vendor 寄存器表不进入 `app` / `services`
3. 平台公共 API 已从 vendor 细节中抽离。
4. profile 裁剪模型已成为唯一配置真相。

当前状态：

- 基本满足骨架要求。
- 仍需继续用代码 review 和后续实现防止回退。

### 4.2 接口契约硬条件

必须满足：

1. `drivers/soc` 公共 API 文档化。
2. 单位口径固定并统一：
   - 电压：`mV`
   - 电流：`mA`
   - 功率：`mW`
   - 温度：摄氏度整数
3. 事件类型、错误码、状态码有统一定义。
4. 对外说明哪些能力是平台能力，哪些是 vendor 私有能力。

当前状态：

- 架构文档已写出方向。
- 还缺独立的 API 契约文档和稳定示例。

### 4.3 硬件闭环硬条件

必须满足：

1. 至少一颗真实 SoC 完成联调：
   - 初始化
   - 读状态
   - 下发参数
   - 中断或事件回调
2. 至少一份真实 I2C 波形抓包记录存在。
3. 快路径 / 慢路径已至少验证一条真实链路。
4. 至少一块 F030 候补样片完成最小验证。

当前状态：

- 尚未满足。
- 这是当前最主要短板。

### 4.4 平台复用硬条件

必须满足：

1. 至少支持两类变化中的一类真实落地：
   - 更换 SoC vendor adapter
   - 更换 MCU/board profile
2. 至少有一个最小示例说明：
   - 如何接入一颗 SoC
   - 如何新增一个 board
   - 如何切换一个 profile
3. 平台不能只对当前单板单器件有效。

当前状态：

- 已有设计方向。
- 还缺真实复用案例和最小示例。

### 4.5 文档与交付硬条件

必须满足：

1. 至少有以下对外文档：
   - 顶层架构说明
   - 平台支持矩阵
   - SoC API 契约说明
   - board porting 指南
   - profile 使用说明
   - 最小 bring-up / demo 指南
2. 文档中必须明确：
   - 已验证
   - 未验证
   - 不支持
3. 文档不能依赖内部上下文才能理解。

当前状态：

- 顶层架构和路线文档已具备。
- 对外接入所需文档仍明显不足。

### 4.6 发布治理硬条件

必须满足：

1. 有明确 license。
2. 有版本/分支发布规则。
3. 有 issue / known limitations 归档规则。
4. 有对外贡献边界：
   - 哪些目录允许扩展
   - 哪些接口不能随意破坏
5. 有“支持范围声明”，避免外部用户误以为平台已支持所有 MCU / 所有 SoC。

当前状态：

- 尚未形成完整治理约束。

---

## 5. 开源平台最小交付包

如果未来要发布第一版开源基础平台，最小交付包建议包含以下内容：

### 5.1 代码

1. `F103 bring-up` 参考工程
2. `drivers/soc` 公共 API
3. 至少一个真实 SoC adapter
4. 至少一个最小 `services` 闭环
5. 至少一个 `PROFILE_F030_MIN_PRODUCT`

### 5.2 文档

1. `architecture`
2. `support matrix`
3. `porting guide`
4. `soc api guide`
5. `bring-up guide`
6. `verification status`

### 5.3 证明材料

1. SoC 联调记录
2. I2C 抓包图
3. F030 样片验证结果
4. Flash / RAM 资源记录
5. 异常恢复记录

---

## 6. 支持矩阵要求

开源平台不能只写“支持多 MCU / 多 SoC”，必须明确支持矩阵。

建议至少维护两张表。

### 6.1 MCU / Board 支持矩阵

| 对象 | 状态 | 说明 |
| --- | --- | --- |
| `STM32F103C8T6` 当前学习板 | Bring-up reference | 当前主开发入口 |
| `STM32F030C8T6` 基准样片 | Pending validation | 量产候补验证对象 |
| 国产 F030 候补样片 | Pending validation | 用于成本替代验证 |
| `STM32G0 / GD32E230` | Fallback candidate | 仅作为回退路线，不应提前宣称支持 |

### 6.2 SoC 支持矩阵

| 对象 | 状态 | 说明 |
| --- | --- | --- |
| `demo_soc` | Internal only | 仅用于软件结构验证，不代表真实支持 |
| 首颗真实 SoC（英集芯或智融） | Pending real bring-up | 完成联调后才能写为 supported |
| 第二颗 SoC | Future | 不应提前宣传 |

原则：

- `demo` 不等于 `supported`
- `能编译` 不等于 `已验证`
- `有文档` 不等于 `已支持`

---

## 7. 验证矩阵要求

一个开源基础平台，必须能告诉外部用户“哪些东西真的验证过”。

建议至少维护以下验证矩阵：

### 7.1 构建验证

- `F103 profile` 构建通过
- `F030 profile` 构建通过

### 7.2 硬件验证

- 下载
- UART 日志
- I2C 读写
- EXTI
- ADC
- watchdog

### 7.3 平台能力验证

- SoC 初始化
- 电压 / 温度读取
- 参数下发
- 事件回调
- 过温降额
- 故障锁存

### 7.4 异常验证

- I2C 超时
- 总线挂死恢复
- SoC 无响应
- 中断异常
- 恢复或降级路径

---

## 8. 当前缺口映射到后续计划

当前缺口与 `1_Plan/Plan-Stage-Post-0518.md` 的关系如下：

| 当前缺口 | 对应阶段 |
| --- | --- |
| 真实 SoC 初始化与 I2C 闭环缺失 | `Stage A` |
| 服务层最小策略闭环缺失 | `Stage B` |
| F030 样片验证缺失 | `Stage C` |
| 最小量产功能集与资源预算缺失 | `Stage D` |
| 验证矩阵、支持矩阵、对外资料缺失 | `Stage E` |

因此，当前最合理的判断是：

> 在 `Stage A ~ C` 完成之前，`openBattery` 仍不应对外宣称“已经是开源基础平台”，最多只能说“正在朝开源基础平台演进”。  

---

## 9. 不应误用的表述

在未达到相应就绪条件前，不应使用以下表述：

1. “已经支持多 SoC”
2. “已经是开源平台”
3. “F030 已确定为量产主控”
4. “平台已可复用到量产”
5. “当前 demo adapter 代表真实芯片已支持”

更准确的表达应该是：

- “已形成平台骨架”
- “已完成内部结构验证”
- “正在补真实硬件闭环”
- “F030 当前为量产候补，待样片验证”

---

## 10. 当前建议结论

当前 `openBattery` 的平台成熟度建议定义为：

> **L1 / Skeleton，正在向 L2 / Internal Platform 推进。**

要进入下一层，最关键的 4 个动作是：

1. 完成真实 SoC 初始化、读值、事件闭环  
2. 完成最小 `services` 策略闭环  
3. 完成 F030 样片验证  
4. 补齐平台对外契约文档和验证矩阵  

只要这四件事打实，`openBattery` 就会从“方向正确的 startup 骨架”，进入“真正可内部复用、并可准备对外开源”的平台阶段。
