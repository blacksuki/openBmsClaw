# AI Native 团队 Skill 体系与推进手册

> 最新更新：2026-07-17
> 文档状态：Draft
> 文档性质：团队工作方法 / skill 体系说明
> 适用范围：`openBattery` 当前阶段的对外沟通、资料理解、方案判断和联调问题收口

---

## 1. 结论先说

当前团队完全可以把 `Agent + skill` 作为一种 `AI Native` 工作方式，用来补足嵌入式理解和原厂沟通上的能力短板。

但这个体系的正确定位必须保持清楚：

- skill 是项目过程中的手段，不是项目本身；
- skill 是为了增强团队的理解、把关和输出质量，不是为了直接替代嵌入式开发；
- 第一阶段最适合做的是文档型、判断型和对外沟通型 skill；
- 当前 skill 的核心价值不在“自动写代码”，而在“减少认知失真，提升对外输出专业度”。

换句话说，当前更合理的目标不是“做一个嵌入式全能 AI”，而是：

> 把团队最容易失真的几类认知工作拆开，沉淀成稳定、可复用、可交付的 skill，让团队在合作推进过程中持续提升专业性和判断质量。

---

## 2. 为什么当前阶段值得做 skill 体系

结合当前团队背景与和英集芯合作的现实，最值得优先 skill 化的工作，不是直接开发，而是下面这类更稳定、更高频、也更容易出错的认知工作：

- 把产品诉求翻译成 SoC 接口需求；
- 把 `datasheet`、寄存器说明、参考设计整理成团队可执行知识；
- 把现场问题整理成原厂愿意高效响应的问题单；
- 判断哪些需求该问原厂，哪些应由我方自己消化，哪些应该延后；
- 判断某项能力是“芯片具备”，还是“当前样板可稳定落地”。

这些工作一旦做得稳定，团队的对外输出质量、会议效率和合作可信度都会明显提升。

## 3. skill 设计原则

### 3.1 每个 skill 只解决一种认知问题

不要把一个 skill 设计成“嵌入式全能助手”。

更合理的做法是：

- 一个 skill 只解决一个判断任务；
- 一个 skill 只对应一类输入；
- 一个 skill 只产生一组固定输出；
- 一个 skill 只在特定场景触发。

这样做的好处是：

- 输出更稳定；
- 更容易检查质量；
- 更容易积累模板和样例；
- 更容易在团队内分工使用。

### 3.2 每个 skill 都必须定义 4 件事

每个 skill 必须明确写清楚：

1. 输入是什么
2. 输出是什么
3. 验收标准是什么
4. 何时交回给人，或者需要人执行什么动作

如果这 4 件事不清楚，skill 很容易退化成“有时有用、有时没法落地”的聊天工具。

### 3.3 每个 skill 的输出都必须落成固定工件

skill 的输出不能只停留在对话结论里，必须沉淀成固定工件。

例如：

- `interface_matrix.md`
- `datasheet_notes.md`
- `issue_report.md`
- `boundary_decision.md`
- `reality_check.md`

只有落成固定工件，skill 才能真正承担：

- 对外沟通前的准备节点；
- 对内评审时的输入材料；
- 联调推进中的交接资料；
- 后续复用时的知识资产。

### 3.4 第一阶段先做文档型和判断型 skill

当前最合适的优先级不是“让 Agent 直接写代码”，而是先做：

- 文档型 skill
- 判断型 skill
- 对外沟通输入输出型 skill

原因很现实：

- 当前团队的主要短板首先是理解和判断；
- 对原厂合作来说，专业性首先体现在问题定义、边界表达和材料质量上；
- 文档型和判断型 skill 更容易标准化，也更容易验证有效性。

### 3.5 skill 是增强团队把关能力，不是替代工程责任

skill 可以帮助团队：

- 快速消化资料；
- 形成结构化结论；
- 暴露理解空白；
- 提升问题单质量；
- 强化会议前准备和会后收口。

但 skill 不能替代：

- 硬件实测
- 板级联调
- 安全边界判断
- 热设计判断
- 量产可行性判断
- 最终工程责任签字

因此，skill 的定位应始终是“辅助增强”，而不是“责任转移”。

## 4. 统一 skill 模板

为了避免每个 skill 风格不一、产出不稳，建议统一采用下面的模板：

### 4.1 基本字段

- `skill 名称`
- `适用场景`
- `不适用场景`
- `输入`
- `输出`
- `验收标准`
- `交回给人的条件`
- `需要人执行的动作`

### 4.2 输入模板建议

每个 skill 的输入至少应覆盖：

- 当前项目目标
- 当前阶段范围
- 当前料号 / 版本
- 当前问题描述或待判断问题
- 当前已知文档和证据
- 当前不做项和边界条件

### 4.3 输出模板建议

每个 skill 的输出至少应覆盖：

- 本次结论
- 事实与推断的区分
- 未知项
- 风险项
- 下一步动作
- 是否可以直接对外使用

### 4.4 交回给人的条件模板

出现下面这些情况时，应停止自动推进，交回给人判断：

- 文档互相冲突；
- 关键限制缺失；
- 安全边界不清楚；
- 需要实板验证；
- 已进入商务或责任边界判断。

### 5.1 基于启动期团队配置的优先级判断原则

结合 [17.injoinic_cooperation_analysis.md](/abs/path/C:/Users/MingZhengHuo/workspace/101.openBattery/openBmsClaw/0_System/17.injoinic_cooperation_analysis.md:930) 中已经确认的启动期团队配置，当前 skill 的优先级不应按“看起来厉不厉害”排序，而应按下面 4 条原则排序：

#### 1. 是否直接服务最关键的 3 个角色

当前最关键的角色不是全体成员平均受益，而是：

- 项目负责人
- 系统架构 / 技术负责人
- 嵌入式集成工程师

优先 skill 应该首先增强这 3 个角色的核心交付物。

#### 2. 是否直接降低对英集芯沟通成本

当前阶段最值钱的，不是让 Agent 多做一些事情，而是让团队：

- 更会提问题；
- 更会整理资料；
- 更会给出可执行、可复现、可回应的材料。

因此，凡是能直接改善对原厂输入质量的 skill，优先级都应该更高。

#### 3. 是否在没有大量实板条件下也能先跑起来

第一阶段更适合先建设那些：

- 不依赖完整板测闭环；
- 不要求大量实时硬件数据；
- 只要有资料、需求和基础问题记录就能先落地的 skill。

这决定了 skill 能不能尽快进入真实工作流。

#### 4. 是否能产出立即可复用的固定工件

优先级更高的 skill，应当在第一次使用后就能沉淀出后续仍可复用的工件。

例如：

- 接口清单
- 手册笔记
- 对外问题单

这些工件不仅服务当前轮沟通，也能服务下一轮沟通、下一颗料号和后续内部培训。

### 5.2 先做哪 3 个最值

基于上面的排序原则，当前最值得先做的 3 个 skill 建议如下。

#### P0-1. `soc-interface-modeler`

优先级结论：

- 这是第一优先级；
- 由系统架构 / 技术负责人主用，项目负责人协同；
- 它直接决定团队和英集芯沟通时，到底是在讲愿景，还是在讲可执行接口。

为什么它排第一：

1. 它直接服务当前最关键的技术窗口角色。
2. 它对应了启动期最危险的第一个能力缺口：
   “没人负责把产品语言翻译成 SoC 接口需求”。
3. 它能最直接改善第一次技术沟通、接口确认和 `SAL` 边界收敛。
4. 它产出的工件可以直接进入会议提纲、接口清单和内部架构讨论。

当前阶段最主要的输出价值：

- 把“我们想做什么”变成“原厂可以回答什么”；
- 把“想开放更多能力”变成“哪些状态、参数、事件、限制需要确认”；
- 防止对外沟通停留在泛泛表达。

如果没有它，后续很多沟通都会失真，甚至会把拿不到的能力写进架构设想里。

#### P0-2. `datasheet-digester`

优先级结论：

- 这是第二优先级；
- 由系统架构 / 技术负责人和嵌入式集成工程师共同主用；
- 它是整个 skill 体系的资料底座。

为什么它排第二：

1. 它对应了启动期最危险的第二个能力缺口：
   “没人能把原厂资料翻译成团队可执行的联调动作”。
2. 软件背景团队最容易在这里失分，尤其是在 `datasheet`、寄存器说明、参考设计理解上。
3. 它不依赖完整实板条件，最容易先跑起来。
4. 它沉淀出来的工件几乎是所有其他 skill 的上游输入。

当前阶段最主要的输出价值：

- 把厚手册压缩成团队真正要看的关键页和关键点；
- 区分“文档明确写了什么”和“目前只是推断什么”；
- 为 `soc-interface-modeler`、`embedded-reality-checker` 和后续联调提供统一知识底稿。

如果没有它，团队即使开了很多会，也很容易因为资料理解不一致而反复返工。

#### P1. `vendor-issue-closer`

优先级结论：

- 这是第三优先级；
- 由嵌入式集成工程师主用，测试验证 / 联调支持协同；
- 它是把“沟通质量”真正拉到原厂合作标准的关键 skill。

为什么它排第三：

1. 它对应了启动期最危险的第三个能力缺口：
   “没人能把现场问题整理成原厂愿意高效响应的问题单”。
2. 它最直接影响英集芯后续是否愿意持续投入支持。
3. 它虽然依赖一定联调证据，但一旦进入联调阶段，价值会立刻放大。
4. 它能显著减少来回沟通中的噪音和重复问答。

当前阶段最主要的输出价值：

- 把“这里不工作”变成结构化问题单；
- 把现象、条件、配置、证据、已排查项、待确认点整理清楚；
- 让英集芯看到团队具备基本技术承接能力。

如果没有它，即使前两项做得不错，进入联调后也会很快暴露出问题收口能力不足。

### 5.3 为什么另外 2 个 skill 不建议排进前三

下面两个 skill 很重要，但不建议在第一轮就排进“最先做的 3 个”。

#### `cooperation-boundary-reviewer`

它的重要性在于防止范围失控、避免提出不合理诉求。

但它当前更适合作为：

- 项目负责人 + 系统技术负责人共同使用的审查器；
- 建立在前两个 skill 已经产出一定接口和资料工件基础上的二次判断工具。

换句话说，它现在更像是“收口器”，而不是第一批最该先落地的“发动机”。

#### `embedded-reality-checker`

它的重要性在于防止把“芯片能力”误写成“当前就能落地的产品能力”。

但它当前更适合作为：

- 方案评审和对外口径确认时的复核器；
- 建立在已经读过资料、已经形成接口理解、已经有初步样板判断基础上的校验工具。

也就是说，它很重要，但在第一轮落地时更像“质量闸门”，而不是最先要建的“生产器”。

### 5.4 建议的分批建设顺序

更实际的建设顺序建议如下：

#### 第一批：立即建设

1. `soc-interface-modeler`
2. `datasheet-digester`
3. `vendor-issue-closer`

目标：

- 先把对外输入质量、资料理解质量、问题升级质量做起来。

#### 第二批：在第一批跑通后补上

4. `cooperation-boundary-reviewer`
5. `embedded-reality-checker`

目标：

- 再把范围控制、现实校验和对外口径收口做强。

### 5.5 对当前团队更直接的建议

如果现在只能先做 1 到 2 个，那么建议顺序是：

1. 先做 `soc-interface-modeler`
2. 再做 `datasheet-digester`

如果可以一次做 3 个，就加上：

3. `vendor-issue-closer`

这套组合最符合当前启动期团队结构，也最能直接支撑和英集芯合作时的专业输出。

## 6. 作为 AI Native 团队的推进方式

### 6.1 先做“认知基础设施”，不是先做“自动开发”

当前团队更现实的推进方式，不是先追求 Agent 自动完成开发，而是先把下面三类基础设施做稳：

1. `资料基础设施`
   把手册、寄存器、参考设计整理成稳定可复用的知识工件。
2. `沟通基础设施`
   把对外提问、会前准备、会后收口做成固定格式。
3. `判断基础设施`
   把边界判断、现实校验和阶段范围控制做成固定审查动作。

只要这三类基础设施稳定下来，团队在嵌入式经验不足的前提下，仍然可以显著提升合作质量。

### 6.2 先服务外部沟通，再服务内部联调

第一阶段建议 skill 先服务下面这些输出：

- 对外会前问题清单
- 对外会后结论整理
- 方案评审判断材料
- 联调问题升级材料

原因是这些场景：

- 出错成本高；
- 最容易暴露团队专业性；
- 最适合用固定工件来提升质量。

只有在这些基础场景跑顺之后，才适合进一步把 skill 扩展到更贴近联调和验证的场景。

### 6.3 先让 skill 帮团队“问对问题”，再考虑“做对事情”

当前团队最应该先解决的，不是“Agent 能不能帮我多做点”，而是：

- 团队能不能更会读资料；
- 团队能不能更会提问题；
- 团队能不能更会判断边界；
- 团队能不能更会把结论讲清楚。

如果这几件事没有先成立，即使 Agent 能输出很多内容，也很容易只是增加噪音。

### 6.4 先在小闭环里验证 skill，再做长期沉淀

更合适的推进顺序是：

1. 先确定 3 到 5 个最核心 skill
2. 给每个 skill 固定输入模板和输出模板
3. 先只服务于 1 个料号、1 个样板、1 轮合作推进
4. 每次使用后复盘：输出有没有帮上忙，哪里仍然失真
5. 有效后再沉淀为团队长期工作方式

这样可以避免一开始做出很多 skill，但没有一个真正进入工作流。

## 7. 关键节点与工件思路

如果希望 skill 真正成为团队稳定能力，而不是临时工具，它的输出必须进入项目关键节点。

### 7.1 料号筛选节点

目的：

- 快速判断某料号是否值得进入下一轮沟通或样板验证

建议调用：

- `datasheet-digester`
- `embedded-reality-checker`

应输出：

- `datasheet_notes.md`
- `capability_vs_product_gap.md`
- `evidence_needed.md`

人需要做的动作：

- 决定是否继续投入该料号
- 决定是否需要向英集芯追问关键限制

### 7.2 对外首次技术沟通前节点

目的：

- 把产品诉求转换成原厂能回答的问题

建议调用：

- `soc-interface-modeler`
- `cooperation-boundary-reviewer`

应输出：

- `interface_matrix.md`
- `open_questions_for_vendor.md`
- `ask_vs_build_vs_defer.md`

人需要做的动作：

- 确认问题优先级
- 删除不成熟或超范围的问题
- 最终形成会议提纲

### 7.3 对外会议结束后节点

目的：

- 防止会议结论散失，及时把新信息并入当前认知

建议调用：

- `soc-interface-modeler`
- `datasheet-digester`

应输出：

- `meeting_outcome.md`
- `confirmed_interface_updates.md`
- `unknowns_after_meeting.md`

人需要做的动作：

- 确认会议纪要
- 指定下一轮验证动作
- 明确哪些结论能进入公开团队资料

### 7.4 联调问题升级前节点

目的：

- 让问题单足够清楚，减少和原厂来回往返

建议调用：

- `vendor-issue-closer`

应输出：

- `issue_report.md`
- `hypothesis_list.md`
- `next_test_actions.md`

人需要做的动作：

- 补齐日志、波形或接线信息
- 判断是否已经具备对外升级条件

### 7.5 方案评审节点

目的：

- 防止把“看起来可行”的方案过早写成“已经可落地”

建议调用：

- `cooperation-boundary-reviewer`
- `embedded-reality-checker`

应输出：

- `boundary_decision.md`
- `reality_check.md`
- `risk_and_validation_list.md`

人需要做的动作：

- 决定是否进入下一阶段
- 决定是否需要补充验证资源

### 7.6 阶段复盘节点

目的：

- 判断当前 skill 是否真的帮助团队减少误判

建议调用：

- 任一已使用过的 skill，配合人工复盘

应输出：

- `skill_effect_review.md`
- `template_update_list.md`

人需要做的动作：

- 判断哪些模板要改
- 判断哪些 skill 暂不值得继续投入

## 8. 当前阶段最实际的落地建议

基于当前阶段，更实际的推进方式不是一下子建设很多 skill，而是：

1. 先确定 3 到 5 个核心 skill
2. 先把 `输入模板 + 输出模板 + 验收标准 + 交回条件` 固定下来
3. 先只服务外部沟通和内部判断，不急着服务直接开发
4. 先在 1 个料号、1 条联调链路、1 轮合作推进里验证有效性
5. 有效后再决定是否继续沉淀到 `.agents/skills/`

这条路线的重点不在“Agent 做了多少事”，而在：

- 团队是否更会读资料；
- 团队是否更会提问题；
- 团队是否更会判断边界；
- 团队是否更能把输出做得专业、稳定、可复用。

如果这几件事成立，那么 `AI Native` 对当前团队的真正价值就成立了。
