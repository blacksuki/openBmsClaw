## 0. 状态约定与阅读须知

本文严格区分四种状态,不得混淆(沿用总文档纪律):

- **【已确认输入】** 用户/项目已明确给定。
- **【当前假设】** 尚无证据支撑的判断,不得当成事实。
- **【待验证】** 需要证据(打样/日志/抓包/供应商资料/市场调研)才能落地。
- **【规划中】** 本 PRD 定义的目标需求,当前代码**尚未实现**。

> 重要:本 PRD 全部条目默认为 **规划中**,不代表已实现。当前代码事实以 `openBmsClaw/` 为准(现处于 F103 bring-up 基线,连接/记录能力尚未落地)。

## 1. 产品定位与主线

**主线(本次共识):以「MCU + 专用电源 SoC」双芯协同为底座,辅以 NFC(ST25DV)作为数据读出窗口,面向日本中端客户,围绕「安全 + 功能 + 数据可展示可追踪」形成差异化的移动电源平台。**

- 【已确认输入】首要客户定位:**日本中端客户**。
- 【已确认输入】技术底座:**MCU + 专用电源 SoC**(英集芯为当前 SoC 合作主线)。
- 【已确认输入】数据读出主线:**NFC / ST25DV**(tap 读取);实时/隔空/远程监控/OTA **不在需求内**(2026-07-18 锁定)。
- 【当前假设】日本中端客户愿意为「安全可视 + 数据可追踪」付费——**此为核心假设,尚无付费/渠道行为证据支撑,须验证**(见 §9)。

## 2. 目标客户与性能指标(中端客户)

### 2.1 目标客户
- 【已确认输入】面向**日本市场中端客户**。
- 【待验证】客户身份需收敛:是 C 端中端用户、中小移动电源厂商、还是大手供应链模组客户。三者对认证、商业模式、SKU 定义影响不同,当前唯一具名线索指向大手供应链(未确认到具体项目),须先定性。

### 2.2 性能指标(规划中,具体数值待定)
「中端」是定位而非规格,以下轴线需在首发 SKU 定型时填入具体值:

| 指标 | 目标方向 | 状态 |
|---|---|---|
| 电池容量 | 中端主流区间(便携,受 ≤100Wh 约束) | 【待验证】具体容量待定 |
| 最大输入功率 | 自充速度,中端主流水平(目标待定) | 【待验证】具体值待定 |
| 最大输出功率 | 总输出 + 单口最大,分别标定(目标待定) | 【待验证】具体值待定 |
| 输出功率档位 | 覆盖中端快充主流档位 | 【待验证】具体档位待定 |
| SOC 精度 | 电量(SOC)显示误差(±%),tap 可读;影响用户信任 | 【待验证】目标 ±X% 待定 |
| 接口数量/类型 | 以 Type-C 为主,数量待定 | 【待验证】 |
| 厚度/重量 | 便携约束下的中端水平 | 【待验证】 |
| 转换效率/温升 | 中端合理区间,满足安全温控 | 【待验证】需样机热测 |

> 形态说明:磁吸为**可选形态**,非硬约束(可换普通形态)。

## 3. 功能需求(Functional Requirements)

按「安全 / 功能 / 数据可展示可追踪」三块拆分。优先级:P0=首发必须,P1=首发应有,P2=后续。

### 3.1 安全(Safety Function)

| ID | 需求 | 优先级 | 说明/状态 |
|---|---|---|---|
| FR-S1 | 过压/过流/过温/短路保护 | P0 | **快保护由 SoC + 保护件 + 保险丝 + NTC 硬件承担**(总文档 5.3);【规划中】 |
| FR-S2 | MCU 慢路径安全策略 | P0 | 读状态→降额/锁定/提示/恢复,不替代硬件快保护;【规划中】 |
| FR-S3 | 温度采样与热策略 | P0 | NTC 输入,过温降额/告警;【规划中,待样机热测】 |
| FR-S4 | 异常状态锁存与恢复 | P1 | 快路径锁存(EXTI/INT),慢路径记录+恢复;【规划中】 |

### 3.2 功能(Core Function)

| ID | 需求 | 优先级 | 说明/状态 |
|---|---|---|---|
| FR-F1 | SoC 电源转换/协议/端口控制 | P0 | 由电源 SoC 承担,MCU 经 SAL 读写;【规划中】 |
| FR-F2 | MCU 状态读取与策略仲裁 | P0 | 电压/电流/温度/端口/故障状态读取;【规划中,依赖英集芯接口】 |
| FR-F3 | 电量(SOC)计算与显示 | P0 | tap 可读电量;【规划中】 |
| FR-F4 | SOH/循环寿命相关计算 | P1 | 在数据层之上做计算;【规划中】 |
| FR-F5 | 快充协议支持(PD/QC/UFCS) | P0 | **由电源 SoC 提供**,MCU 经 SAL 配置/读取;目标 **PD3.0 / PD3.1 SPR(~100W)**;PD3.1 EPR(>100W,28/36/48V,最高 240W)原则不在中端便携/≤100Wh 范围;【待验证,依赖 SoC 能力与 SKU 定型】 |

### 3.3 数据可展示可追踪(差异化核心)

**需求本质:数据为「安全合规 + 监控」而记录;用户以 tap 读取当前状态,再次 tap 取更新;实时查询非需求。**

| ID | 需求 | 优先级 | 说明/状态 |
|---|---|---|---|
| FR-D1 | 定义 v0 电池状态数据模型(带版本) | P0 | 快照 + 记录两部分,一次定义、多传输层复用;【规划中】 |
| FR-D2 | 即时快照可读 | P0 | 电量、电压、温度、故障标志等,tap 读取;【规划中】 |
| FR-D3 | 记录/日志(可追踪) | P0 | **充放电次数、过温/故障事件+时间戳、累计运行时长**;**MCU 侧掉电持久化**(内部 Flash/EEPROM);为安全合规/售后诊断服务;【规划中】 |
| FR-D4 | NFC tap 读出 | P0 | MCU 经 I²C 把快照+日志摘要写入 ST25DV,手机/Web NFC tap 读回;再次 tap 取更新;【规划中】 |
| FR-D5 | 断电可读 | P1 | ST25DV EEPROM 掉电保留,充电宝没电也能贴读最后状态;【规划中,依赖天线验证】 |
| FR-D6 | 免 App 读取(可选) | P2 | Type5/ISO15693 支持 Web NFC(Android),降低 App 门槛;【规划中】 |
| FR-D7 | 日志管理策略 | P1 | 定义**记录内容、容量上限、满时覆盖/轮转策略、读取与清除接口、掉电保存**;区分“关键计数/事件(小体量,合规必须)”与“大量历史序列(不进 F030 最小版)”;与 tech-arch F030 约束一致;【规划中】 |

## 4. 安全要求(符合日本市场)

- 【已确认输入】需满足日本市场安全相关要求。
- 【规划中】保护链:过压、过流、过温、短路,硬件优先兜底(FR-S1)。
- 【待验证】电池容量便携约束(≤100Wh 边界意识,涉及航空携带);量产还需哪些额外硬件保护件,须结合合规与实际使用风险拆解。
- 【原则】不得以「MCU 软件判断」替代硬件快保护(总文档 5.3、10.2)。

## 5. 成本目标(基于成本分析)

> 价格来源:**LCSC 国际版 lcsc.com 公开阶梯价(USD,2026-07)**。国内立创商城(szlcsc)价格本次未取到;量产报价须以国内自营/代理实时询价为准。BLE 模组、国产蓝牙芯单价未核实,为市场概算。

### 5.1 关键器件成本锚(实取)

| 器件 | 角色 | 实价(LCSC 国际版) |
|---|---|---|
| STM32G031 系 | 最省主控(NFC 路线足够) | $2.01(1+)→ **$1.30(1k)** |
| STM32G0B1 | 主控(带 USB,若需 Type-C 数据) | $4.61(1+)→ **$2.95(1k)** |
| ST25DV04K | NFC 动态标签(读出窗口) | **~$0.40–0.65/个(5k+)** |
| (参考)STM32WB09 | 若走 BLE 单芯 | $4.04(1+)→ $2.63(1k) |

### 5.2 成本目标与结论
- **NFC 路线成本最优:** 只需「便宜 MCU(F0/G0)+ 一颗 ST25DV(<$1)」,无射频芯、**无日本技適认证成本**。
- 目标:MCU + NFC 读出部分的增量 BOM 控制在低成本区间(具体目标值随首发 SKU 定型);【待验证:国内 szlcsc 实价】。
- 对比:走 BLE 会引入模组成本 + 技適认证成本;当前需求不含实时,故不承担该成本。

## 7. 产品路线与技术架构

### 7.1 架构主线(规划中)
```text
[电芯]→[电源SoC(英集芯)]→ I²C →┌──────────────┐→ I²C →[ST25DV]))) NFC )))[手机 tap]
                                │ MCU(F0/G0/F103)│
                                │ ·BMS策略/仲裁   │
                                │ ·SOC/SOH计算    │
                                │ ·掉电持久化日志  │
                                └──────────────┘
```
- **电源 SoC(Body):** 升降压、协议握手、快保护。
- **MCU(Brain):** 策略、状态读取、日志记录、驱动 ST25DV。
- **ST25DV:** 被动 NFC 标签,MCU 经 I²C 写、手机经 RF 读;**非单片机,不跑固件**。
- 分层沿用 `app/ → services/ → drivers/soc(SAL) → drivers/ → hal/ → board/ → config/`;ST25DV 作为 `drivers/` 下新增 I²C 外设,走现有 `hal/i2c`,架构不推翻。

### 7.2 MCU 选型方向
- 【规划中】NFC 路线仅需 I²C → **最省 F0/G0 即可**,现有 F103 可先验证。
- 量产候补仍评估 F030/G0 档位;NFC 无射频、只 I²C,资源压力小(不同于 BLE)。

### 7.3 范围界定
- **In scope(首发):** MCU+SoC 双芯、NFC(ST25DV)tap 读出、MCU 侧掉电日志、安全保护链、v0 数据模型。
- **Out of scope(锁定不做):** 实时/隔空监测、远程监控、隔空 OTA、复杂 BLE 协议栈、大型 UI/重显示、多 SoC/多屏并发。BLE 仅列为未来 premium 备选。
- **可选后续(非首发,P2):** 无线充电(Qi)——与总文档 §3.3 一致列为第二阶段扩展项;涉及线圈/热/成本,对软件出身团队与首发是实风险,首发不默认包含。
- **PD3.1 EPR(>100W):** 原则不在中端便携/≤100Wh 首发范围(见 FR-F5);如未来上探需重新评估电芯/热/成本。

## 8. 验收标准(Acceptance)

- 【规划中】存在 v0 电池状态数据模型(快照 + 记录,带 `schema_version` 版本约定)。
- 【规划中】MCU 经 I²C 把快照+日志摘要写入 ST25DV;手机/Web NFC tap 读到 `电量` 与 `充放电次数`;再次 tap 取到更新值。
- 【规划中】MCU 侧日志掉电持久化:重启后 `充放电次数`/故障记录不丢。
- 【规划中】安全保护链在样机上验证(过压/过流/过温/短路),且快保护不依赖 MCU 软件。
- 【待验证】NFC 天线在金属外壳/电芯附近的读距(需铁氧体屏蔽 + 打样实测)。

## 10. 术语与依据

- **MCU:** 微控制器(主控/大脑),本项目 STM32 F103/F0/G0。
- **SoC:** 专用电源管理芯片(Body),承担变换/协议/快保护。
- **SAL:** SoC Abstraction Layer,屏蔽不同 SoC 接口差异。
- **NFC:** 近场通信;本项目用 **ST25DV**(NFC Forum Type 5 / ISO15693,I²C + RF 双接口被动标签,支持能量收割)。
- **PSE:** 日本电气用品安全法认证;**技適:** 日本无线电发射设备认证。
