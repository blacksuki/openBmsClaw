# 附录 12：合作与联调术语表

> 适用范围：与 SoC 原厂、FAE、硬件工程师、嵌入式开发同事沟通时的基础术语对照
> 使用目的：避免我方因不理解对方口中的嵌入式与硬件用语而产生误解
> 使用建议：第一次开会、收资料、做联调、看数据手册时，都先过一遍本文

---

## 1. 这份附录怎么用

这份附录不是学术定义集，而是面向真实合作场景的“沟通翻译表”。

建议用下面的方式理解每个词：

1. 这个词在字面上是什么意思。
2. 对方在什么场景下会说这个词。
3. 我们听到这个词时，真正要追问的是什么。

如果只记住字面意思，还是容易误判。  
例如对方说“支持 I2C”，不等于“支持完整状态读取、参数配置、异常定位和版本稳定”。

---

## 2. 先掌握的核心判断

在合作和开发里，很多误解都来自下面几类混淆：

1. 把“芯片能做”误解成“接口开放给你做”。
2. 把“有数据手册”误解成“寄存器、事件、限制都说清了”。
3. 把“能跑 Demo”误解成“适合量产集成”。
4. 把“支持某协议”误解成“协议状态可读、异常原因可解释”。
5. 把“有 SDK”误解成“以后版本能稳定维护”。

所以本文每个术语后面，都会尽量补一句“真正要确认的点”。

## 3. 产品与芯片基础术语

| 术语 | 简要解释 | 对方常见说法 | 我方真正要确认什么 |
| --- | --- | --- | --- |
| `料号` | 一个具体芯片或器件的完整型号编号 | “首批推荐料号是 `IP5383`” | 是不是具体到可下单、可联调、可拿资料的型号 |
| `SKU` | 一个面向市场或产品管理的具体款型 | “首发先做一个 SKU” | 这是市场款型，不一定等于单一芯片型号 |
| `SoC` | System on Chip，这里通常指高集成度电源管理芯片 | “SoC 负责体力活” | 它具体负责哪些底层能力，哪些能力仍留给 MCU |
| `MCU` | 微控制器，负责控制、策略、日志、交互 | “MCU 侧来做状态管理” | MCU 需要拿到哪些输入、接口和事件 |
| `BMS` | 电池管理系统，负责监测、保护、均衡等 | “BMS 数据怎么读” | 是芯片内部集成，还是外部独立 BMS 芯片 |
| `PMIC` | Power Management IC，电源管理芯片的泛称 | “这是 PMIC，不是完整 SoC” | 它是单一功能芯片，还是全集成方案 |
| `全集成芯片` | 把充放电、协议、保护、测量等做进一颗芯片 | “这颗就是移动电源全集成芯片” | 集成了哪些功能，哪些仍需外部器件或 MCU 配合 |
| `协议芯片` | 偏快充协议控制，不一定含完整功率路径 | “这个料号更偏 PD 协议控制” | 它能否独立承担整机，还是只能做协处理 |
| `主控` | 通常指系统里负责上层控制的 MCU | “主控通过 I2C 读取状态” | 主控和 SoC 的边界是不是已经说清楚 |
| `原厂` | 芯片厂商本人 | “原厂可以提供资料” | 是官网公开资料，还是 NDA 后资料，还是 FAE 口头说明 |
| `FAE` | Field Application Engineer，现场应用工程师 | “让 FAE 支持一下” | 是答疑、联调、参考设计支持，还是仅商务转达 |

## 4. 手册与资料相关术语

| 术语 | 简要解释 | 对方常见说法 | 我方真正要确认什么 |
| --- | --- | --- | --- |
| `Datasheet` | 数据手册，介绍功能、封装、电气参数、典型应用 | “先给你 datasheet” | 里面有没有寄存器、时序、限制、典型电路 |
| `Reference Manual` | 参考手册，通常比 datasheet 更详细 | “详细寄存器在 RM 里” | 是否真的对外提供，还是只内部使用 |
| `Register Map` | 寄存器表，列出地址、位定义、默认值、读写属性 | “I2C 寄存器表后面补” | 没有它基本无法稳定做驱动封装 |
| `Application Note` / `AN` | 应用笔记，解释某场景如何使用芯片 | “有应用笔记可参考” | 是否覆盖我们真实场景，而不是泛泛介绍 |
| `Errata` | 勘误表，列出芯片已知问题 | “某版本有 errata” | 哪些坑会影响联调、量产或软件绕过策略 |
| `Reference Design` | 参考设计，原厂推荐的原理图或板级方案 | “按参考设计做风险最低” | 是完整可落地电路，还是只有示意图 |
| `Schematic` | 原理图 | “把原理图给你们” | 是参考板原理图，还是目标产品原理图 |
| `BOM` | Bill of Materials，物料清单 | “BOM 里外围件有要求” | 哪些外围器件不能随便替换 |
| `Layout` | PCB 布局布线 | “这个芯片对 layout 敏感” | 是电流路径、散热、地线，还是信号完整性要求 |
| `Gerber` | PCB 制造文件 | “必要时可以参考 Gerber” | 更偏硬件制造，不等于软件接口资料 |
| `NDA` | 保密协议 | “签 NDA 后可给详细资料” | 签了后能拿到哪些关键资料 |
| `版本冻结` | 某资料或芯片版本在一定周期内不再变化 | “先按这个版本冻结” | 是芯片版本冻结，还是寄存器/API 文档冻结 |

## 5. 寄存器与接口资料术语

| 术语 | 简要解释 | 对方常见说法 | 我方真正要确认什么 |
| --- | --- | --- | --- |
| `寄存器` | 芯片内部可读写的配置或状态单元 | “状态在寄存器里读” | 哪些寄存器可读，哪些可写，哪些不能碰 |
| `地址` | 寄存器地址或 I2C 从机地址 | “地址是 `0xEA`” | 这是 7-bit、8-bit 还是寄存器地址，不要混淆 |
| `默认值` | 芯片上电后的寄存器初始值 | “默认已经开这个功能” | 默认值是否稳定、不同版本是否一致 |
| `R/W` | Read/Write，寄存器可读可写 | “这个寄存器是 R/W” | 写入后何时生效，是否要重启或软复位 |
| `RO` | Read Only，只读 | “故障状态是 RO” | 只读是否会自动清除，还是需额外动作清除 |
| `WO` | Write Only，只写 | “写这个寄存器触发动作” | 写入格式、时序、是否有危险操作 |
| `Bit` | 比特位 | “看 bit 3 是不是置位” | 对方说的是单个位还是一段位域 |
| `Bit Field` | 位域，多位组合表达一个含义 | “bit[4:0] 是 OCP 阈值” | 要确认换算关系，而不是只记位号 |
| `Mask` | 位掩码，用于读写某几个 bit | “按 mask 修改” | 写寄存器时要避免误伤其他位 |
| `LSB` | Least Significant Bit，最小单位步进 | “1 LSB = 10mA” | 寄存器原始值怎么换算成真实物理量 |
| `Reserved` | 保留位 | “保留位必须写 0” | 能不能保持原值，千万别随便写 |
| `Status Register` | 状态寄存器 | “先读状态寄存器确认” | 是实时状态、锁存状态，还是故障历史 |
| `Fault Register` | 故障寄存器 | “异常原因在 fault 里” | 故障如何清除，是否可区分热/流/协议异常 |
| `Config Register` | 配置寄存器 | “限流参数在 config 里配” | 是一次性写入，还是运行时可动态改 |
| `Latch` / `锁存` | 事件发生后状态保持，不会立刻消失 | “故障是锁存的” | 是好事，便于排查；但要问清如何清除 |
| `Clear on Read` | 读后清除 | “读一下就会清掉” | 调试时别因为多读了一次把证据读没了 |
| `软复位` | 软件触发芯片重新初始化 | “改配置后需要 soft reset” | 是否会丢状态、是否影响正在充放电 |

## 6. 通信与总线术语

| 术语 | 简要解释 | 对方常见说法 | 我方真正要确认什么 |
| --- | --- | --- | --- |
| `I2C` | 双线串行总线，常用于 MCU 与 SoC 通信 | “支持 I2C 控制” | 是否支持稳定读状态、写参数、异常定位 |
| `UART` | 串口，常用于日志或模块通信 | “可以走 UART 打日志” | 是给调试用，还是业务数据接口 |
| `SPI` | 更高速的串行通信接口 | “这个外设走 SPI” | 本项目当前是否真的需要，不要混入主链路 |
| `GPIO` | 通用输入输出引脚 | “拉高这个 GPIO 就启用” | 电平逻辑、上拉下拉、默认状态要搞清楚 |
| `INT` / `IRQ` | 中断信号线，芯片主动通知异常或事件 | “有个 INT 脚” | 中断触发条件、清除机制、是否必接 |
| `EXTI` | MCU 外部中断线 | “INT 进 MCU 的 EXTI” | 是哪根脚、上升沿还是下降沿触发 |
| `Polling` | 轮询，MCU 周期性读取状态 | “先轮询也能做” | 轮询频率、延迟、会不会错过短暂异常 |
| `Interrupt` | 中断，外设主动触发 MCU 响应 | “最好配合中断做” | 快路径里能做什么，不能做什么 |
| `DMA` | Direct Memory Access，硬件搬运数据 | “串口接收建议 DMA” | 是否是效率优化，不等于业务逻辑本身 |
| `Timeout` | 超时保护 | “I2C 要加 timeout” | 没有超时很容易把系统拖死 |
| `Bus Recovery` | 总线恢复，I2C 异常后尝试自愈 | “SDA 被拉低要做 recovery” | 这属于底层健壮性，不是可选项 |
| `7-bit 地址` | I2C 标准从机地址格式 | “芯片地址是 `0x75`” | 很多人把 7-bit 和 8-bit 混着说 |
| `8-bit 地址` | 含读写位的 I2C 地址写法 | “写地址 `0xEA`，读地址 `0xEB`” | 要统一到代码里用哪种表达 |

## 7. 测量与采样术语

| 术语 | 简要解释 | 对方常见说法 | 我方真正要确认什么 |
| --- | --- | --- | --- |
| `ADC` | Analog-to-Digital Converter，模数转换器，把模拟电信号变成数字数据 | “芯片内置 ADC” | 是芯片内部自带测量能力，还是外部还要另加采样电路 |
| `模拟量` | 连续变化的物理电信号，如电压、电流、温度对应电压 | “这个量要先采模拟量” | 到底测的是电池电压、NTC、电流采样，还是别的节点 |
| `数字量` | 已经离散化、可直接计算或传输的数据 | “ADC 转完就是数字量” | 这个值能不能直接用，还是还要换算 |
| `采样` | 对某个物理量进行一次测量并取值 | “采样一下电池电压” | 采样频率、采样点和采样精度是否够用 |
| `采样点` | 实际被测的电路节点或物理位置 | “NTC 采样点放哪” | 测的是电芯、板温、芯片温度，还是输入输出口 |
| `分辨率` | ADC 可区分的细度，常见如 10-bit、12-bit、14-bit | “这颗是 14-bit ADC” | 精度是否真的足够，还是只是宣传参数 |
| `量程` | 可测范围 | “这个 ADC 量程够不够” | 测量上限下限是否覆盖真实工况 |
| `LSB` | ADC 或寄存器的最小步进单位 | “1 LSB = 1mV” | 原始数值怎么换算成真实电压、电流、温度 |
| `校准` | 用已知参考值修正测量误差 | “电量算法前最好先校准” | 是否需要工厂校准，还是默认值就够 |
| `滤波` | 对采样值做平滑处理，减少抖动和噪声 | “温度值要做滤波” | 是芯片内部已滤波，还是 MCU 侧还要再处理 |
| `噪声` | 干扰引起的测量波动 | “这个点噪声比较大” | 是电路问题、布局问题，还是采样策略问题 |
| `电量算法` | 根据电压、电流、温度等推算剩余电量 | “芯片内置电量算法” | 算法结果是否可读、是否可校准、是否足够解释给用户 |
| `Coulomb Counter` / `库仑计` | 通过积分电流估算电量的方式 | “有些芯片带库仑计” | 这颗芯片到底是简单估算还是较完整电量统计 |

这里特别容易误解的一点是：

1. `ADC` 负责“测到原始数字值”。
2. `I2C` 负责“把这些值提供给 MCU 读取”。
3. `电量算法` 负责“把原始值进一步变成电量或状态结论”。

所以对方说“芯片有 ADC”，不等于：

- MCU 不需要了
- 测量结果已经可直接展示给用户
- 电量算法一定可靠

更准确的理解是：

> SoC 可能已经具备基础测量能力，但是否足够用于产品级解释，还要看接口开放程度、换算方式、校准机制和算法质量。

## 8. 电源、协议与保护术语

| 术语 | 简要解释 | 对方常见说法 | 我方真正要确认什么 |
| --- | --- | --- | --- |
| `Power Path` | 电源路径管理，决定电能怎么流动 | “支持 power path” | 是否能边充边放、热压力怎么处理 |
| `Buck` | 降压 | “内部带 buck” | 是输入降压还是给电池充电路径降压 |
| `Boost` | 升压 | “支持 boost 输出” | 输出功率和效率边界是多少 |
| `Buck-Boost` | 升降压 | “是双向 buck-boost” | 是否可同时覆盖输入输出多工况 |
| `Source` | 供电端 | “Type-C 口做 source” | 是对外放电角色 |
| `Sink` | 受电端 | “Type-C 口做 sink” | 是对内充电角色 |
| `DRP` | Dual Role Power，双角色供电口 | “这个口支持 DRP” | 双角色切换条件和策略是谁控制 |
| `PDO` | PD 协议中的电压电流档位对象 | “广播哪些 PDO” | 是固定档位，还是可配置 |
| `PPS` | Programmable Power Supply，可编程细粒度输出 | “支持 PPS” | 状态是否可读，细粒度控制是否开放 |
| `OCP` | 过流保护 | “OCP 阈值可配” | 是硬件阈值、软件阈值，还是运行时可改 |
| `OVP` | 过压保护 | “有 OVP” | 触发后是锁死、自动恢复还是上报事件 |
| `OTP` | 过温保护 | “有 OTP/温保护” | 温度来源是芯片内温、NTC，还是两者都有 |
| `NTC` | 负温度系数热敏电阻，用来测温 | “接 NTC 做热管理” | 采样点在哪，是测电芯还是测板温 |
| `Derating` / `降额` | 因温度或条件限制主动降低功率 | “高温下需要降额” | 这是整机策略，通常需要 MCU 参与 |
| `效率` | 输入输出功率转换效率 | “这颗效率高” | 在什么电压、电流、温度条件下高 |
| `纹波` | 输出电压上的高频波动 | “纹波要看” | 更偏硬件验证，但会影响稳定性与协议 |

## 9. 软件架构与接口术语

| 术语 | 简要解释 | 对方常见说法 | 我方真正要确认什么 |
| --- | --- | --- | --- |
| `API` | 对上层暴露的调用接口 | “我们给 API” | 是口头能力清单，还是稳定文档化接口 |
| `SDK` | 软件开发包 | “可以提供 SDK” | 是源码、库文件、示例代码，还是仅头文件 |
| `HAL` | Hardware Abstraction Layer，硬件抽象层 | “先走 HAL” | 当前项目里主要指 MCU 外设抽象，不是 SoC 业务抽象 |
| `SAL` | SoC Abstraction Layer，SoC 抽象层 | “后续走 SAL 适配” | 上层不能直接绑定某厂寄存器，必须经过 SAL |
| `Vendor Adapter` | 某个原厂 SoC 的具体适配实现 | “先落一个 adapter” | 一期只做一个，但结构上要可扩展 |
| `Callback` | 回调函数，事件发生时由底层通知上层 | “支持事件 callback” | 事件粒度、上下文、时机、线程/中断限制 |
| `Event` | 事件，如握手成功、故障触发、角色切换 | “需要事件模型” | 事件是否可区分来源，是否有事件码 |
| `State Machine` | 状态机 | “这个逻辑用状态机更稳” | 是产品行为模型，不要散落在 if-else 里 |
| `Bring-up` | 上电与基础功能打通阶段 | “先做 bring-up” | 目标通常是最小可运行，不是量产完成 |
| `Driver` | 驱动 | “把驱动先写起来” | 是 HAL 驱动、SoC adapter，还是业务层封装 |
| `Porting` | 移植 | “后续好移植” | 需要哪些边界提前隔离好 |
| `Version Compatibility` | 版本兼容性 | “新版本兼容老接口” | 有没有正式承诺，还是经验判断 |

## 10. 调试、验证与联调术语

| 术语 | 简要解释 | 对方常见说法 | 我方真正要确认什么 |
| --- | --- | --- | --- |
| `联调` | 两边一起把接口和行为跑通 | “安排联调” | 联调目标、输入资料、责任边界、成功标准 |
| `Log` | 日志 | “先看 log” | 日志是否足以区分初始化、读值、异常、恢复过程 |
| `Waveform` | 波形 | “抓一下波形” | 常指示波器或逻辑分析仪看到的真实时序 |
| `Logic Analyzer` | 逻辑分析仪 | “抓 I2C 波形” | 对数字总线排查非常重要 |
| `Oscilloscope` | 示波器 | “看一下电压波形” | 更适合看模拟量、电源质量、纹波、瞬态 |
| `Repro Step` | 复现步骤 | “请给复现步骤” | 没有复现步骤，对方很难有效支持 |
| `Golden Sample` | 金样/基准样机 | “按金样对比” | 用来判断是设计问题还是个体样机问题 |
| `Smoke Test` | 冒烟测试，快速确认最基本功能没坏 | “先做 smoke test” | 不是深度验证，只是第一道门 |
| `Stress Test` | 压力测试 | “高负载跑一下 stress” | 看稳定性、热、恢复能力 |
| `Corner Case` | 边界场景 | “边界场景要补” | 最容易暴露异常，不应只测正常场景 |
| `Issue Triage` | 问题分流归类 | “先 triage 一下” | 先分清硬件、驱动、协议、架构、资料问题 |
| `Workaround` | 临时绕过方案 | “先给个 workaround” | 不是根治，要记录适用范围和风险 |

## 11. 合作沟通中最容易误解的话

下面这些话，表面上听起来没问题，但都容易误判。

### 11.1 “支持 I2C”

可能只表示：

1. 芯片可以通过 I2C 做少量配置。
2. 有一些状态能读。
3. 并不保证事件、异常、版本差异、寄存器说明完整。

正确追问：

1. 能读哪些状态？
2. 能写哪些参数？
3. 故障和事件能否区分？
4. 是否提供正式寄存器表？
5. 芯片版本变化后接口是否稳定？

### 11.2 “可以提供 SDK”

可能只表示：

1. 给你一个 demo。
2. 给你几个示例函数。
3. 不一定适合你们当前架构，也不一定长期维护。

正确追问：

1. SDK 的组成是什么，源码、库还是头文件？
2. 版本如何管理？
3. 有没有最小可用例程？
4. 不用 SDK，是否能直接按寄存器集成？

### 11.3 “这颗芯片都集成了”

可能只表示：

1. 底层能力集成了很多。
2. 但并不表示产品层面的日志、可视化、策略、诊断都不需要 MCU。

正确追问：

1. 集成的是底层能力，还是整机能力？
2. MCU 还能读到什么、控制什么、解释什么？
3. 芯片是否支持上层做策略仲裁和异常说明？

### 11.4 “可以跑 Demo”

可能只表示：

1. 参考板在某些典型条件下工作正常。
2. 但不代表你们的板子、热条件、结构、外围件、软件架构都没问题。

正确追问：

1. Demo 的硬件条件是什么？
2. 我们改外围件后风险点在哪？
3. 是否有已知限制或典型失败模式？

## 13. 建议的实际用法

后续和英集芯或其他原厂开会时，建议把术语理解落成三个动作：

1. 会前：先用本文把陌生术语扫一遍。
2. 会中：听到 `寄存器表`、`事件`、`INT`、`SDK`、`参考设计`、`版本` 这些词，立刻追问边界。
3. 会后：把对方的口头说法转成书面清单，例如“可读状态清单”“可写参数清单”“已知限制清单”“联调验证项清单”。

这样做的价值不是显得专业，而是减少误解，避免把“听起来支持”误判成“工程上真的可落地”。
