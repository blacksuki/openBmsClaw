# 第 1 章：起步与基建 —— 让 AI 搭建项目工作台

> **导语**：在理清了产品需求后，下一步是搭建代码工程环境。对于像 `openBattery` 这种需要适配多种芯片、且具备高级智能策略的平台来说，传统的一锅炖式单体结构就是灾难。本章将讲解如何利用 AI，一步到位地生成企业级的分层防耦合架构。

## 1.1 为什么 AI 需要标准化的目录？

很多初学者习惯把所有代码堆在 `main.c` 里。如果让 AI 在这样一个几十个函数揉成一团的文件里去“修改一个充电逻辑”，它很容易顾此失彼，甚至改坏原本正常的 LED 显示。

**让 AI 写代码的前提，是给它一个职责清晰的物理边界（文件与目录层级）。** 只有目录标准化了，你才能准确地对 AI 说：“请只修改 `drivers/` 目录下的某文件，不要碰 `app/` 里的逻辑。”

## 1.2 openBattery 的“双核脑体协同”架构

在 `openBattery` 项目中，我们的选型是：**STM32 (MCU) + 高集成度 SoC (如英集芯、智融等)**。

- **SoC 负责体力活**：高频电压转换、快充协议握手 (PD/QC)、底层电芯物理保护。
- **MCU 负责脑力活**：实时数据获取、剩余时长预测、动态温度调度、多设备功率仲裁及 OLED 可视化。

为了支撑这个双核架构，我们废弃了旧版分散的硬件代码思维，全面走向分层架构。

## 1.3 核心目录的职责划分

我们可以直接使用一行提示词让 AI（或自动化脚本）生成如下结构，并为每个目录写入说明性的空文件 `README.md`。

```text
openBmsClaw/
├── app/        (可视化与交互)
├── services/   (智能调度策略)
├── drivers/    (SoC SAL 抽象层)
├── hal/        (底层基建)
├── board/      (板级映射)
└── config/     (系统配置宏)
```

### 1) `app/` (应用层)
负责数字化呈现与最直观的体验。例如：OLED 渲染、LED 跑马灯、剩余使用时长模型、按键业务状态机。
> **铁律**：绝不能在 `app/` 层面直接调用 I2C 写寄存器控制 SoC！

### 2) `services/` (服务层)
大脑的核心决策区。例如：
- `srv_power_alloc`：多设备接入时的功率仲裁。
- `srv_thermal`：结合 SoC 内温和板级 NTC，动态干预充电系统（如过温降额）。

### 3) `drivers/` (SoC 抽象适配层 - SAL)
负责抹平各家（英集芯、智融等）芯片差异的“接头人”。对外统一暴露标准 API：`soc_get_voltage()`、`soc_set_ocp()` 等。

### 4) `hal/` 与 `board/` (底层基建)
- `hal/`：纯粹的 MCU 自身外设读写透传，如 `hal_i2c`、`hal_uart`。
- `board/`：定义当前的管脚绑定与外设初始化入口。

## 1.4 openBmsClaw 真实目录与代码地图
在我们的实际仓库中，`openBmsClaw` 工程已经建立好了这种企业级的分层目录。建议你立即打开 [openBmsClaw/](file:///Users/huoward/Project/11.openBattery/openBmsClaw/openBmsClaw) 目录进行对照：
- `openBmsClaw/hal/i2c/hal_i2c.c`：底层纯净的非阻塞 I2C 驱动。

## 1.5 实战 Bring-up：点亮板子并输出第一行 Log

理论千遍，不如手敲一遍。在第一周，我们的目标是使用**野火小智 STM32F103C8T6 核心板（双 USB 款）**跑通最小系统，并通过板载 UART 打印出调试日志。

### 步骤 1：物理接线 (硬件连接)
小智核心板自带 CH340C 串口芯片和双 USB 接口，以及 SWD 调试针脚：
1. **连接调试器**：将你的 ST-Link 或 DAPLink 调试器的 `SWDIO`、`SWCLK`、`GND`、`3.3V` 分别连接到核心板的 4-Pin SWD 调试接口上。
2. **连接电脑串口**：使用一根 Type-C 数据线，连接电脑与核心板上的 **“USB转串口”接口**（注意：是 CH340 芯片连接的那个 USB 口，而不是直连 MCU 的 USB 口）。这会同时给板子供电，并在电脑上虚拟出一个 COM 端口（如 macOS 上的 `/dev/tty.usbserial-*`，或 Windows 上的 `COM3`）。

### 步骤 2：编译项目 (VS Code 官方插件与沙箱工具链配置)
项目使用 CMake 与 Ninja 进行极其高效的构建。为了在 macOS/Windows 上一键打通工具链，请按以下实战步骤操作：

1. **导入官方插件**：因 ST 官方 `STM32CubeIDE for VS Code` 插件未发布在开源的 Open VSX 市场中，您可以直接获取扩展的离线 `.vsix` 文件（项目根目录下已为您备好 `stm32-vscode-extension.vsix`）。在插件面板右上角选择 **"Install from VSIX..."** 并导入即可。
2. **激活沙箱交叉编译器**：当您首次使用 VS Code 插件打开工程并触发 **"CMake: Configure"** 时，官方扩展会自动在后台为您拉取 `gnu-tools-for-stm32` 包（内含 `arm-none-eabi-gcc` 交叉编译器），自动将路径绑定至 `${env:CUBE_BUNDLE_PATH}`，您**无需全局手动配置 PATH**！
3. **极速构建**：点击底部状态栏的 **Build**（或 `CMake: Build`）按钮，系统将自动使用 `Ninja` 引擎输出 `openBmsClaw.elf`、`openBmsClaw.bin` 与 `openBmsClaw.hex`。

### 步骤 3：烧录与日志观测 (GUI 烧录与 macOS 一键命令行烧录)

- **GUI 烧录方式**：
  1. 打开 **STM32CubeProgrammer** (macOS 默认安装于 `/Applications/STMicroelectronics/STM32Cube/STM32CubeProgrammer/STM32CubeProgrammer.app`)。
  2. 选择 `ST-LINK` 模式并点击 `Connect`。
  3. 选中编译出来的 `openBmsClaw.bin` (写入地址 `0x08000000`)，点击 `Start Programming` 烧录。
- **macOS 高级一键命令行烧录方式**：
  您也可以直接利用内置的 CLI 工具，在终端里直接一键极速烧录（建议在 `.zshrc` 中添加 `alias stlink` 别名）：
  ```bash
  /Applications/STMicroelectronics/STM32Cube/STM32CubeProgrammer/STM32CubeProgrammer.app/Contents/MacOs/bin/STM32_Programmer_CLI -c port=SWD -w build/Debug/openBmsClaw.bin 0x08000000 -v -rst
  ```

- **日志观测**：
  1. 打开串口助手（如 VS Code 里的 Serial Monitor 插件，波特率 `115200`，配置 `115200, 8N1`）。
  2. 按下核心板上的 **RST（复位）按键**，您将在串口控制台上清晰地看到系统初始化的第一行标志性输出：
     ```text
     boot ok
     PowerService: Init OK, emergency callback registered.
     ```
看到这行 Log，恭喜你，你的 openBattery 大脑已经正式苏醒，开发工作台基建彻底打通！

## 1.6 实操指南：一句话搭建框架 (当需要新增子模块时)

如果你后续需要在架构中新增一个类似“BLE蓝牙服务”的模块，可以用如下方式指示 AI 编写接口骨架：

> **Prompt 示例**：
> “你现在是一位资深嵌入式架构师。我正在重构 openBattery 项目，采用 MCU(大脑) + SoC(体力) 的双核架构。
> 请帮我在 `services/` 目录下规划一个 `srv_ble` 模块的头文件与源文件框架。
> 要求：模板中需包含标准的文件头注释，符合 MISRA-C 风格，并且声明 `srv_ble_init()` 和 `srv_ble_process()` 接口。该层禁止直接调用底层串口寄存器，必须说明它应当通过 `hal_uart` 传输数据。”

通过这一步，你不仅规范了自己的开发思路，更重要的是，**为后续 AI 生成代码划定了不可逾越的安全红线**。
