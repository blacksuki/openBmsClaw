# 第 6 章：降维接盘 —— 让 AI 理解、维护和修改已有项目

> **导语**：在真实的工程世界中，你很少有机会总是从零开始写代码。更多的时候，你需要接手前人留下的一摊“祖传代码”，甚至要把旧平台的东西移植到新平台上。本章将展示如何在 `openBattery` 项目的演进过程中，利用 AI 完成业务剥离与无缝移植。

## 6.1 审查与重构：治理“越权行为”

在项目初期或者由新手接手时，很容易写出破坏“脑体协同、分层防腐”架构的代码。

**典型错误场景**：
在可视化应用层 `app_display.c` 中，为了快速在屏幕上显示当前电池温度，直接写了一段：
`I2C_ReadRegister(IP5356_ADDR, 0x14, &temp_raw);`

这打破了架构红线！一旦芯片从英集芯换成了智融，应用层的 UI 代码就得跟着大改。

**AI 纠错工作流**：
你可以定期将 `app/` 层的源码丢给 AI 进行“架构审查”：
> “你是本项目的架构师。我们的规范是 `app/` 层严禁直接调用底层的 I2C 函数操作硬件，必须通过 `drivers/` 层的 SAL API 获取数据。
> 请帮我审查这段 `app_display.c`，找出所有的越权调用，并将它们重构为调用 `soc_get_temperature()` 等标准接口。同时请给出修改前后的对比差异。”

AI 能瞬间发现这些架构坏味道（Bad Smell），帮你维护系统长期的纯洁性。

## 6.1.1 为什么 SoC 公共 API 不能泄露 HAL 细节？

这次 `openBmsClaw` 的 review 里，一个很典型的问题是：

- `soc_sal.h` 是面向上层 `app/` 和 `services/` 的公共头文件。
- 但它却直接 include 了 `hal_i2c.h`、`hal_exti.h`。

这就是一种很典型的**接口泄露**。

### 泄露会带来什么问题？

1. 上层会不知不觉看到底层 HAL 类型和实现细节。
2. F103 -> F030 -> 其他 MCU 迁移时，公共 API 会被底层外设实现牵着走。
3. 原本应该稳定的 SoC 抽象层，会退化成“带着 HAL 尾巴的驱动包装”。

### 更合理的拆法是什么？

推荐至少分成四层：

1. `soc_types.h`
   - 状态码
   - 端口类型
   - 事件类型

2. `soc_api.h`
   - 上层只允许 include 这一层
   - 只暴露 `soc_get_voltage()`、`soc_set_ocp()` 这类平台能力

3. `soc_transport.*`
   - 处理 I2C/UART 传输
   - 内部才依赖 `hal_i2c`

4. `vendor/*.c`
   - 英集芯、智融等具体 SoC 的寄存器和转换逻辑

一句话理解：

> 公共 API 应该描述“平台能力”，而不是暴露“底层外设怎么实现”。

### 6.1.1.1 小白版理解：什么是 `vtable` / 函数指针表？

在 C 语言里，没有 C++ 那种现成的“接口”或“虚函数”机制。
但我们仍然可以用一种很轻的办法做出类似效果，这就是 **函数指针表**。

你可以把它想成一张“能力表”：

```c
typedef struct {
    int (*init)(void);
    int (*get_voltage)(uint16_t *mv);
    int (*set_ocp)(uint16_t ma);
} soc_driver_ops_t;
```

这张表里放的不是普通数据，而是一组函数地址。
意思是：

- 这里有一个初始化动作
- 这里有一个读电压动作
- 这里有一个设置过流保护动作

### 它解决什么问题？

上层代码只认这张表，不关心底下到底是谁在做事。

也就是说，上层不用知道当前接的是：

- demo SoC
- 英集芯 SoC
- 智融 SoC

上层只知道：

> 我现在要“读电压”，那就调用 `ops->get_voltage(...)`。

这样做的好处是：

> 换 SoC 时，尽量不改 `app/` 和 `services/`。

### 为什么这叫“接口不要源码”？

因为上层依赖的是统一动作接口，例如：

- `get_voltage()`
- `set_ocp()`

而不是去碰：

- 某家芯片的寄存器地址
- 某家芯片的私有读写函数
- 某家芯片的具体实现细节

一句话理解：

> 上层只看“能做什么”，不看“底下具体是谁来做”。

### 为什么它会多占一点 Flash？

函数指针表的代价是：代码通常会比“全部写死在一个文件里”更大一点。

原因主要有两个：

1. 通过函数指针调用时，编译器通常更难做内联优化。
2. 每个操作往往会变成独立函数，代码片段会增多。

所以在 64KB Flash 这种受限 MCU 上，可能会看到类似：

```text
+1160B FLASH
```

这样的体积代价。

### 为什么这笔代价仍然值得？

因为它换来的不是“功能变多”，而是“结构变稳”：

- 上层不容易被具体 SoC 绑死。
- 换芯片时改动更集中。
- 后续维护时更容易判断问题属于哪一层。

对 `openBattery` 当前路线来说，这笔账通常是划算的，因为我们的目标本来就不是把 demo 永远写死，而是后续能从 demo 平滑走向真实 SoC 适配。

### 一句最短记忆法

> `vtable` / 函数指针表 = 上层只认“功能菜单”，不认“具体厂商实现”。

## 6.1.2 demo 骨架和正式平台代码的边界

startup 阶段出现 demo 代码很正常，但真正危险的不是“有 demo”，而是“demo 永远不退出”。

例如下面这些代码，在 bring-up 阶段是可以接受的：

- demo register 地址
- 软件触发 `EXTI_SWIER` 自测
- 用 LED 熄灭模拟紧急切断
- 直接串口打印 bring-up 日志

但它们都必须有退出条件。否则后续就会出现两个问题：

1. 正式路径和自测路径混在一起。
2. 新成员误以为当前系统已经具备真实产品能力。

这也是为什么 `startup 技术债` 需要单独记录在 `2_Action/` 中，而不是只靠口头记忆。

## 6.1.3 什么是 startup 技术债？

startup 技术债不是“代码写得差”，而是：

> 为了先跑通最小链路，暂时接受的架构偏差和验证性写法。

例如：

- `app/` 直接触发底层自测 hook
- SAL 里混着 demo register
- service 直接操控 board LED
- 配置宏含义重叠

这些东西短期不一定错，但必须满足两个条件：

1. 明确为什么现在暂时接受。
2. 明确未来在哪个阶段前必须清理。

如果没有退出条件，它就不再是“startup 技术债”，而是会固化成长期设计缺陷。

## 6.2 深度剖析：避开 STM32F103 的“64KB 资源陷阱”

在学习与验证阶段，我们使用 `STM32F103C8T6` 作为主控，它的 **Flash 空间只有 64KB**。
很多新手在开发过程中，兴致勃勃地导入了：
- 几千个汉字的中文字库（用于 OLED 展示高级中文字体）。
- 复杂的双精度浮点数预测算法模型（`double` 类型计算剩余电量与时长）。
- 完整的串口协议通信解析包。

结果在编译时，编译器无情地抛出报错：
`arm-none-eabi-ld: region 'FLASH' overflowed by 12540 bytes`

这就是著名的**嵌入式资源陷阱**。为了能在 64KB 的有限身躯里跑起 openBattery 的核心脑力逻辑，我们必须采取两大自救手段：

### 1. 编译优化自救（在 `CMakeLists.txt` 中）：
确保你的编译配置开启了尺寸优化，并且剥离了未使用的死代码（Dead Code）：
- **使用 `-Os` 优化标志**：告诉 GCC 编译器一切以“减小生成体积”为最高优先级。
- **死代码裁剪**：
  - 在编译器 CFLAGS 中加入：`-ffunction-sections -fdata-sections`（将每个函数和数据块放入独立的段中）。
  - 在链接器 LDFLAGS 中加入：`-Wl,--gc-sections`（在链接时，自动将没有被调用的函数段彻底从 Flash 中删除）。

### 2. 宏裁剪与自适应字库（在 `config/sys_config.h` 与 `config/app_config.h` 中）：
当前仓库不是使用单独的 `config.h`，而是通过 `sys_config.h`、`app_config.h`、`feature_config.h` 组合实现功能裁剪。下面的示例改写为与当前工程一致的风格：

```c
/**
 * @file sys_config.h
 * @brief openBattery 系统功能裁剪宏配置文件
 */

#ifndef OPEN_BMS_CLAW_CONFIG_SYS_CONFIG_H
#define OPEN_BMS_CLAW_CONFIG_SYS_CONFIG_H

/* F103 学习板阶段，默认关闭高资源占用功能 */
#define CONFIG_ENABLE_OLED_DISPLAY     0
#define CONFIG_ENABLE_LARGE_FONT       0
#define CONFIG_ENABLE_COMPLEX_UI_ANIM  0
#define CONFIG_ENABLE_BLE_COMM         0

#include "app_config.h"
#include "feature_config.h"

#endif /* OPEN_BMS_CLAW_CONFIG_SYS_CONFIG_H */
```

### 3. 代码中根据宏自动切换渲染逻辑
```c
#include "config/sys_config.h"

void display_render_status(void)
{
#if (CONFIG_ENABLE_OLED_DISPLAY == 0)
    // 极简渲染：只显示 ASCII "V: 4.2V"
    board_uart_write_string("V: 4.2V\r\n");
#else
    // 豪华渲染：显示中文 “当前电压: 4200 毫伏”
    oled_show_chinese(0, 0, "当前电压: 4200 毫伏");
#endif
}
```

### 4. 进一步收敛：`PROFILE_*`、`FEATURE_*`、`APP_ENABLE_*` 三层配置模型

仅靠零散的几个 `#define` 很容易越写越乱。更合理的做法，是把配置分成三层：

#### `PROFILE_*`
用于选择当前构建目标，例如：

- `PROFILE_F103_BRINGUP`
- `PROFILE_F030_MIN_PRODUCT`
- `PROFILE_G0_EXTENDED`

它回答的是：

> 当前这次构建，面向哪一类平台和资源边界？

#### `FEATURE_*`
用于表达产品能力是否存在，例如：

- `FEATURE_ENABLE_POWER`
- `FEATURE_ENABLE_PROTOCOL`
- `FEATURE_ENABLE_BLE`

它回答的是：

> 这个产品版本，逻辑上有没有这项能力？

#### `APP_ENABLE_*`
用于控制当前 app 是否调度某个 service 或 sample，例如：

- `APP_ENABLE_POWER_SERVICE`
- `APP_ENABLE_TEMPERATURE_ALARM_SAMPLE`

它回答的是：

> 当前这个构建版本，程序入口是否实际运行这条路径？

### 为什么一定要分三层？

因为如果三层不分，很容易出现“双真相”：

- `FEATURE_ENABLE_POWER = 0`
- `APP_ENABLE_POWER_SERVICE = 1`

这会让人无法判断：

- 这是一个正式启用的能力？
- 还是 bring-up 测试路径？
- 还是忘记同步配置了？

一句话记忆：

> `PROFILE` 决定平台边界，`FEATURE` 决定产品能力，`APP_ENABLE` 决定入口调度。

## 6.3 跨平台移植：从验证板向 F030 量产候补平台迁移

当项目进入量产过渡阶段，我们需要先将代码从 F103 学习平台迁移到 **F030 量产候补平台**。根据当前主线架构文档，`F030` 是当前优先评估的 64KB Flash 档位量产候补；如果后续样片验证发现资源、外设或可靠性不满足要求，再上移到 `STM32G0`、`GD32E230` 或更宽裕的 MCU。

### 迁移时必须验证的物理差异：
1. **时钟配置（RCC）**：F103 与 F030 的时钟树、初始化流程和外设寄存器布局不同。迁移时应重新生成或手工核对时钟初始化，不能直接复制 F103 的启动配置。
2. **I2C 与 ADC 外设细节**：F030 的外设时序和寄存器模型与 F103 不同，必须重测 I2C 连续读写、总线恢复、ADC 滤波与 EXTI 告警触发。
3. **Flash/SRAM 预算**：F030 仍属于受限资源平台，迁移目标不是“功能放开来堆”，而是验证 `hal + drivers + services + app_min + log_min` 是否能在量产候补平台上稳定运行，并保留足够余量。
4. **上移预案**：如果在 F030 上出现 Flash 预算、外设能力或联调可靠性问题，就应按主线文档及时上移到 `STM32G0` 或其他更宽裕候选，而不是强行在 F030 上透支架构。

### 为什么说分层架构是移植救星？
因为迁移影响应主要限制在 `board/`、`hal/`、`config/` 和启动/链接配置中。上层的 `services/`（智能功率调度）和 `app/`（显示逻辑）应通过裁剪宏适配资源差异，而不是直接绑定具体 MCU。
