# 第 6 章：降维接盘 —— 让 AI 理解、维护和修改已有项目

> **导语**：在真实的工程世界中，你很少有机会总是从零开始写代码。更多的时候，你需要接手前人留下的一摊“祖传代码”，甚至要把旧平台的东西移植到新平台上。本章将展示如何在 `openBattery` 项目的演进过程中，利用 AI 完成业务剥离与无缝移植。

## 6.1 审查与重构：治理“越权行为”

在项目初期或者由新手接手时，很容易写出破坏“脑体协同、分层防腐”架构的代码。

**典型错误场景**：
在可视化应用层 `app_display.c` 中，为了快速在屏幕上显示当前电池温度，直接写了一段：
`I2C_ReadRegister(IP5356_ADDR, 0x14, &temp_raw);`

这打破了架构红线！一旦芯片从英集芯换成了智融，应用层的 UI 代码就得跟着大改。

**AI 纠错工作流**：
你可以定期将 `app/` 层的源码丢给 AI 进行“架构审查”：
> “你是本项目的架构师。我们的规范是 `app/` 层严禁直接调用底层的 I2C 函数操作硬件，必须通过 `drivers/` 层的 SAL API 获取数据。
> 请帮我审查这段 `app_display.c`，找出所有的越权调用，并将它们重构为调用 `soc_get_temperature()` 等标准接口。同时请给出修改前后的对比差异。”

AI 能瞬间发现这些架构坏味道（Bad Smell），帮你维护系统长期的纯洁性。

## 6.2 深度剖析：避开 STM32F103 的“64KB 资源陷阱”

在学习与验证阶段，我们使用 `STM32F103C8T6` 作为主控，它的 **Flash 空间只有 64KB**。
很多新手在开发过程中，兴致勃勃地导入了：
- 几千个汉字的中文字库（用于 OLED 展示高级中文字体）。
- 复杂的双精度浮点数预测算法模型（`double` 类型计算剩余电量与时长）。
- 完整的串口协议通信解析包。

结果在编译时，编译器无情地抛出报错：
`arm-none-eabi-ld: region 'FLASH' overflowed by 12540 bytes`

这就是著名的**嵌入式资源陷阱**。为了能在 64KB 的有限身躯里跑起 openBattery 的核心脑力逻辑，我们必须采取两大自救手段：

### 1. 编译优化自救（在 `CMakeLists.txt` 中）：
确保你的编译配置开启了尺寸优化，并且剥离了未使用的死代码（Dead Code）：
- **使用 `-Os` 优化标志**：告诉 GCC 编译器一切以“减小生成体积”为最高优先级。
- **死代码裁剪**：
  - 在编译器 CFLAGS 中加入：`-ffunction-sections -fdata-sections`（将每个函数和数据块放入独立的段中）。
  - 在链接器 LDFLAGS 中加入：`-Wl,--gc-sections`（在链接时，自动将没有被调用的函数段彻底从 Flash 中删除）。

### 2. 宏裁剪与自适应字库（在 `config/config.h` 中）：
我们在 `config.h` 中设计了极其灵活的裁剪宏开关，允许在 F103 阶段关闭不必要的外围“视觉特效”：

```c
/**
 * @file config.h
 * @brief openBattery 系统功能裁剪宏配置文件
 */

#ifndef CONFIG_H
#define CONFIG_H

/* 板级芯片平台定义 (1: F103 验证板, 2: G071 量产平台) */
#define CONFIG_MCU_PLATFORM         1 

#if (CONFIG_MCU_PLATFORM == 1)
    /* F103 验证板资源有限，启动严苛裁剪模式 */
    #define CONFIG_USE_MINI_UI      1   /**< 启用极简 UI 渲染，只包含基本 ASCII，不编译汉字字库 */
    #define CONFIG_USE_FLOAT_MATH   0   /**< 关闭浮点数计算，电量与时间预测全部使用定点整型数模拟 */
    #define CONFIG_ENABLE_BLE_STACK 0   /**< 关闭蓝牙通信协议栈，节省 15KB Flash */
#else
    /* G071 量产平台 (128KB Flash)，开启全功能 */
    #define CONFIG_USE_MINI_UI      0   
    #define CONFIG_USE_FLOAT_MATH   1   
    #define CONFIG_ENABLE_BLE_STACK 1   
#endif

#endif /* CONFIG_H */
```

### 3. 代码中根据宏自动切换渲染逻辑
```c
#include "config.h"

void display_render_status(void)
{
#if (CONFIG_USE_MINI_UI == 1)
    // 极简渲染：只显示 ASCII "V: 4.2V"
    oled_show_string(0, 0, "V: 4.2V");
#else
    // 豪华渲染：显示中文 “当前电压: 4200 毫伏”
    oled_show_chinese(0, 0, "当前电压: 4200 毫伏");
#endif
}
```

## 6.3 跨平台无缝移植：从验证板向量产芯片迁移

当项目进入 Phase 4（量产过渡阶段），我们需要将代码从 F103 平台无缝移植到 **STM32G071 (128KB Flash)**。由于我们在前几章就贯彻了“脑体分离、严格分层”的原则，这种移植将变得极其轻松！

### 新旧 HAL 库的两大物理差异：
1. **时钟配置（RCC）**：F103 与 G071 的时钟树架构完全不同，G071 使用全新的 HSI 高速内部时钟源及 LPUART（低功耗串口）。在移植时，直接在 STM32CubeMX 中生成 G0 的 `SystemClock_Config`，然后替换 `board/` 目录下的底层初始化即可。
2. **I2C 外设时序寄存器（TimingR）**：
   - F103 只需要填入简单的速度（如 `100000` 即 100kHz）。
   - G071 使用了复杂的 `I2C_TIMINGR` 32位十六进制寄存器（例如 `0x00B03FDB`），这个值必须由 STM32CubeMX 输入时钟频率后自动生成。严禁手动盲填！

### 为什么说分层架构是移植救星？
因为你只需要修改 `board/`（管脚重新绑定）与 `hal/`（用 G0 的 I2C 初始化接口替换 F103）。而上层的 `services/`（智能功率调度）和 `app/`（显示逻辑），**连一个字符都不需要改**！这就是平台化架构的终极魅力，它能帮你省下几个月的重复开发时间！
