# 第 6 章：降维接盘 —— 让 AI 理解、维护和修改已有项目

> **导语**：在真实的工程世界中，你很少有机会总是从零开始写代码。更多的时候，你需要接手前人留下的一摊“祖传代码”，甚至要把旧平台的东西移植到新平台上。本章将展示如何在 `openBattery` 项目的演进过程中，利用 AI 完成业务剥离与无缝移植。

## 6.1 审查与重构：治理“越权行为”

在项目初期或者由新手接手时，很容易写出破坏“脑体协同、分层防腐”架构的代码。

**典型错误场景**：
在可视化应用层 `app_display.c` 中，为了快速在屏幕上显示当前电池温度，直接写了一段：
`I2C_ReadRegister(IP5356_ADDR, 0x14, &temp_raw);`

这打破了架构红线！一旦芯片从英集芯换成了智融，应用层的 UI 代码就得跟着大改。

**AI 纠错工作流**：
你可以定期将 `app/` 层的源码丢给 AI 进行“架构审查”：
> “你是本项目的架构师。我们的规范是 `app/` 层严禁直接调用底层的 I2C 函数操作硬件，必须通过 `drivers/` 层的 SAL API 获取数据。
> 请帮我审查这段 `app_display.c`，找出所有的越权调用，并将它们重构为调用 `soc_get_temperature()` 等标准接口。同时请给出修改前后的对比差异。”

AI 能瞬间发现这些架构坏味道（Bad Smell），帮你维护系统长期的纯洁性。

## 6.2 深度剖析：避开 STM32F103 的“64KB 资源陷阱”

在学习与验证阶段，我们使用 `STM32F103C8T6` 作为主控，它的 **Flash 空间只有 64KB**。
很多新手在开发过程中，兴致勃勃地导入了：
- 几千个汉字的中文字库（用于 OLED 展示高级中文字体）。
- 复杂的双精度浮点数预测算法模型（`double` 类型计算剩余电量与时长）。
- 完整的串口协议通信解析包。

结果在编译时，编译器无情地抛出报错：
`arm-none-eabi-ld: region 'FLASH' overflowed by 12540 bytes`

这就是著名的**嵌入式资源陷阱**。为了能在 64KB 的有限身躯里跑起 openBattery 的核心脑力逻辑，我们必须采取两大自救手段：

### 1. 编译优化自救（在 `CMakeLists.txt` 中）：
确保你的编译配置开启了尺寸优化，并且剥离了未使用的死代码（Dead Code）：
- **使用 `-Os` 优化标志**：告诉 GCC 编译器一切以“减小生成体积”为最高优先级。
- **死代码裁剪**：
  - 在编译器 CFLAGS 中加入：`-ffunction-sections -fdata-sections`（将每个函数和数据块放入独立的段中）。
  - 在链接器 LDFLAGS 中加入：`-Wl,--gc-sections`（在链接时，自动将没有被调用的函数段彻底从 Flash 中删除）。

### 2. 宏裁剪与自适应字库（在 `config/sys_config.h` 与 `config/app_config.h` 中）：
当前仓库不是使用单独的 `config.h`，而是通过 `sys_config.h`、`app_config.h`、`feature_config.h` 组合实现功能裁剪。下面的示例改写为与当前工程一致的风格：

```c
/**
 * @file sys_config.h
 * @brief openBattery 系统功能裁剪宏配置文件
 */

#ifndef OPEN_BMS_CLAW_CONFIG_SYS_CONFIG_H
#define OPEN_BMS_CLAW_CONFIG_SYS_CONFIG_H

/* F103 学习板阶段，默认关闭高资源占用功能 */
#define CONFIG_ENABLE_OLED_DISPLAY     0
#define CONFIG_ENABLE_LARGE_FONT       0
#define CONFIG_ENABLE_COMPLEX_UI_ANIM  0
#define CONFIG_ENABLE_BLE_COMM         0

#include "app_config.h"
#include "feature_config.h"

#endif /* OPEN_BMS_CLAW_CONFIG_SYS_CONFIG_H */
```

### 3. 代码中根据宏自动切换渲染逻辑
```c
#include "config/sys_config.h"

void display_render_status(void)
{
#if (CONFIG_ENABLE_OLED_DISPLAY == 0)
    // 极简渲染：只显示 ASCII "V: 4.2V"
    board_uart_write_string("V: 4.2V\r\n");
#else
    // 豪华渲染：显示中文 “当前电压: 4200 毫伏”
    oled_show_chinese(0, 0, "当前电压: 4200 毫伏");
#endif
}
```

## 6.3 跨平台移植：从验证板向 F030 量产候补平台迁移

当项目进入量产过渡阶段，我们需要先将代码从 F103 学习平台迁移到 **F030 量产候补平台**。根据当前主线架构文档，`F030` 是当前优先评估的 64KB Flash 档位量产候补；如果后续样片验证发现资源、外设或可靠性不满足要求，再上移到 `STM32G0`、`GD32E230` 或更宽裕的 MCU。

### 迁移时必须验证的物理差异：
1. **时钟配置（RCC）**：F103 与 F030 的时钟树、初始化流程和外设寄存器布局不同。迁移时应重新生成或手工核对时钟初始化，不能直接复制 F103 的启动配置。
2. **I2C 与 ADC 外设细节**：F030 的外设时序和寄存器模型与 F103 不同，必须重测 I2C 连续读写、总线恢复、ADC 滤波与 EXTI 告警触发。
3. **Flash/SRAM 预算**：F030 仍属于受限资源平台，迁移目标不是“功能放开来堆”，而是验证 `hal + drivers + services + app_min + log_min` 是否能在量产候补平台上稳定运行，并保留足够余量。
4. **上移预案**：如果在 F030 上出现 Flash 预算、外设能力或联调可靠性问题，就应按主线文档及时上移到 `STM32G0` 或其他更宽裕候选，而不是强行在 F030 上透支架构。

### 为什么说分层架构是移植救星？
因为迁移影响应主要限制在 `board/`、`hal/`、`config/` 和启动/链接配置中。上层的 `services/`（智能功率调度）和 `app/`（显示逻辑）应通过裁剪宏适配资源差异，而不是直接绑定具体 MCU。
