# 第 3 章：软硬协同 —— 让 AI 读懂原理图和硬件接口

> **导语**：软件工程师常常视原理图为天书，这导致硬件接线错误、上拉漏配或电平极性搞反，成为 bring-up 阶段最大的时间黑洞。在 `openBattery` 这种需要双核通过外部走线协作的项目中，用 AI 快速提取原理图接口，是避免“盲调”的关键。

## 3.1 打破壁垒：让 AI 代替肉眼找引脚

在 `openBattery` 现阶段的 F103 起步验证板中，我们需要让 MCU 控制多个外设（OLED、按键、LED、以及最重要的外部 SoC）。

我们可以将原理图的 PDF（或导出为高清晰度图片），配合简单的引导提示词发给 AI：
> “这是一份移动电源控制板的原理图。请帮我以软件工程师的视角，提取出所有 STM32 MCU 需要配置的 GPIO 引脚。
> 要求按模块分类（如 I2C 通信、按键、LED、中断），表格包含：MCU 引脚名、外设功能名、默认电平、是否需要内部上下拉。”

有了这个由 AI 提取的接口速查字典，我们在配置 `board/` 层的引脚映射时，就有了坚实的依据，再也不用同时开着 PDF 和代码编辑器来回切换。

## 3.2 科普：硬件接口背后的“隐秘规则”

在开始飞线和写代码前，软件工程师必须彻底弄懂以下三个基础硬件概念：

1. **GPIO (通用输入输出)**：MCU 最基础的“触角”。可以配置为“输出模式”（向外输出 3.3V 高电平或 0V 低电平，控制 LED 或开关管），也可以配置为“输入模式”（读取外部是 3.3V 还是 0V，判断按键是否按下）。
2. **EXTI (外部中断)**：如果我们要检测一个按键或告警引脚，用 `while` 循环去不断读取（轮询）是非常低效的，且响应会有几十毫秒的延迟。**外部中断**允许硬件在引脚电平发生跳变（比如从 3.3V 跌落到 0V 的“下降沿”）时，瞬间“叫醒”MCU CPU。CPU 暂停当前手头工作，直接跳转执行一段紧急代码，响应时间在微秒级！
3. **上拉与下拉电阻 (Pull-Up/Pull-Down)**：当一个 GPIO 引脚没有连接任何东西，或者连接的开关处于断开状态时，该引脚的电平是未知的（悬空/浮空状态）。此时空气中的电磁杂波会使引脚在高低电平间疯狂乱跳。为了稳定它，我们必须接一个电阻：
   - **上拉电阻**：通过电阻连接到 3.3V。当外部开关断开时，引脚被牢牢“拉高”到 3.3V。
   - **下拉电阻**：通过电阻连接到 GND。当外部开关断开时，引脚被牢牢“拉低”到 0V。

## 3.3 openBattery 专属实战：I2C 基线与高速中断通道

在双核架构中，MCU 和 SoC 的物理连线不仅只有通信线，还有至关重要的**控制线与告警线**。

通过让 AI 分析双核对接的原型电路图，我们要特别识别并提取以下两部分：

1. **I2C 通信总线 (SDA/SCL)**：这是 MCU 读取 SoC 电压/温度，下发限流策略的“常规数据通道”（一般运行在 100kHz 或 400kHz 速率，必须接 4.7kΩ 的外部上拉电阻）。
2. **紧急告警高速通道 (物理中断引脚 INT)**：
   - **为什么需要硬件中断？**：如果发生物理输出短路或电芯过温，依靠 I2C 轮询（通常几十毫秒级）极易因为主控卡死或轮询不及时导致烧板。
   - **物理接线设计**：SoC 的 `INT` 引脚（通常为开漏输出，平时为高电平，有警报时拉低）直接连接到 MCU 的 `PA4` 引脚。
   - **软件配置与响应流**：
     - 在当前仓库的 `hal/exti/` 中，将 `PA4` 配置为**外部中断模式（下降沿触发，开启内部上拉电阻）**。
     - 一旦触发下降沿，MCU 瞬间进入 `EXTI4_IRQHandler`，再调度到上层紧急回调函数，实现极速物理响应！

### 极速响应 C 代码示例（基于当前仓库的 `hal/exti.c`）：

```c
/**
 * @brief EXTI4 中断服务入口
 * 当 SoC 的 INT 引脚发生下降沿跳变时，硬件自动触发此中断
 */
void EXTI4_IRQHandler(void)
{
    /* 检查是否是 PA4 对应的 EXTI4 触发 */
    if ((EXTI_PR & (1u << 4)) != 0u)
    {
        EXTI_PR = (1u << 4); /* 清除挂起标志 */

        if (s_exti_callbacks[4] != 0) {
            s_exti_callbacks[4]();
        }
    }
}
```

在当前工程里，中断回调最终会落到 `power_service_handle_emergency_stop()`，它做的事情不是复杂诊断，而是**先把系统切进 `POWER_STATE_EMERGENCY_LOCK` 并立即改变板级状态**。真正的 I2C 诊断动作放在主循环里做，而不是在中断里做。

## 3.3.1 开发板、芯片、量产候补在硬件层面怎么区分？

从硬件视角看，这三者的角色完全不同：

- **开发板**：帮你快速接上电源、串口、按键、LED、SWD 调试器。
- **MCU 芯片**：真正提供 GPIO、I2C、ADC、EXTI、Flash、RAM 的器件本体。
- **量产候补**：为了成本、资源和供货去评估的后续 MCU 路线。

例如当前项目里：

- 你手里的 **开发板** 是 `野火小智 F103 核心板`。
- 板上的 **MCU 芯片** 是 `STM32F103C8T6`。
- 未来优先评估的 **量产候补** 是 `F030` 64KB Flash 档位。

这三者不能混着说。
否则很容易把“当前用 F103 学习板做验证”误写成“量产就用 F103”，或者把“F030 是候补”误写成“手上的验证板已经切到 F030”。

## 3.3.2 资源预算也是硬件约束，不只是编译结果

很多软件同学会把 Flash / RAM 占用当成“编译器最后给的一个数字”。
但在受限 MCU 平台里，它本质上是硬件边界的一部分。

### 1. Flash 是什么？

Flash 存放的是：

- 程序代码
- 常量字符串
- 查表数据
- 字库和静态资源

例如：

```text
FLASH: 31348 B / 64 KB = 47.83%
```

它表示当前程序和常量已经占用了 64KB 程序存储空间中的约 47.83%。

### 2. RAM 是什么？

RAM 存放的是：

- 全局变量
- 栈
- I2C / UART 缓冲区
- 事件队列
- 运行中的临时状态

例如：

```text
RAM: 4184 B / 20 KB = 20.43%
```

它表示程序运行时已经占用了约 20KB 可写内存中的 20.43%。

### 3. 为什么这对 F030 候补路线特别重要？

因为当前量产候补路线优先评估的是 `F030` 64KB Flash 档位。
这意味着很多在 PC 上、甚至在更大 MCU 上“看起来无所谓”的东西，在这里都会迅速变成资源压力，例如：

- 大字库
- 复杂 UI
- BLE 协议栈
- 双镜像 IAP
- 大量诊断日志字符串

所以更准确的工程判断应该是：

> 资源占用不是“编不过再优化”的末尾动作，而是从架构阶段就要持续盯住的设计约束。

## 3.4 让 AI 做“避坑审查”

在生成代码前，你可以多问 AI 一句：
> “基于上面梳理的引脚表，从软件配置的角度，请指出原理图中可能存在的开发陷阱。”

AI 可能会敏锐地提醒你：“SoC 的 INT 引脚是开漏输出，如果原理图上没有设计外部上拉电阻，MCU 的 GPIO 必须在初始化时显式开启上拉（当前工程的 `hal_exti_init()` 就是这样处理的），否则中断线悬空会导致主控不断误触发中断而卡死！”

这一步的提前干预，往往能帮你省下抱着示波器死磕一整天的时间！
