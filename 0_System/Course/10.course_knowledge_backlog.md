## 2. 按章节建议落位

### 2.1 第 1 章：起步与基建

建议补入：

- `K-001` 开发板、MCU、量产候补三者的区别
- `K-002` 分层目录的真实作用
- `K-009` 当前工程主线所需工具链与常见误区
- `K-010` 为什么配置和文档都要尽量使用相对路径

适合强调的判断：

- `F103` 当前是学习 / bring-up 入口，不是量产结论。
- `F030` 当前是量产候补，不是已经定板。
- VS Code 插件、命令行工具链、STM32CubeCLT 是不同层次的东西，不能混成一句“已经装了 STM32 环境”。

### 2.2 第 3 章：硬件与资源边界

建议补入：

- `K-001` 开发板与量产 MCU 的区别
- `K-008` Flash / RAM 占用怎么看

适合强调的判断：

- `RAM: 4184 B / 20 KB` 指的是运行时可写内存占用。
- `FLASH: 31348 B / 64 KB` 指的是程序与常量占用。
- 在 F030 候补路线下，资源预算是架构约束，不是后期“编不过再说”的问题。

### 2.3 第 5 章：代码实战

建议补入：

- `K-003` 公共 API 不泄露 HAL 细节
- `K-004` startup 技术债识别
- `K-006` 中断快路径 / 慢路径
- `K-012` bus recovery 的作用和边界

适合强调的判断：

- `soc_sal.h` 这种公共头文件不应直接暴露 `hal_i2c.h`。
- demo register、自测 hook、软件触发 EXTI 都可以存在，但必须明确退出条件。
- bus recovery 能解决一部分 I2C 锁死问题，但不能替代错误状态机和硬件排查。

### 2.4 第 6 章：维护与迁移

建议补入：

- `K-003` SoC SAL 如何从单文件 demo 过渡到 adapter 结构
- `K-004` startup 技术债如何清理
- `K-005` 三层配置模型
- `K-011` demo 骨架与正式平台代码的区别

适合强调的判断：

- 迁移不是“把 F103 的每一行原样搬到 F030”。
- 配置双真相会直接破坏裁剪、验证和协作判断。
- 进入真实 SoC 联调前，应先把自测路径、正式路径、产品路径分开。

### 2.5 第 7 章：工作流与验证

建议补入：

- `K-007` 三层验证观
- `K-010` 配置文件为什么要用相对路径

建议建立的验证分层：

- **软件验证**：能编译、能运行主循环、日志正常、基本路径可走通。
- **板级验证**：能下载、能点灯、能串口、能 I2C、能 EXTI、能抓波形。
- **产品级验证**：能证明真实保护、温升控制、故障处理、功率路径行为和合规要求。

核心提醒：

- 软件触发 `EXTI_SWIER` 不等于真实 SoC INT。
- LED 熄灭不等于真实功率切断。
- 编译通过不等于系统已可量产。

## 3. 当前最应该优先落文档的 4 个主题

如果短期只补最有价值的内容，建议按下面顺序推进：

1. **配置三层模型**
   - `PROFILE_*`
   - `FEATURE_*`
   - `APP_ENABLE_*`

2. **SoC SAL 的正确拆分方式**
   - 公共 API
   - 平台类型
   - transport
   - vendor adapter

3. **startup 技术债**
   - 什么叫“临时合理”
   - 什么叫“长期危险”
   - 如何定义退出条件

4. **验证分层**
   - 软件验证
   - 板级验证
   - 产品级验证

这 4 个主题直接决定后续：

- 新手是否能看懂当前工程状态
- 文档是否和代码状态一致
- 架构是否会在 startup 阶段被临时代码侵蚀

## 5. 使用方式

后续每当会话里反复出现以下类型的问题，都应考虑是否回写到 `Course/`：

- 新手反复混淆的概念
- 当前代码 review 中多次暴露的架构坏味道
- 环境配置中已经踩过、且会重复踩的坑
- 文档、代码、计划三者之间容易失真的判断点

判断标准可以简单记成一句：

> **凡是会影响“下一位开发者是否能做对判断”的内容，都值得进入课程。**
