## 1. 文档目的

本文件不是对 `Embedder` 产品做泛泛介绍，也不是只看 `Quickstart` 后做的局部总结。

本文件的目标更严格：

> 先按官方 docs 索引把可读页面过一遍，再从中筛出对 `openBattery` 真有价值的流程、知识点和模式，最后形成一份适合当前项目阶段的学习文档。

这里要特别区分三件事：

1. **Embedder 是什么**
   - 一个面向固件/嵌入式开发的 agent 平台
2. **它有哪些产品能力**
   - 文档摄入、代码生成、串口、GDB、逻辑分析仪、示波器、电流分析、MCP、VS Code、Headless Server 等
3. **`openBattery` 真正要借鉴什么**
   - 不是整套产品形态，而是它组织嵌入式开发闭环的方式

因此，本文始终坚持一个边界：

> 借鉴工作流，不替代 `openBattery` 自己的 `MCU + SoC` 架构、SoC 抽象层和板级/产品级验证职责。

## 4. 从全站提炼出的 9 个核心模式

这是本次扫描后，对 `openBattery` 最值得吸收的 9 个模式。

### 4.1 模式 1：项目启动不是“开聊”，而是“先绑定硬件上下文”

Embedder 的启动流程不是一上来写代码，而是先完成：

1. 选项目
2. 选平台
3. 选外设
4. 生成 `EMBEDDER.md`

这背后的方法含义是：

> AI 在写固件前，必须先知道“这块板子是谁、接了什么、构建方式是什么、该参考哪些资料”。

对 `openBattery` 的意义非常直接：

1. 当前 MCU：`STM32F103C8T6`
2. 当前板：野火小智双 USB 核心板
3. 当前目标：bring-up，不是完整产品功能
4. 当前重点外设：LED、UART、I2C、ADC、EXTI、watchdog
5. 当前代码入口：`openBmsClaw/`
6. 当前主线边界：`MCU + SoC`，先抽象后接真实 SoC

换句话说，`openBattery` 后续也应该有自己的“项目启动上下文文件”，而不是每次临时口述。

---

### 4.2 模式 2：项目级上下文文件要短、硬、稳定

`Best Practices` 里一个非常重要的点是：`EMBEDDER.md` 必须短、清晰、长期有效。

它不是：

1. 整个项目说明书
2. 逐文件介绍
3. 随时变化的任务记录

它应该只放：

1. 常用构建命令
2. 代码风格
3. 工作流约束
4. 非显而易见的环境差异
5. 长期有效的项目规则

这对 `openBattery` 很有借鉴价值。后续如果做自己的上下文文件，建议只放：

1. 当前主控与板卡
2. 当前 `profile`
3. build / flash / log 命令
4. 分层依赖禁令
5. 当前阶段最小验证项
6. 当前 SoC 接入边界

不应放：

1. 大量阶段性任务
2. 冗长背景
3. 能从代码直接看出的事实

---

### 4.3 模式 3：Plan / Act / Debug 明确分模

Embedder docs 的一个核心设计，不是“一个 agent 什么都干”，而是把工作分成至少三种模式：

1. `Plan`
   - 只读、分析、出计划
2. `Act`
   - 真正改代码、构建、刷写
3. `Debug`
   - 进入硬件感知的调试模式

这件事很重要，因为它把三种不同脑力活动分开了：

1. 理解问题
2. 执行修改
3. 探测硬件现象

对 `openBattery` 的借鉴不是照搬命令，而是照搬思维分层：

1. 架构或复杂改动前，先分析和出计划
2. 简单改动直接执行
3. 真涉及板级现象时，必须切到“调试闭环思维”

这和当前 `openBattery` 文档主线其实高度一致：

1. 先 bring-up
2. 再固化分层
3. 再接 SoC
4. 再做服务层闭环

---

### 4.4 模式 4：串口不是附属品，而是第一层运行时证据

从 `Quickstart`、`Common Workflows`、`VS Code extension`、`Changelog` 可以看到，Embedder 把串口当成一等输入：

1. 自动读串口输出
2. build & flash 后立刻接串口
3. 串口支持过滤
4. 多串口 tab
5. 直接向串口发送命令

这对 `openBattery` 的现实意义很强。

当前阶段最先要稳定的，不是 fancy UI，而是：

1. 启动日志
2. 错误码
3. 状态切换日志
4. I2C timeout / recovery 日志
5. EXTI / watchdog / fault 关键日志

可以说：

> 对当前 `openBattery` 而言，串口日志就是最便宜、最稳定、最早可落地的 HIL 入口。

---

### 4.5 模式 5：调试必须走 Observe → Hypothesize → Probe → Verify

`Debug Mode` 总览最重要的，不是“支持哪些仪器”，而是它把调试抽象成固定闭环：

1. Observe
2. Hypothesize
3. Probe
4. Verify

这正好可以被 `openBattery` 直接吸收成统一排障法。

例如当前项目常见问题：

1. I2C NACK
2. 串口没日志
3. EXTI 不触发
4. 上电后卡死
5. SoC 无响应

建议统一改成下面模板：

1. **现象**
   - 例如：读 SoC ID 偶发失败
2. **候选根因**
   - 地址错、时序错、上拉不足、总线挂死、INT 链路未通
3. **探测动作**
   - 看串口、抓 I2C、断错误回调、读 fault 寄存器
4. **验证标准**
   - 最后一帧事务、错误寄存器值、恢复后是否重新 ACK

这样可以大幅减少“先拍脑袋改代码”的无效试错。

---

### 4.6 模式 6：数字协议问题和模拟电气问题必须分开

全站扫描后，一个很值得重视的点是：Embedder 把 `logic analyzer` 和 `oscilloscope` 明确区分。

逻辑分析仪适合回答：

1. 有没有波形
2. 地址对不对
3. 数据帧是什么
4. 协议层有没有 ACK/NACK

示波器适合回答：

1. 上升沿够不够快
2. 电源有没有跌落
3. 时钟有没有振铃
4. 复位脚有没有毛刺

这对 `openBattery` 是非常关键的实践提醒：

> 数字 decode 正常，不代表电气质量正常。  
> 电气质量差，常常正是偶发协议错误的根因。

因此后续对 I2C/SPI/UART 问题，建议固定两层判断：

1. 先抓数字协议
2. 有偶发/边缘问题时，再上示波器查模拟质量

---

### 4.7 模式 7：功耗调试不是独立任务，而要和事件/日志/寄存器关联

`Power analyzer` 和 `Combined workflows` 提炼出的关键思想是：

> 功耗问题必须做“时间对齐”，不能只看一条平均电流曲线。

它强调把下面几类信息放到同一时间轴：

1. 电流/电压
2. GPIO marker
3. UART 日志
4. GDB 寄存器快照
5. 中断或状态变量

这对 `openBattery` 当前不是第一优先级，但对后续非常有价值，尤其是：

1. 休眠电流异常
2. 上电峰值异常
3. 功率切换瞬态
4. 过热降额行为
5. 无线/显示/SoC 联动的瞬时功耗

短期可先不买齐全套工具，但方法上应先记住：

> 功耗问题后面本质上也是“多源证据对齐”的问题。

---

### 4.8 模式 8：需要有“可读但不可自动改”的保护区

`.embedderignore` 的设计非常值得 `openBattery` 借鉴。

它的核心思想不是“禁止 agent 看文件”，而是：

1. 允许读
2. 禁止自动改
3. 一旦要改，必须显式确认

对嵌入式项目尤其重要，因为以下内容经常不适合被随手改：

1. vendor SDK
2. 上游 HAL
3. 自动生成文件
4. 构建产物
5. 敏感配置
6. 已验证稳定的板级启动文件

对 `openBattery` 的直接建议是：

后续如果要建立自己的 agent 保护机制，优先把以下目录视为候选保护区：

1. `build/`
2. 自动生成代码目录
3. vendor / third_party
4. 未来真实 SoC 提供的原始 SDK 适配层
5. 不应被频繁重写的底层启动/链接脚本

这能显著降低“为了修一个问题，顺手破坏底层稳定代码”的风险。

---

### 4.9 模式 9：Agent 平台本身也要可编排、可扩展、可嵌入

全站中最“平台味”的内容，集中在：

1. `Headless server mode`
2. `MCP servers`
3. `GitHub bot setup`
4. `Changelog` 中的 skills / subagents / MCP 演化

这些内容对当前 `openBattery` 不是第一优先级，但说明了一个方向：

> 一个成熟的嵌入式 agent 不只是聊天窗口，而是可被 IDE、脚本、自动化、外部工具和组织流程共同调用的系统。

对 `openBattery` 的实际含义是：

短期先不要做这块；

但中长期如果你们要把自己的“嵌入式开发方法”沉淀成工具，至少可以预留三个方向：

1. 自定义 skill / prompt workflow
2. 外部工具接入层
3. 程序化调用接口

## 5. 按五大主题重写后的 `openBattery` 可借鉴清单

下面按你要求的五部分重新归纳。

### 5.1 Bring-up

#### 应借鉴什么

1. 启动前先绑定平台、板卡、外设、构建方式
2. 生成一个短小稳定的项目上下文文件
3. 先问“当前工程如何初始化”，再动代码
4. build / flash / serial 要连成一个最小闭环
5. Windows 下更适合把 IDE 和串口视图放一起

#### 对 `openBattery` 的具体落地

建议把当前 bring-up 固化成一个标准顺序：

1. 确认 `STM32F103C8T6` 板卡和调试器
2. 跑通 `cmake --preset Debug`
3. 跑通 `cmake --build --preset Debug`
4. 烧录并确认 `boot log`
5. 再进入 LED / UART / I2C / ADC / EXTI 单项验证

#### 当前最值得沉淀的产物

1. 项目级上下文文件
2. bring-up 清单
3. 启动日志基线
4. 最小验证表

---

### 5.2 Driver 接入

#### 应借鉴什么

1. 先选平台和外设文档，再接 driver
2. 先从 datasheet / errata 提取事实
3. 先定义公共契约，再写 adapter
4. 让 agent 基于现有 HAL/SDK 模式生成代码
5. 对 vendor / HAL / generated 文件设保护边界

#### 对 `openBattery` 的具体落地

SoC 接入建议固定成下面 6 步：

1. 提取寄存器、状态、时序、INT 条件
2. 写公共 API 与类型
3. 绑定 transport
4. 落 vendor adapter
5. 接 `services`
6. 最后再让 `app` 使用

#### 不能借鉴错的地方

不能因为 docs 里展示“可直接生成 driver”，就跳过：

1. SoC 公共能力定义
2. 单位口径统一
3. 错误码模型
4. 快路/慢路安全边界

---

### 5.3 调试

#### 应借鉴什么

1. 现象-假设-探测-验证闭环
2. 串口、GDB、逻辑分析仪、示波器、电流分析分工明确
3. 多工具联合调试要围绕同一故障窗口组织
4. 非侵入式 attach 很重要
5. 捕获结果要保存，方便复查

#### 对 `openBattery` 的具体落地

建议固定一条“工具升级路径”：

1. 先串口
2. 再 GDB
3. 再逻辑分析仪
4. 再示波器
5. 有功耗问题再上电流分析

#### 最适合当前项目的典型场景

1. I2C NACK
2. 总线恢复失败
3. EXTI 无响应
4. 上电卡死
5. fault handler 触发
6. 电源切换瞬态异常

---

### 5.4 验证

#### 应借鉴什么

1. 改动前就写成功标准
2. 验证必须可执行
3. 不只验证“编译过”，还要验证运行证据
4. 长会话要压缩和清理上下文，避免错误累积
5. 失败两三轮后要重开干净上下文，而不是无限补丁

#### 对 `openBattery` 的具体落地

建议继续坚持三层验证：

1. 软件层
   - build、链接、日志、错误码
2. 板级层
   - 烧录、串口、I2C ACK、EXTI、ADC、watchdog
3. 产品层
   - 热、保护、功率分配、法规边界

#### 当前阶段最现实的做法

1. 只要求最小必要验证
2. 每次输出都写明已验证/未验证
3. 不宣称未做的板测和联调已完成

---

### 5.5 Skills 化

#### 应借鉴什么

1. 固定入口动作
2. 固定上下文来源
3. 固定模式切换
4. 固定验证格式
5. 用子 agent / 子任务隔离探索上下文

#### 对 `openBattery` 的具体落地

建议优先沉淀 6 类 skill：

1. `bringup-analyzer`
   - 输入：板卡、MCU、现象、目标外设
   - 输出：初始化路径、关键文件、最小验证项

2. `datasheet-extractor`
   - 输入：datasheet、errata、应用笔记
   - 输出：寄存器表、时序限制、风险点

3. `soc-adapter-planner`
   - 输入：SoC 型号、公共能力目标
   - 输出：API、类型、错误码、事件模型

4. `debug-investigator`
   - 输入：现象、复现条件、已有日志、可用仪器
   - 输出：候选根因、探测动作、验证标准

5. `verification-checklist`
   - 输入：任务类型、改动范围
   - 输出：最小验证表、未验证项、残余风险

6. `edit-boundary-guard`
   - 输入：目录结构
   - 输出：哪些路径可读不可改，哪些路径必须确认后才可改

#### 后续高级方向

如果未来真要做“agent 化开发平台”，再考虑：

1. 程序化 headless 接口
2. MCP 外部工具接入
3. PR / issue 自动流转

这三项现在不该提前扩 scope。

## 6. 哪些内容当前不建议重投入

全站扫描后，有几类内容我认为现在不值得 `openBattery` 立刻重投入：

### 6.1 不建议先做的产品化能力

1. 自己做完整 VS Code 扩展
2. 自己做 JSON-RPC Headless Server
3. 自己做 GitHub bot 守护进程
4. 自己做 MCP marketplace

原因很简单：

当前项目的主矛盾不是 agent 外壳，而是：

1. 真实 SoC 闭环还没完成
2. F103 bring-up 与 F030 候补验证还在推进
3. SoC 抽象和服务层闭环还没完全坐实

### 6.2 不建议过早照搬的调试设施

1. 一口气搭完整多仪器自动化
2. 一开始就做复杂功耗联合分析平台
3. 过度依赖自动脚本取代人工波形判断

更现实的顺序仍应是：

1. 串口
2. GDB
3. 逻辑分析仪
4. 示波器
5. 后续再扩展功耗工具

## 7. 对当前主线的最终建议

把全站扫描后的高价值内容压缩成一句话：

> `openBattery` 最该从 Embedder 借鉴的，不是“AI 自动写更多代码”，而是“先绑定硬件事实、再规划、再执行、再用多源证据完成闭环验证”的嵌入式工程方法。

因此，当前建议顺序是：

1. 先建立 `openBattery` 自己的项目级上下文文件
2. 先建立最小验证表
3. 先建立 `bringup / adapter / debug / verification` 四类 skill
4. 先把串口 + GDB + 逻辑分析仪三件套用顺
5. 等真实 SoC 联调稳定后，再考虑更重的自动化外壳

最终判断：
