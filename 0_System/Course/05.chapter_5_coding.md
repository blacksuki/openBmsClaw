## 5.1 深度剖析：为什么你的 I2C 驱动会“无故挂死”？

在智能移动电源中，由于外部快充大电流开关噪声、插拔火花干扰，主控 MCU 与电池/快充 SoC 之间的 I2C 通信极易挂死。其中最致命的顽疾就是 **I2C 总线死锁（SDA 锁死拉低）**。

### 1. 锁死拉低的物理本质
I2C 是开漏双向总线。在读取数据时，外部 SoC（从机）会接管 SDA 并将其拉低输出 `0`。
若恰逢此时：
- MCU（主机）因高优先级中断、看门狗复位或异常打断了当前传输。
- 此时主机的 SCL 停止翻转（保持高电平），而外部从机还固执地处于等待第 9 个 Clock 脉冲来释放 SDA 的状态，它会**持续将 SDA 物理拉低**。
- 当 MCU 重新初始化复位启动时，它首先去检测总线状态，发现 SDA 已经是低电平（从机在拉低），MCU 认为总线被占用了（Bus Busy），于是卡死在 `while(Busy)` 循环中，大脑彻底罢工！

### 2. 破局之道：9 脉冲自愈 (Bus Recovery)
解决它的物理逻辑非常精妙：MCU “假装”自己是 I2C 硬件控制器，用普通 GPIO 去手动上下翻转 SCL 线。每翻转一次，就相当于给从机发了一个 Clock 脉冲。从机以为主机在继续读取，就会往后移位。通常在 9 个脉冲内，从机必然会交出控制权，释放 SDA（使其重新被上拉电阻拉高）。

## 5.2 openBattery 工业级自愈与非阻塞驱动代码

在 `openBmsClaw` 项目中，我们严格遵守不写死等 `while` 循环的底线。下面是我们在 `hal/` 与 `drivers/` 层面推行的两个核心函数实现，请开发者直接作为模板和理解标准：

### 1. I2C 总线自愈函数 `soc_bus_recovery()` (在 `drivers/` 中)

```c
#include "board.h"
#include <stdint.h>

/**
 * @brief I2C 总线死锁自愈恢复
 * 当检测到总线 Busy 且 SDA 持续被拉低时调用
 */
void soc_bus_recovery(void)
{
    GPIO_InitTypeDef GPIO_InitStruct = {0};

    log_warn("I2C Bus Lockup Detected! Starting recovery...");

    // 1. 临时将 SCL 引脚配置为普通 GPIO 开漏输出模式
    GPIO_InitStruct.Pin = BOARD_I2C_SCL_PIN;
    GPIO_InitStruct.Mode = GPIO_MODE_OUTPUT_OD;
    GPIO_InitStruct.Pull = GPIO_PULLUP;
    GPIO_InitStruct.Speed = GPIO_SPEED_FREQ_HIGH;
    HAL_GPIO_Init(BOARD_I2C_GPIO_PORT, &GPIO_InitStruct);

    // 2. 临时将 SDA 引脚配置为普通 GPIO 输入模式 (用于监测状态)
    GPIO_InitStruct.Pin = BOARD_I2C_SDA_PIN;
    GPIO_InitStruct.Mode = GPIO_MODE_INPUT;
    HAL_GPIO_Init(BOARD_I2C_GPIO_PORT, &GPIO_InitStruct);

    // 3. 循环发送最多 9 个时钟脉冲，直到从机释放 SDA
    for (uint8_t i = 0; i < 9; i++)
    {
        // 如果读取到 SDA 已经被释放拉高，说明从机已退出挂起状态
        if (HAL_GPIO_ReadPin(BOARD_I2C_GPIO_PORT, BOARD_I2C_SDA_PIN) == GPIO_PIN_SET)
        {
            log_info("SDA released after %d clocks", i);
            break;
        }

        // 翻转 SCL 制造一个时钟脉冲 (周期约 10us，等效 100kHz 速率)
        HAL_GPIO_WritePin(BOARD_I2C_GPIO_PORT, BOARD_I2C_SCL_PIN, GPIO_PIN_RESET);
        HAL_Delay_us(5);
        HAL_GPIO_WritePin(BOARD_I2C_GPIO_PORT, BOARD_I2C_SCL_PIN, GPIO_PIN_SET);
        HAL_Delay_us(5);
    }

    // 4. 手动产生一个 I2C STOP 信号以彻底使总线归位
    // STOP 信号定义：当 SCL 为高电平时，SDA 从低电平跳变到高电平
    GPIO_InitStruct.Pin = BOARD_I2C_SDA_PIN;
    GPIO_InitStruct.Mode = GPIO_MODE_OUTPUT_OD;
    HAL_GPIO_Init(BOARD_I2C_GPIO_PORT, &GPIO_InitStruct);

    HAL_GPIO_WritePin(BOARD_I2C_GPIO_PORT, BOARD_I2C_SDA_PIN, GPIO_PIN_RESET);
    HAL_Delay_us(5);
    HAL_GPIO_WritePin(BOARD_I2C_GPIO_PORT, BOARD_I2C_SCL_PIN, GPIO_PIN_SET);
    HAL_Delay_us(5);
    HAL_GPIO_WritePin(BOARD_I2C_GPIO_PORT, BOARD_I2C_SDA_PIN, GPIO_PIN_SET); // SDA 上升沿
    HAL_Delay_us(5);

    // 5. 恢复引脚的 I2C 硬件外设复用配置
    board_i2c_gpio_reinit(); 
    log_info("I2C GPIO re-initialized.");
}
```

### 2. 带软件 Timeout 的非阻塞 I2C 读操作 `hal_i2c_read()` (在 `hal/` 中)

当前 `openBmsClaw` 仓库中的 `hal_i2c.c` 采用的是**纯寄存器访问实现**，不是对 STM32 HAL 库 `HAL_I2C_Mem_Read()` 的二次封装。下面给出一个与当前代码风格一致的简化示例：

```c
#include "hal_i2c.h"

/**
 * @brief 带软件超时保护的 I2C 读字节函数
 * @retval HAL_OK 成功, HAL_TIMEOUT 超时失败
 */
hal_i2c_status_t hal_i2c_read(uint8_t dev_addr, uint8_t reg_addr,
                              uint8_t *data, uint16_t len, uint32_t timeout_ms)
{
    if ((data == 0) || (len == 0)) {
        return HAL_I2C_ERR_BUS_ERROR;
    }

    if ((I2C1_SR2 & I2C_SR2_BUSY) != 0u) {
        hal_i2c_bus_recovery();
        if ((I2C1_SR2 & I2C_SR2_BUSY) != 0u) {
            return HAL_I2C_ERR_BUSY;
        }
    }

    I2C1_CR1 |= I2C_CR1_START;
    if (!hal_i2c_wait_mask(&I2C1_SR1, I2C_SR1_SB, true, timeout_ms)) {
        I2C1_CR1 |= I2C_CR1_STOP;
        return HAL_I2C_ERR_TIMEOUT;
    }

    /* 后续按顺序执行：发送地址、发送寄存器地址、Repeated START、切换读方向、读取数据 */
    return HAL_I2C_OK;
}
```

## 5.3 进阶实战：紧急告警高速通道 (EXTI 中断与两阶段异步自锁)

在智能移动电源的 BMS 与大功率充放电系统中，一旦发生**电芯短路、瞬时过流、严重温升（NTC 触发危急值）**等突发性灾难事件，系统必须以**微秒级（Microseconds）**的速度切断功率通路。

然而，传统的“同步 I2C 轮询获取状态”耗时几毫秒到几十毫秒，对于短路保护来说慢得像蜗牛；但如果直接在硬件中断服务程序（ISR）中去发起 I2C 读写，又会挂死在中断里导致看门狗复位。

### 1. 两阶段异步响应自锁设计
为了彻底破除这一顽疾，`openBattery` 在 `services/power` 中实现了 **“两阶段异步自锁状态机”**：

- **【第一阶段】反射区 (硬件中断级，微秒响应)**：
  SoC 中断引脚 (PA4 -> EXTI4) 拉低 $\rightarrow$ 进入 MCU 中断服务函数 `EXTI4_IRQHandler`。在此处**严禁任何 I2C/格式化/Delay**，仅瞬间将全局状态标记为 `POWER_STATE_EMERGENCY_LOCK`，并直接修改 GPIO 寄存器（拉低 ODR 物理熄灭 LED 灯），几微秒内退出中断。
- **【第二阶段】决策区 (主循环慢速诊断，毫秒判定)**：
  在主循环中捕捉到 `POWER_STATE_EMERGENCY_LOCK` 自锁标记后，安全地在中断外发起带超时保护的慢速 `hal_i2c` 去读取 SoC 状态寄存器，精确定位故障原因是短路还是过温，在终端上格式化打印诊断报告，并高频闪烁状态灯持续警示。

### 2. 纯寄存器级底层 EXTI 驱动 (`hal_exti.c`)
为了实现微秒级响应，我们避开了沉重繁琐的标准库，用纯寄存器操作实现中断服务程序：

```c
#include "hal_exti.h"

// EXTI4 硬件中断服务子程序 (ISR)
void EXTI4_IRQHandler(void) {
  // 检查中断挂起寄存器 EXTI_PR 是否由于 EXTI4 触发
  if ((EXTI_PR & (1u << 4)) != 0u) {
    // 必须向 Pending 寄存器写 1 清除标志，防止发生重复进入中断死循环
    EXTI_PR = (1u << 4);

    // 调度注册的极速非阻塞回调函数
    if (s_exti_callbacks[4] != 0) {
      s_exti_callbacks[4]();
    }
  }
}
```

而在 `power_service.c` 中注册的极速回调则简单干练：
```c
void power_service_handle_emergency_stop(uint32_t alarm_mask) {
  // 1. 瞬间锁定状态，防止主循环调度普通充放电配置
  s_power_state = POWER_STATE_EMERGENCY_LOCK;
  s_emergency_alarm_mask = alarm_mask;

  // 2. 秒切物理 IO 寄存器 (拉低 status LED 灭灯表现切断)
  board_status_led_set(false);
}
```

### 3. 100% 软件闭环中断自测
在没有 SoC 评估板和逻辑分析仪的“空档期”，如何验证这套极速中断机制是否工作正常？

我们利用了 STM32 EXTI 中断控制器中一个极其精妙的寄存器 —— **`EXTI_SWIER` (软件中断事件寄存器)**。只要向 `SWIER` 寄存器对应引脚线写入 `1`，硬件内部就会强行产生一次真实的硬件中断触发事件，完美越过无物理板卡的物理限制！

我们在 `app_init()` 结尾处加入了自测代码：
```c
#if APP_ENABLE_POWER_SERVICE
    if (board_has_uart_log()) {
        board_uart_write_string("[Test] Simulating hardware INT low pulse via SWIER...\r\n");
    }
    // 强行往 EXTI_SWIER 对应位写 1，触发硬件 EXTI4 中断
    board_soc_int_sim_trigger();
#endif
```

当前仓库里的软件闭环自测，能够确认的是“`EXTI_SWIER -> EXTI4_IRQHandler -> power_service_handle_emergency_stop()` 这条通路会被触发”。但因为尚未完成真实 SoC 初始化和寄存器联调，诊断日志通常只能证明**中断链路和自锁状态机工作**，而不能证明真实 SoC 事件读取已经打通。

因此，当前阶段更合理的预期日志应理解为：
```text
PowerService: Init OK, emergency callback registered.
boot ok
[Test] Simulating hardware INT low pulse via SWIER...
```

如果后续真正接入 SoC 并完成 `soc_sal_init()`、`soc_poll_events()` 联调，那么才有资格把“事件寄存器读值”和“故障类型解析”当成自测通过标准。

## 5.4 从源码到硬件调试的闭环

代码写完只是开始，调试才是深水区。当你在终端上始终看不到 SoC 传回真实的电流电压跳动时：

1. **喂报错信息**：将编译器的报错（如 `undefined reference to soc_get_voltage`）扔给 AI，它能快速指出你的 CMakeLists 或是头文件路径没有正确包含 `drivers/` 目录。
2. **喂物理现象**：如果你用逻辑分析仪抓包，发现第九个时钟周期没有收到 ACK（应答位），把这个现象告诉 AI，它会立刻指导你检查 SoC 的从机地址（是否忘了左移一位包含读写标志），或者让你检查外部电源是否确实给 SoC 供电了。

**在这个阶段，AI 就是你身边那位“有问必答、不会发脾气”的高级带教导师！**
