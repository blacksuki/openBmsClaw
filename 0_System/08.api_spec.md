## 1. 设计原则

公共 API 应描述**平台领域能力**，而不是描述某颗芯片的寄存器动作。

正例（描述能力，可用）：

```c
soc_sal_status_t soc_get_voltage(uint32_t *voltage_mv);
```

反例（描述寄存器，禁止进入公共头）：

```c
uint8_t ip53xx_read_reg(uint8_t reg);
void sw35xx_write_page(uint8_t page, uint8_t reg, uint8_t value);
```

这条原则保证 `app/` 和 `services/` 不被绑定到某一颗 SoC，换芯片只需新增 `vendor/<chip>.c`，不动上层。

## 2. 当前代码事实（soc_api.h）

以下签名直接来自 `openBmsClaw/drivers/soc/soc_api.h`，是**当前代码事实**。

> ⚠️ 与 03.tech_architecture.md §5.3 的差异：该节（2026-07-18 版）列出的签名（`soc_get_voltage_mv`、`soc_set_port_power_limit`、`soc_poll_events(soc_event_t *, uint8_t)`）是**规划中的理想接口**，当前代码中不存在。本节为准确版本；理想接口作为演进目标保留在 §5。

### 2.1 初始化与总线管理

```c
/* 初始化 SoC 硬件抽象层及底层总线句柄 */
bool soc_sal_init(uint8_t i2c_addr);

/* 查询 SAL 是否已完成运行时初始化 */
bool soc_sal_is_initialized(void);

/* 触发总线死锁自愈（GPIO 模拟 9 个时钟脉冲） */
void soc_sal_bus_recovery(void);
```

契约：
- `soc_sal_init()` 必须在其余所有 `soc_*` 函数之前调用，否则行为未定义。
- `soc_sal_bus_recovery()` 属于**慢路径**操作（含 GPIO 时序，有延时），**禁止在 ISR 上下文调用**。

### 2.2 状态读取

```c
/* 获取电池总电压（毫伏） */
soc_sal_status_t soc_get_voltage(uint32_t *voltage_mv);

/* 获取电池电流（毫安；正数 = 充电，负数 = 放电） */
soc_sal_status_t soc_get_current(int32_t *current_ma);

/* 获取 SoC 内部核心温度（摄氏度整数） */
soc_sal_status_t soc_get_temperature(int16_t *temp_celsius);
```

契约：
- 三个函数均为**阻塞调用**（底层走 I²C 同步读），**禁止在 ISR 上下文调用**。
- 传入 `NULL` 指针行为未定义；调用方负责保证指针非 NULL。
- 返回 `SOC_SAL_OK` 时参数已写入有效值；非 `OK` 时参数内容不保证有效，调用方不得使用。

### 2.3 配置下发

```c
/* 动态下发过流保护阈值（毫安） */
soc_sal_status_t soc_set_ocp(soc_port_t port, uint32_t limit_ma);

/* 动态分配/限制指定端口最大输出功率
 * 当前单位：瓦特（W）—— 已知单位不一致，Phase 1 收敛为毫瓦（mW），见 §4.1 */
soc_sal_status_t soc_set_port_power(soc_port_t port, uint16_t max_power_w);
```

契约：
- 两个函数均为**阻塞调用**（I²C 写），**禁止在 ISR 上下文调用**。
- `port` 参数必须为 `soc_port_t` 枚举值之一（`SOC_PORT_TYPE_C1 / TYPE_C2 / TYPE_A1`）；传入越界值应返回 `SOC_SAL_ERR_PARAM`（当前 demo SoC 实现未保证，Phase 1 需补充校验）。
- `soc_set_ocp` 的 `limit_ma` 参数范围由具体 SoC 型号决定；接入真实 SoC 后须补充范围检查。

### 2.4 事件轮询与紧急回调

```c
/* 轮询/捕获 SoC 异步事件与告警（慢路径，主循环调用） */
soc_sal_status_t soc_poll_events(uint32_t *event_mask);

/* 注册紧急告警回调（在 EXTI 中断上下文直接调用） */
void soc_sal_register_emergency_callback(soc_sal_emergency_callback_t callback);
```

契约：
- `soc_poll_events` 为**阻塞调用**（I²C 读），属于**慢路径**，**禁止在 ISR 中调用**；只在主循环或服务层调度上下文轮询。
- `soc_sal_register_emergency_callback` 注册的回调在 **EXTI 中断上下文直接调用**（快路径）；回调内部**严禁**执行 I²C 读写、`printf`、`delay` 或任何阻塞/慢速操作。
- `soc_poll_events` 的 `event_mask` 各位含义**当前未在头文件中定义**（已知缺口，见 §4.2）。
- 紧急回调参数的掩码位由 `soc_types.h` `SOC_ALARM_*` 宏定义，与 `soc_poll_events` 的 `event_mask` 是独立通道，含义不同。
- `soc_poll_events` 传入 `NULL` 行为未定义。

## 3. 类型定义（soc_types.h）

### 3.1 错误码 `soc_sal_status_t`

| 枚举值 | 数值 | 含义 |
|--------|------|------|
| `SOC_SAL_OK` | 0 | 通信与解析成功 |
| `SOC_SAL_ERR_OFFLINE` | -1 | 从机无响应/离线（未收到 ACK） |
| `SOC_SAL_ERR_BUSY` | -2 | 总线繁忙或被拉低 |
| `SOC_SAL_ERR_PARAM` | -3 | 参数范围错误 |
| `SOC_SAL_ERR_HW_LOCKED` | -4 | 硬件被锁死，需调用 `soc_sal_bus_recovery()` |

错误码处理指引：
- `OFFLINE`：SoC 未响应 ACK，通常表示 SoC 断电或 I²C 地址错误；检查供电和地址配置。
- `BUSY`：总线被拉低；调用 `soc_sal_bus_recovery()` 后重试一次；若仍 BUSY 则上报服务层。
- `HW_LOCKED`：SAL 状态机进入死锁态；需 bus recovery；若恢复后仍 LOCKED 则需 MCU 重启。
- **调用方应对每次 `soc_*` 调用检查返回值**；忽略错误码的调用在 Phase 1 代码评审中视为技术债。

### 3.2 端口类型 `soc_port_t`

| 枚举值 | 数值 | 含义 |
|--------|------|------|
| `SOC_PORT_TYPE_C1` | 0 | Type-C 1 主充放接口 |
| `SOC_PORT_TYPE_C2` | 1 | Type-C 2 充放接口 |
| `SOC_PORT_TYPE_A1` | 2 | USB-A 放电接口 |

### 3.3 紧急告警掩码（`soc_sal_emergency_callback_t` 回调参数）

以下宏用于解析紧急回调参数 `alarm_mask`，**不适用于** `soc_poll_events` 的返回值。

| 宏 | 位 | 含义 |
|----|-----|------|
| `SOC_ALARM_NONE = 0u` | — | 无告警 |
| `SOC_ALARM_SHORT_CIRCUIT = (1u<<0)` | bit 0 | 负载短路告警 |
| `SOC_ALARM_OVER_TEMP = (1u<<1)` | bit 1 | 电池或芯片过温 |
| `SOC_ALARM_OVER_CURRENT = (1u<<2)` | bit 2 | 严重充放电过流 |
| `SOC_ALARM_SYS_ERROR = (1u<<3)` | bit 3 | 底层系统或通信死锁故障 |

## 4. 已知问题与 Phase 1 演进计划

### 4.1 功率单位不一致：瓦特 vs 毫瓦

**当前代码事实：**

```c
soc_sal_status_t soc_set_port_power(soc_port_t port, uint16_t max_power_w);
```

使用**瓦特（W）**，`uint16_t` 上限 65535 W，当前 100 W 以下场景够用。

**规划中（Phase 1 目标）：** 统一收敛为毫瓦（mW），`uint32_t`：

```c
/* Phase 1 目标签名（规划中，当前代码中不存在） */
soc_sal_status_t soc_set_port_power(soc_port_t port, uint32_t max_power_mw);
```

收敛时机：Phase 1 接入第一颗真实 SoC 时，同步更新签名、所有调用方和单位注释，一次性完成。

### 4.2 `soc_poll_events` 事件掩码位未定义

**当前代码事实：** `soc_poll_events` 返回 `uint32_t *event_mask`，各位含义仅在注释中举例（"快充握手成功、短路告警"），**未在头文件中以宏常量形式定义**，调用方无法可靠地按位解析。

**规划中（Phase 1 目标）：** 在 `soc_types.h` 增加轮询事件宏：

```c
/* Phase 1 目标（规划中，当前代码中不存在） */
#define SOC_EVENT_NONE              0u
#define SOC_EVENT_PD_HANDSHAKE_OK   (1u << 0)  /* PD/QC 协议握手成功 */
#define SOC_EVENT_SHORT_CIRCUIT     (1u << 1)  /* 短路告警 */
/* 其余位由接入具体 SoC 后根据芯片事件寄存器补全 */
```

### 4.3 状态读取函数缺少端口参数

**当前代码事实：** `soc_get_voltage`、`soc_get_current` 无 `port` 参数，只能读取全局总值。

**规划中：** 多口充电宝场景可能需要分口读取电流；但在接入真实 SoC 并确认硬件能否支持分口读取之前，不提前变更签名。Phase 1 SoC 接口文档归档后再决策。
