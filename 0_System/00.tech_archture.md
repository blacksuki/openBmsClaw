# openBattery 项目技术架构与工作路径 (SoC 双核协同版)

> 最新核对日期：2026-05-18

## 1. 文档目的
本文档用于明确 `openBattery` (原 `bmsClaw`) 的战略定位、核心架构思想、软件分层规范以及当前演进路径。重点阐述“基于集成 SoC + MCU 双核协同”的新型技术底座，确保所有协作者在架构边界和开发顺序上保持一致。

---

## 2. 当前项目定位

### 2.1 战略与商业目标
`openBattery` 的首发落地场景定位为**日本市场高品质移动电源平台**。
区别于传统的“黑盒”快充产品，本平台致力于通过“安全 + 性能数字化管理（数据可视化）”实现高端产品的差异化体验。

### 2.2 技术架构定位
本平台不再由 MCU 大包大揽底层电源与 BMS 硬件逻辑，而是采用**“SoC 负责体力活，MCU 负责脑力活”**的协同架构：
- **SoC (如英集芯、智融等)**：作为独立系统运行，负责高频的电压转换、快充协议握手 (PD/QC) 及基础的电芯安全保护。
- **MCU (STM32 等)**：作为平台超级大脑，通过标准通信总线 (I2C/UART) 实时获取 SoC 运行数据，进行高级计算与交互（剩余时长预测、动态温度控制、多设备功率分配、屏幕 UI 展示及 BLE 通信）。

## 3. 当前仓库结构与核心原则

### 3.1 当前主线目录
当前主线目录主要包括：

```text
openBattery/
├── 0_System/                当前系统文档与架构规范
├── 1_Plan/                  当前执行计划
├── 90.mini-Lite/            参考项目 (提供行为参考样本，非主线代码)
├── 91.reference/            外部资料归档 (IC 手册、参考设计)
├── openBmsClaw/             当前 STM32 平台实现工程
└── build/                   自动生成构建目录
```

### 3.2 外部 SDK 对接原则 (要接口不要源码)
平台硬件抽象的核心在于兼容多家集成 SoC。在与芯片原厂对接时，坚持以下基准：
1. **拒绝黑盒**：必须提供动态数据读取与参数修改通道。
2. **拒绝硬要源码**：尊重厂商 IP，不需要底层闭源驱动。
3. **标准 API 对接**：依赖且仅依赖厂商提供的应用层库文件或寄存器通信手册，封装标准的 Get/Set 和事件回调机制。

## 4. 核心软件分层架构

为了兼容多家集成 SoC，系统必须进行严格分层设计，防止产品业务逻辑与特定芯片深度耦合。推荐目录与职责划分如下：

```text
app/        (数字化与可视化应用层)
services/   (智能策略与系统服务层)
drivers/    (SoC SDK 封装与抽象层 / SAL)
hal/        (MCU 基础硬件层)
board/      (板级支持与引脚映射)
config/     (系统配置)
```

### 4.1 各层职责拆解

#### `app/` (应用层)
负责终端用户最直观的体验。
- **人机交互**：按键状态机处理、OLED/LED 屏幕渲染更新。
- **数字化引擎**：获取底层数据，运行剩余使用时长预测模型和电池健康度 (SOH) 算法。
- **外部对接**：处理 IAP 固件升级与外部 BLE/APP 通信。

#### `services/` (服务层)
负责高级系统策略的仲裁和调度。
- **功率调度器 (`srv_power_alloc`)**：多设备接入时，动态仲裁并向 SoC 下发端口输出限额。
- **热管理守护 (`srv_thermal`)**：结合 SoC 内置温度与外部 NTC，动态干预 SoC 运行（如下调 OCP 阈值、降低输出功率）。

#### `drivers/` (驱动与抽象层 - SAL)
负责抹平英集芯、智融等各家 SoC 的底层差异。
- **提供通用 API**：对上层暴露一致的 `soc_get_voltage()`, `soc_get_temperature()`, `soc_set_port_power()` 接口。
- **适配具体芯片**：内部实现针对 IP53xx 或 SW35xx 的具体通信逻辑。
- **事件上报机制**：将 SoC 发生的中断（如快充握手成功、短路告警）统一转化为平台标准事件通知上层。

#### `hal/` 与 `board/` (底层)
- 仅负责 MCU 自身的 I2C/UART/ADC 等基础外设驱动，和具体的 IO 映射。

## 5. 平台演进与工作路径

平台实施路径已从传统的“自底向上造轮子”，调整为“以 SoC 协同验证为中心”的迭代路线：

### 5.1 阶段 1：稳定 MCU 起步环境
- 基于现有的生成工程，调通基础的 GPIO, I2C, UART。
- 确保具备稳定的日志打印和调试手段。

### 5.2 阶段 2：构建 SoC 抽象层与基线接入 (核心里程碑)
- 选取首发选型的 SoC（如英集芯系列），调通 MCU 与其之间的 I2C/UART 通信。
- 完成标准数据读取、参数动态配置、中断信号捕获的联调验证。
- 形成第一版稳定的 `drivers/` SoC 适配器。

### 5.3 阶段 3：服务层与智能策略落地
- 编写 `srv_thermal` 和 `srv_power_alloc`。
- 重点验证 MCU 能否根据策略实时改变 SoC 的充放电行为，完成“脑体协同”闭环。

### 5.4 阶段 4：数字化应用与交互
- 接入 OLED/LED，基于抽取的真实数据，验证时长预测算法和 UI 呈现。
- 完善按键长短按业务逻辑流转。

### 5.5 阶段 5：合规与系统级优化
- 针对日本 PSE 认证、≤100Wh 航空安全标准、热设计以及 EMC 进行软硬件联合调试。
- 固化各配置项，为其他同构产品建立快速克隆基线。

---

> 注：一切代码修改请严格遵循上述“脑体分离”原则，严禁在 `app/` 中直接通过 I2C 读写 SoC 寄存器。
