# bmsClaw 项目技术架构与当前工作路径

> 最新核对日期：2026-03-28

## 1. 文档目的
本文档用于明确 `bmsClaw` 当前的项目定位、仓库边界、技术目标和工作路径，解决“当前主项目是谁、参考项目是谁、当前代码从哪里开始”这三个最容易混淆的问题。

本文档优先回答：

1. 当前项目主名称是什么
2. 当前代码主线在哪里
3. `90.mini-Lite` 在本仓库里扮演什么角色
4. 现阶段应该先做什么、后做什么

---

## 2. 当前项目定位

### 2.1 当前主项目名称
当前主项目名称统一为：

> **`bmsClaw`**

后续所有文档、计划、说明和代码分析，默认都应围绕 `bmsClaw` 展开。

### 2.2 当前目标
`bmsClaw` 当前目标不是直接复制某个现成产品，而是构建一套：

- 可定制
- 低成本
- 合规导向
- 适合中小厂商二次开发

的开源移动电源 / 电池管理平台方案。

### 2.3 当前不是目标本体的内容
以下内容不是当前项目主线：

- `90.mini-Lite/`
  这是参考项目，只用于帮助理解典型产品需求。
- `91.reference/`
  这是外部资料归档区，不是当前实现。
- `build/`
  这是自动生成构建目录，不是主源码目录。

## 4. 当前代码事实

### 4.1 当前已有实现
当前仓库中已经存在一个 STM32 生成工程：

- 工程路径：`bmsClaw/bmsClaw/bmsClaw/`
- 工程名：`bmsClaw`
- 生成方式：`STM32CubeIDE for Visual Studio Code`

### 4.2 当前工程状态
从当前文件可确认：

- `main.c` 仍处于最小空循环状态
- 工程尚未形成 `app/services/hal/drivers/board/config` 分层
- 构建相关文件由 STM32 工具生成
- 启动文件和链接脚本显示当前起步工程更接近 `STM32F072xB` 基线

这意味着：

- 当前代码主线已经从“纯文档阶段”进入了“有实际工程骨架”的阶段
- 但离完整平台框架仍有明显距离

### 4.3 当前代码和推荐路线的关系
当前代码事实不等于最终平台路线完全定型。

更准确的表述应是：

- **当前代码事实**：已有 `STM32F072xB` 风格的 STM32 起步工程
- **当前推荐路线**：平台架构仍优先朝主流 STM32、模块化和可迁移方向收敛

因此后续工作中必须区分：

- 当前能直接修改和验证的工程
- 长期推荐的平台基线

## 5. 参考项目的角色

### 5.1 `90.mini-Lite` 是什么
`90.mini-Lite` 是一个参考项目目录，主要价值是提供一份典型移动电源产品样本，帮助理解：

- `C1 + A + C2` 多口能力
- 底座 / 托盘充电模式
- LED / 按键状态机
- 温度保护和功率限制

### 5.2 它不是什么
`90.mini-Lite` 不是：

- 当前项目的代码主线
- 当前产品规格源头
- 当前工程目录
- 当前平台的唯一芯片路线

### 5.3 正确使用方式
对 `90.mini-Lite` 的正确使用方式是：

1. 抽取有价值的产品能力样本
2. 把这些能力转成平台模块需求
3. 再落到 `bmsClaw` 当前代码和文档中

而不是：

1. 直接把它当主项目
2. 直接照搬它的实现和选型

---

## 6. 当前技术架构目标

### 6.1 总体方向
`bmsClaw` 当前架构目标仍然围绕以下能力展开：

- 主控 MCU
- BMS 模块
- 电源转换模块
- 协议模块
- 显示 / 按键
- BLE / 通信
- 底座 / 托盘扩展

### 6.2 推荐软件分层
当前推荐的软件分层为：

```text
产品逻辑层
-> 服务层
-> HAL 层
-> 驱动层
-> 板级层
```

更具体可落为：

```text
app/
services/
hal/
drivers/
board/
config/
```

### 6.3 为什么这样分层
这样做的目的是：

- 让产品逻辑不直接依赖芯片寄存器
- 让 `BMS`、`protocol`、`power` 可以替换
- 让参考项目行为可以被提炼，但不绑死到某个客户项目
- 让后续增加 SKU 时不需要复制整个工程

## 7. 当前工作路径

### 7.1 阶段 1：统一主线和目录边界
先完成：

1. 统一项目名为 `bmsClaw`
2. 明确 `90.mini-Lite` 是参考项目
3. 明确 `build/` 是生成物
4. 明确当前代码事实来自 `bmsClaw/bmsClaw/bmsClaw/`

这是当前最先要完成的认知层整理工作。

### 7.2 阶段 2：稳定当前 STM32 起步工程
在现有工程上优先完成：

1. GPIO / LED
2. UART / 日志
3. ADC
4. I2C
5. 基础下载调试闭环

这一步的目标不是做完整产品，而是建立最小可观测环境。

### 7.3 阶段 3：把工程从“生成模板”推进到“可扩展骨架”
在最小工程稳定后，逐步引入：

- `board/`
- `drivers/`
- `hal/`
- `services/`
- `app/`

并把 `main.c` 的职责尽量收敛为初始化和主循环入口。

### 7.4 阶段 4：按模块接入真实硬件
随后按顺序推进：

1. 输入链路
2. 协议模块
3. BMS 模块
4. 电源转换控制
5. 底座 / 托盘
6. BLE / OTA

### 7.5 阶段 5：提炼参考项目能力
当基础平台稳定后，再从 `90.mini-Lite` 提炼：

- 典型多口产品配置
- 典型底座模式
- 典型 LED 行为
- 典型保护策略

把它们转成可配置的平台能力，而不是客户项目特化逻辑。
