# openBmsClaw 输入链路联调前置准备

## 1. 文档目的
本文件对应 `Plan-Week-0325 item-06`，用于把“真实输入链路联调”从抽象计划收敛成当前 `openBmsClaw` 可以承接的准备动作。

## 2. 先说明当前变化
原始 `item-06` 是围绕：

- `NUCLEO-G071RB`
- `X-NUCLEO-DRP1M1`
- `BQ76942EVM`

这类官方评估板组合来写的。

但当前活动工程已经切换为：

- `openBmsClaw`
- `STM32F103C8T6`
- 当前学习实物统一按 `野火小智 STM32F103C8T6 核心板（双 USB 款）`

这意味着：

- 当前软件主线先围绕 F103 起步工程推进
- 当前 F103 学习板并不是“只有 USB-UART”，它板上也有标准 `SWD`
- 真正的输入链路联调，还需要等实际 `PD / USB-C` 板和 `BMS` 板型号最终落定

## 3. 当前 item-06 能做什么
在当前没有真实协议板和 BMS 板接入的前提下，`item-06` 最合理的完成方式不是“假装已经联调”，而是先把联调入口准备好。

当前已经为联调准备了这些代码落位点：

- `services/protocol/`
- `services/power/`
- `services/bms/`
- `hal/protocol/`
- `hal/power/`
- `hal/bms/`

它们的意义是：

1. 上层行为不会直接依赖某一个协议芯片。
2. 后续输入链路联调时，可以把“协议芯片适配”和“BMS 芯片适配”限制在对应目录里。
3. 不会因为接了第一块评估板，就把整个工程写成某一块板的硬编码版本。

## 4. 输入链路联调前必须先确认的四件事

### 4.1 第一块 `USB-C / PD` 板的具体型号
当前如果没有最终定型，就不能进入真实协议联调。

至少要确认：

- 板卡型号
- 芯片型号
- 与 MCU 的通信接口
- 板卡供电方式

### 4.2 第一块 `BMS` 板的具体型号
同样需要确认：

- 板卡型号
- 芯片型号
- 通信方式
- 资料完整度

### 4.3 当前 F103 板的稳定下载路径
在没有稳定下载路径前，不要开始真实外设联调。

需要先确认是：

- 外部 `SWD` 调试器
- 还是串口 Bootloader

### 4.4 当前最小观测路径
输入链路联调前，至少要先有一种可观测手段：

- `UART log`
- 或稳定的调试会话

否则接外设后出了问题很难判断。

## 5. 当前已经完成的“联调前置软件准备”
当前 `openBmsClaw` 已经完成：

- `protocol / power / bms` 三类服务入口预留
- `hal / drivers / board / config` 分层落位
- 当前工程主循环已经可承接这些服务

这使得未来到货的真实硬件可以按下面方式接入：

```text
真实板卡资料
  -> hal/<module>/
  -> services/<module>/
  -> board/ 当前板级绑定
  -> app/ 主流程调用
```

## 6. 当前还不能写成“已完成”的部分
下面这些事仍然没有发生，因此当前不能写成“输入链路联调已完成”：

- `PD / USB-C` 评估板真实接入
- `BMS` 评估板真实接入
- 真实握手状态读取
- 真实输入状态日志
- 真实上电时序验证

因此，当前更准确的状态应写成：

- 已完成输入链路联调前的软件准备
- 真实板级联调待硬件条件满足后开始

## 7. 当前最合理的下一步顺序
建议顺序如下：

1. 先把 `openBmsClaw` 的下载路径确认稳定
2. 先把 `UART` 作为最小日志通道跑通
3. 锁定第一块 `USB-C / PD` 板型号
4. 锁定第一块 `BMS` 板型号
5. 再开始输入链路真实联调

## 8. 当前结论
基于当前项目状态，`item-06` 的合理结论是：

- 当前已为输入链路联调建立软件承接结构
- 真实联调尚未开始
- 阻塞点主要在“真实板卡型号”和“当前下载 / 调试路径”两件事上

这比直接把 `item-06` 写成“已完成联调”要准确得多，也更符合当前开源项目的真实推进方式。
