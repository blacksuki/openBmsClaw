# bmsClaw 首周 V0 范围说明

## 1. 文档目的
本文件对应 `Plan-Week-0325 item-01`，用于把本周工作收敛到“最小可运行验证”。

这里的重点不是做完整移动电源产品，而是先把当前可用硬件、当前 STM32 起步工程、当前软件结构准备成一个可继续迭代的平台起点。

## 2. 当前事实
- 当前主项目是 `bmsClaw`，不是 `90.mini-Lite`。
- `90.mini-Lite` 只作为行为样本参考，用来帮助理解输入/输出角色、LED 状态和底座类需求。
- 当前已有代码工程位于 `bmsClaw/bmsClaw/`，属于 `STM32CubeIDE for Visual Studio Code` 生成的 STM32 起步工程。
- 当前工程可见的器件基线是 `STM32F072RBT6 / STM32F072xB` 风格。
- 本周现成开发硬件以“已购 `STM32F07` 教学套件”作为首要执行平台推进。
- `USB-C/PD` 扩展板和 `BMS` 评估板如果本周未到位，不阻塞本周主交付。

## 3. 本周唯一目标
本周唯一目标是：

> 在现有 `STM32F07` 教学套件上，完成 `bmsClaw` 的最小可运行验证。

这里的“最小可运行验证”指的是：
- 工程能编译、下载、启动。
- 至少有一种可观察输出，例如 `LED` 或 `UART` 日志。
- 至少完成 `ADC`、`I2C` 的基础 bring-up 或探测能力准备。
- 工程结构开始脱离“所有逻辑都塞进 `main.c`”的硬编码路线。
- 为后续 `BMS / protocol / power` 接入保留明确接口，而不是把产品行为写死在首周代码里。

## 4. 输入链路与输出链路
为了避免后续讨论混乱，先统一这两个概念：

- **输入链路**：外部电源进入系统的路径。例如适配器或 `USB-C` 给设备充电，再进入电池或系统母线。
- **输出链路**：电池或系统母线对外供电的路径。例如设备给手机、耳机或其他负载输出电能。

本周默认优先级是：

> 先做输入链路相关准备与探测能力，再考虑输出链路。

但由于当前实物条件主要是 `STM32F07` 教学套件，所以本周“输入链路优先”主要体现为：
- 优先搭好后续接 `USB-C/PD` 板所需的 `I2C / ADC / GPIO / 日志` 基础；
- 优先预留输入状态检测、错误记录和接口抽象；
- 不强行承诺本周一定完成真实电源路径闭环。

## 5. 本周纳入范围
| 模块 | 本周动作 | 完成定义 |
| --- | --- | --- |
| MCU 起步工程 | 整理当前 `bmsClaw` STM32 生成工程 | 工程能正常编译，目录与职责开始清晰 |
| `board` / `config` | 建立板级配置入口 | 板级差异不直接散落在业务逻辑里 |
| `LED` | 建立最小状态指示 | 能表达“启动中 / 运行中 / 错误”中的至少一种状态 |
| `UART` | 建立串口日志输出 | 上电后可看到基础日志，便于新手排错 |
| `ADC` | 建立基础采样入口 | 能完成基础采样或至少完成采样驱动占位 |
| `I2C` | 建立总线访问入口 | 能初始化总线，并为后续外设探测预留调用点 |
| 可观测性 | 固化日志和错误入口 | 出问题时能先看日志，而不是盲调 |
| 工程骨架 | 先做模块边界，不做复杂业务 | 为后续 `BMS / protocol / power` 接入留出位置 |

## 6. 本周只做接口、不做完整实现
| 模块 | 本周处理方式 | 说明 |
| --- | --- | --- |
| `BMS` | 只定义接口与数据结构 | 不在本周完成真实寄存器接入与校准 |
| `protocol` | 只预留驱动与状态接口 | 不在本周实现 `PD/QC` 完整握手 |
| `power conversion` | 只预留控制入口 | 不在本周实现功率路径、限流、降额策略 |
| `dock` | 只保留模式识别入口 | 不在本周实现单底座/五联托盘行为 |
| `ui` 高阶行为 | 只保留基础状态机入口 | 不在本周实现完整产品灯效表 |

## 7. 本周明确不做
- 不做完整 `USB-C/PD` 协议联调闭环。
- 不做真实 `BMS` 芯片的寄存器适配、校准和保护阈值验证。
- 不做多口输出功率分配算法。
- 不做 `BLE / OTA`。
- 不做底座和五联托盘整机逻辑。
- 不做量产 PCB、器件定型和热设计验证。
- 不把 `90.mini-Lite` 的完整产品规格直接搬成 `bmsClaw` 本周交付内容。

## 8. 从参考样本中只借鉴什么
从 `90.mini-Lite/miniLite_system_design.md` 本周只提取三类“行为样本”：

1. 输入与输出端口角色需要分清，不要把“充电”和“放电”写成一套混乱状态机。
2. `LED` 不只是装饰，后续应承担运行态、充电态、错误态的最小反馈职责。
3. 采样、告警、保护入口必须尽早预留，否则后面只能回到硬编码补丁路线。

这些内容仅用于帮助当前框架定边界，不代表 `bmsClaw` 本周已经实现了对应完整功能。

## 10. 本周完成判定
满足以下条件，即认为 `item-01` 对应的范围收敛完成：

1. 团队能够明确回答“本周只做什么，不做什么”。
2. 当前主线明确收敛到“最小可运行验证”，而不是“整机功能实现”。
3. 当前执行平台明确为已购 `STM32F07` 教学套件，而不是假定手头已有完整官方评估板组合。
4. `BMS / protocol / power` 被视为后续接入模块，而不是本周一次性硬上。
5. 本周每个 action item 都能映射到这个范围说明。

## 11. 对新手的执行提醒
- 首周目标小，不代表目标弱。首周真正要建立的是“能看见、能定位、能继续接模块”的起点。
- 先把下载、日志、引脚、总线、目录结构做好，比提前写复杂功率策略更重要。
- 只要本周能把工程从“空循环”推进到“可观察、可扩展、可排错”，这周就是有效进展。
