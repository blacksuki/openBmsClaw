## 1. 目标与背景

`openBattery` 进入启动期，一个直接课题是"按项目实际需求与设想，组建团队、合理分工、界定职责边界"。

经澄清，本次要做的**不是**给真人排班，也**不是**定义 AI agent 角色，而是一份**概念性职责框架**：定义"未来该有哪些 owner、各自边界"，作为招人 / 外包 / 顾问时的对照；现阶段一人多岗，用"谁兼"字段标注现实。

框架主线是**工作流分解 + owner**：先按项目实际工作流切块，每块配一个 owner 角色。

### 1.1 已确认输入（本次澄清锁定）

- 产物形态：**方案 A** —— 新建 `0_System/04.team_and_ownership.md`，与其它 `0_System/*.md` 规格并列。
- 角色粒度：**合并为 4 块**（不是 7 条独立，也不是纯矩阵）。
- 角色卡字段：**精简五字段** —— 使命 / 边界 / 决策权 / 交付物 / 现阶段谁兼。
- 硬件（卡2）：**已在使用外包硬件开发团队**（已合作），非空缺口。
- 项目协调（卡4）决策权：**门禁 pass/打回，而非技术方案拍板**。

### 1.2 依据的项目事实

- master 文档 §1.6：团队软件出身，嵌入式 / 硬件保护 / 电源 / 热 / 量产理解不足，需主动标出能力边界与风险点。
- 核心架构：MCU（Brain）+ 专用电源 SoC（Body，英集芯）双芯协同，MCU 只经 `soc_api.h` 抽象层与 SoC 通信（"要接口不要源码"）。
- 首发主线：日本中端便携、安全 + 数据可追踪、NFC(ST25DV) tap 读出、PSE 合规。

## 4. 4 张角色卡（内容实质）

工作流合并映射：
- 卡1 = 原「MCU 固件 / Brain」+「SoC 集成 / Body」
- 卡2 = 原「硬件 / 电源 / 热」
- 卡3 = 原「NFC / 数据」+「PSE / 合规」+「产品 / 市场」
- 卡4 = 原「项目协调」+ 供应链

### 卡1 · 固件 + SoC 集成 owner（Brain↔Body）
- **使命**：MCU 侧策略 / 日志 / UI + 英集芯 SoC 适配，守住 `soc_api.h` 抽象边界。
- **边界**：碰得了 `app`/`services`/`drivers/soc`/`hal`；**碰不了** 在 app/service 里直接戳寄存器，碰不了硬件电路。
- **决策权**：固件架构、SoC 接口契约**建议权**；`vendor/<chip>.c` 寄存器表实现**拍板权**。
- **交付物**：固件架构、SAL 适配、编码规范、集成报告。
- **现阶段谁兼**：你（软件出身，主力）。

### 卡2 · 硬件 / 电源 / 热 owner
- **使命**：原理图、PCB、电源设计、热、EMC 及 PSE 相关硬件侧。
- **边界**：碰得了电路 / PCB / 选型；**碰不了** 固件架构与业务策略。
- **决策权**：硬件方案**拍板权**（团队最大能力缺口，由外包团队承担并需评审确认）。
- **交付物**：原理图、PCB、BOM、热 / EMC 评审。
- **现阶段谁兼**：**外包硬件开发团队（已在合作）**。

### 卡3 · 产品 / 数据 / 合规 owner（NFC + PSE + 市场）
- **使命**：产品定义、竞品 / 价格带、NFC 数据模型 v0、PSE 认证路径。
- **边界**：碰得了需求 / 数据模型 / 合规策略；**碰不了** 寄存器与电路实现。
- **决策权**：首发范围、数据模型 v0 **拍板权**；合规路径**建议权**（需认证机构确认）。
- **交付物**：产品定义、数据模型 v0、竞品 / 合规调研、待验证项清单。
- **现阶段谁兼**：你 + 待补市场 / 合规输入。

### 卡4 · 项目协调 / 供应链 owner
- **使命**：里程碑、stage gate 把关、决策记录、英集芯合作接口。
- **边界**：碰得了计划 / 门禁 / 对外接口；**不替各 owner 做技术拍板**。
- **决策权**：阶段门 **通过 / 打回 拍板权**；技术方案仅协调，不定技术方向。
- **交付物**：阶段计划、stage-gate 记录、供应链验收清单。
- **现阶段谁兼**：你。

## 5. 与三件套 skill 的关系

本框架是后续三个 skill 的锚点，但**本次只交付框架文档**，skill 由后续计划分别落地：

- `stage-gate` —— 由卡4 owner 把关；框架提供"谁有 pass/打回 权"。
- `interface-contract`（SoC 契约评审）—— 由卡1 owner review、卡4 owner 记录；框架提供 review 归属。
- `project-charter` —— 引用本框架的 owner 定义，避免重复定义角色。
